Domain-Driven Design, the Useful Parts

Bounded contexts and ubiquitous language, minus the 600-page book — the DDD concepts that actually change how you write software.

Eric Evans' Domain-Driven Design (2003) is 560 pages. Most engineers who say they "do DDD" mean they use a subset of about four ideas from it, and that subset is genuinely useful. The rest of the book is tactics for pre-framework Java enterprise architecture that you don't need unless you're building a monolith in 2004.

The useful subset: ubiquitous language, bounded contexts, aggregates, and domain events. These four ideas solve a real problem: keeping the code's model of the world aligned with the business's model of the world, across teams and over time.

Ubiquitous language: name things the way the business names them

If the business says "policy" and the code says "InsuranceContract," one of you is translating on every conversation. Stop translating.

The most impactful DDD practice is the simplest: use the business's words in the code. Not "close enough" translations. The exact words. If the domain expert says "a shipment is manifested, not created," your code should have a manifest() method, not a create() method. If they distinguish between a "lead" and a "prospect," those should be different types, not different statuses on a single type.

This sounds trivial. It's not. The gap between "what the business asked for" and "what engineering built" is often traceable to a single translation error — a term that meant one thing to the product manager and another thing to the engineer. Ubiquitous language closes that gap by making the code literally speak the business's language. When a domain expert reads your class names, they should recognize them as concepts they use daily.

Bounded contexts: the most important DDD concept, period

A bounded context is a boundary within which a term has a single, unambiguous meaning. Outside that boundary, the same term might mean something different — and that's not a bug, it's the design.

The classic example: "Customer" means different things to different parts of the business. To sales, a Customer is a person in a CRM with a pipeline stage. To billing, a Customer is an entity with a payment method and an account balance. To support, a Customer is a person with a ticket history and an SLA tier. If you try to build one Customer class that serves all three, you get a 200-field monster where most fields are null in most contexts, and a change for the sales team breaks billing's queries.

DDD's answer: let each context have its own Customer. SalesCustomer, BillingCustomer, SupportCustomer. They might share an ID. They might be populated from the same source events. But they are separate types with separate schemas, and a change to one doesn't affect the others. Translation happens at the boundary — when data crosses from one context to another, it's explicitly mapped.

This is also how microservices should be split. A bounded context is the right boundary for a service — not a database table, not a REST resource, not a team's preferred language. The context boundary is where the language changes. Split services along language boundaries, and the organizational boundaries (teams own contexts) naturally align with the code boundaries (contexts are deployable units).

Aggregates: the unit of consistency

An aggregate is a cluster of objects that are treated as a single unit for data changes. Exactly one entity in the cluster is the aggregate root — the only entity that external code can reference. All changes to objects inside the aggregate go through the root.

The rule: one aggregate = one transaction. You don't modify two aggregates in one database transaction. If two things need to be modified together, they belong in the same aggregate. If they belong in different aggregates, they're modified in separate transactions, and you accept eventual consistency between them.

This is the DDD concept that most directly shapes your database schema and your API design. An aggregate that's too large (an entire Order containing LineItems, Customer, Payment, and Shipment) causes contention — two operations that touch different parts of the order block each other. An aggregate that's too small (each LineItem as its own aggregate) can't enforce invariants across line items (e.g., "an order must have at least one line item"). Sizing aggregates correctly is the hardest DDD skill and the one that most determines whether your system is performant under concurrent load.

predict You're modeling an e-commerce system. An Order contains LineItems, a ShippingAddress, and a Payment. Should Payment be part of the Order aggregate, or a separate aggregate referenced by ID? What's the invariant that decides?

If the rule is "an order can only be placed when payment is confirmed," Payment might need to be part of the Order aggregate — the invariant spans both. But if payment can happen separately (e.g., pay later, partial payments, payment retries), Payment should be a separate aggregate referenced by ID from the Order. The deciding question: does a change to Payment require the Order to be locked? If yes, they're the same aggregate. If no, separate them. Most e-commerce systems separate them, because payment processing is slow, retryable, and shouldn't block modifications to the order (like adding items or changing the shipping address during a payment retry).

what to skip The DDD book's tactical patterns — repositories, factories, value objects, domain services, application services, specification pattern — are useful vocabulary but not the point. The point is the strategic design: bounded contexts and ubiquitous language. If you get the context boundaries right and name things consistently, the tactical patterns emerge naturally. If you focus on the tactical patterns without the strategic design, you get a beautifully factored monolith that models the wrong thing.
