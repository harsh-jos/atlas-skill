Sampling & Decoding

Temperature, top-p, and what "creativity" actually means to a model — the four knobs between your prompt and the token that comes out.

An LLM doesn't "choose" the next word. For each position, it produces a vector of logits — raw, unnormalized scores — over its entire vocabulary (128K to 200K tokens in 2026 models). The model's raw output is a list of 200,000 numbers, each representing how plausible that token is as the next one. Turning that into an actual token — the word that appears on your screen — is decoding, and there are more ways to do it than most users realize.

The pipeline: logits → probabilities → token

From model output to next token
Raw logits:     [2.1, -0.5, 8.7, 1.2, ..., -3.4]   ← 200K numbers
↓ softmax
Probabilities:  [0.001, 0.000, 0.72, 0.003, ..., 0.000]  ← sums to 1.0
↓ sampling strategy (temperature, top-p, top-k)
Filtered probs: [0.0, 0.0, 0.85, 0.0, ..., 0.0]         ← only valid candidates
↓ sample
Token ID:       2874  → "the"

The art of decoding is everything that happens between the probability distribution and the sampled token. Four knobs control it, and they interact in ways that surprise people who treat them as independent.

Temperature: sharpening or flattening the distribution

Temperature is the simplest knob: divide every logit by T before softmax.

pᵢ = exp(zᵢ / T) / Σⱼ exp(zⱼ / T)
  • T = 0 (greedy): Always pick the most probable token. Deterministic, reproducible, boring. Good for factual extraction and evaluation benchmarks. Bad for anything where you want the model to explore alternatives.
  • T < 1 (low): Sharpens the distribution — the model becomes more confident in its top choices. At T=0.3, the probability mass concentrates heavily on the top few tokens. Good for code generation and structured data where correctness matters more than variety.
  • T = 1 (neutral): Uses the model's raw probabilities as-is. The model's unmodified judgment about what should come next.
  • T > 1 (high): Flattens the distribution — makes unlikely tokens more likely. At T=2.0, the model starts considering tokens it would normally dismiss. Good for creative brainstorming and ideation where surprising connections are the point.
Temperature doesn't make the model "more creative" — it makes the model more willing to bet on tokens it thinks are unlikely.

A crucial detail: temperature changes the shape of the distribution but doesn't truncate it. At T=0.7, an improbable token (0.001 probability) might become 0.01 — still improbable, but 10× more likely to be sampled than at T=1.0. This is why setting T=1.2 and being surprised by a nonsensical output is a user error, not a model hallucination — you told the model to take risks, and it did.

Top-k: fixed-size candidate pool

Top-k truncation keeps only the k highest-probability tokens, zeros out the rest, renormalizes, and samples. k=50 means the model chooses from the top 50 candidates regardless of how confident it is.

The problem: "top 50" means very different things in different contexts. When the model is highly confident (next word is almost certainly "the"), the 50th token has near-zero probability, but top-k includes it anyway. When the model is uncertain (creative writing, many plausible next words), the 50th token might still be genuinely plausible. A fixed k applies the same truncation to both situations, which is mathematically unsatisfying and practically leads to either including noise or excluding valid options.

Top-k is increasingly used as a coarse filter (k=50 or 100) in combination with top-p, rather than as a standalone strategy. The Anthem recipe (popularized in 2025 for agentic LLM outputs) pairs top-k=50 with top-p=0.95 and min-p=0.05 for a belt-and-suspenders approach to tail control.

Top-p (nucleus sampling): dynamic candidate pool

Top-p addresses top-k's fixed-width problem by making the pool size dynamic. It keeps the smallest set of tokens whose cumulative probability exceeds p (typically 0.9–0.95), zeros out the rest, renormalizes, and samples.

When the model is confident, the nucleus might be 3 tokens that collectively represent 0.95 probability. When the model is uncertain, the nucleus might be 200 tokens. The pool size adapts to the model's confidence, which is the behavior you almost always want.

This is why top-p has largely displaced top-k as the primary truncation strategy in production LLM APIs. OpenAI's default is top-p=0.95 (temperature varies by model). Anthropic's Claude defaults to top-p=0.95. The dynamic behavior means you set one number and it works reasonably across different kinds of text.

Min-p: the relative floor (the 2025 addition)

Min-p is the newest widely-adopted knob. It sets a threshold relative to the maximum probability: if the most probable token has probability 0.6, min-p=0.05 discards any token below 0.6 × 0.05 = 0.03. This acts as an adaptive floor that moves with the model's confidence level.

The insight: what counts as "too improbable" depends on how confident the model is about its best option. If the top token has p=0.8, a token with p=0.01 is genuinely a long shot. If the top token has p=0.1 (the model is genuinely unsure), a token with p=0.01 might be worth considering. Min-p captures this relativity automatically, which fixed thresholds (like "drop tokens below 1%") don't.

predict You set T=0 (greedy) to get deterministic outputs for your eval suite. But the same prompt with T=0 produces different answers on different runs. How?

Even at T=0, non-determinism leaks in. Floating-point non-associativity means the order of parallel operations on the GPU can produce slightly different logit values (a + b + c might not equal a + c + b in float32). Batch composition changes which operations run concurrently. Mixture-of-experts models route tokens to different experts based on load, and different load produces different routing decisions. For true determinism, you need: T=0, a pinned seed, a pinned model version (dated snapshot, not latest), and — for self-hosted models — fixed batch sizes and deterministic CUDA kernels. Even then, some variance remains. The practical answer: evaluate with multiple runs (3–10 per prompt) and report mean ± std, not single-run results.

when not to tune The default settings from the API provider (T=0.7, top-p=0.95 for OpenAI; T=1.0, top-p=0.95 for Anthropic) are the result of extensive testing across diverse workloads. For most use cases, they're good enough. Tuning decoding parameters before you've tuned your prompt is optimization at the wrong layer — a better prompt will improve output quality more than any combination of sampling parameters. Start with the defaults, write a good prompt, and only reach for the decoding knobs when you have a specific reason.
