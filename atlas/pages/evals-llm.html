Evaluating LLM Systems

Why "it looked good in the demo" is the most expensive sentence in AI — and the eval frameworks that catch problems before your users do.

A human evaluates an LLM's output by reading it and thinking "that looks right." This works once, for one output. It doesn't work for a thousand outputs, across prompt changes, model version upgrades, and RAG pipeline tweaks. A system whose evaluation is "vibes" will silently regress — a prompt change improves one case and degrades ten others, and you won't know until users complain.

LLM evaluation is the discipline of measuring output quality systematically, automatically, and continuously. It's the hardest problem in production LLM engineering — harder than RAG, harder than fine-tuning — because "correctness" for free-text output is fundamentally fuzzier than accuracy for classification.

The eval taxonomy

There is no single "accuracy" metric for LLM outputs. You measure different dimensions with different tools, and the combination tells you whether a change is an improvement or a regression.

1. Reference-based (n-gram overlap): Compare the output to a known-good reference answer. BLEU, ROUGE, METEOR. Fast, automatic, and useful when the acceptable answer space is narrow (translation, extractive summarization). Useless when many answers are valid (creative writing, open-ended QA) — "the cat sat on the mat" and "the feline rested on the rug" get a low BLEU score despite being semantically equivalent.

2. LLM-as-judge (model-graded): Use a stronger LLM to evaluate the output of a weaker one. "Rate the following response on accuracy, relevance, and helpfulness on a scale of 1–5." The judge model is given the prompt, the response, and optionally a reference answer. This is the dominant evaluation method in 2026, used by the Chatbot Arena, AlpacaEval, and most production eval pipelines. It correlates reasonably well with human judgment and scales to thousands of examples. The cost: the judge model has its own biases (preferring longer responses, its own style), and you need to calibrate it against human judgments periodically.

3. Code/structured evaluation: If the output is code or structured data, run it through a verifier. Code passes tests? Correct. JSON parses and matches a schema? Correct. SQL returns the expected results? Correct. This is the most reliable evaluation — deterministic and fast — but only applies to structured outputs.

4. Human evaluation: The ground truth. Slow, expensive, and essential for calibrating automatic metrics. Run a human eval on a representative sample (100–500 examples) regularly (monthly), and use it to validate that your automatic metrics still correlate with human judgment.

What to measure

  • Accuracy / correctness: Is the answer factually correct? (Hardest to measure automatically — requires ground truth or LLM-as-judge with reference.)
  • Groundedness / faithfulness: If using RAG, is the answer supported by the retrieved documents, or did the model hallucinate? Measure: extract claims from the output, check each claim against the retrieved context.
  • Relevance: Does the answer address the question? A factually correct answer about the wrong topic is a failure.
  • Safety / refusal: For unsafe prompts, does the model refuse correctly? For safe prompts, does the model avoid false refusals? Both over-refusal and under-refusal are bugs.
  • Tone / format: Does the output match the expected tone and structure? Especially important for customer-facing applications.
predict Your eval pipeline uses LLM-as-judge (GPT-4) to rate outputs on a 1–5 scale. You switch to a new generation model and your eval scores jump from 3.8 to 4.3. But human raters say quality is unchanged. What's the likely cause?

Judge bias — specifically, the judge model might prefer the style of the new generation model (longer responses, more confident tone, certain formatting patterns) even though factual accuracy is unchanged. Or the judge model itself changed (if you upgraded GPT-4 versions). Or your eval prompts are leaking information that makes the judge prefer one model's style. The fix: calibrate the judge against human judgments on a held-out set, use multiple judge models (GPT-4 + Claude), and track relative changes rather than absolute scores. An absolute score of "4.3" is meaningless across model versions; a relative change of "+0.5 vs baseline" is meaningful — but only if the judge is calibrated.

The eval pipeline

Production LLM eval pipeline
1. Curate eval set: 100–1000 representative prompts + expected outputs
   (or reference answers for LLM-as-judge)
2. On every change (prompt, model, RAG config):
   a. Run eval set through the system
   b. Score each output (reference-based + LLM-as-judge)
   c. Compare to baseline: did any dimension regress?
3. Block deployment if any critical dimension regresses
4. Periodically (monthly): human eval on a random sample to recalibrate
the eval bootstrapping problem You need a good eval set to improve your system. But building a good eval set requires understanding what "good" means for your specific use case, which requires evaluating outputs, which requires an eval set. The bootstrap: start with 20 hand-written prompts and grade outputs manually. As you see patterns, expand to 100, then 500. Use the manual grades to train an LLM-as-judge prompt that correlates with your judgments. Automate the grading, but keep the manual set as a calibration benchmark. The eval set is never finished — it grows as you discover new failure modes. An eval set from January will not catch a failure mode discovered in June. Treat it as a living asset that accrues value over time.
