Rate Limiting: Token & Leaky Bucket

The two little algorithms behind every "429 Too Many Requests" — and why the difference between them is the difference between smooth and bursty.

You expose an API. A client starts hammering it — 1,000 requests per second, accidentally (bug) or deliberately (scraper). Your database can handle 200 queries per second. Without rate limiting, the client saturates the database, and every other user experiences degraded service. One misbehaving client shouldn't be able to hurt everyone else.

Rate limiting is the answer: cap the number of requests a client can make in a given time window. But the implementation matters — a simple counter-based limiter ("100 requests per minute") has sharp edges that the token bucket and leaky bucket algorithms smooth over.

Token bucket: allow bursts, cap the average

A token bucket fills at a constant rate. Each request consumes a token. If tokens are available, the request proceeds. If the bucket is empty, the request is rejected. The bucket's capacity allows bursts.
Token bucket, conceptually
Bucket capacity: 100 tokens
Refill rate: 10 tokens/second (one every 100ms)

Steady state: a client sending 10 req/s never exhausts the bucket
Burst: a client can send 100 requests instantly (full bucket), then
       is limited to 10 req/s until the bucket refills

Parameters: rate (tokens per second) and burst (bucket capacity). The burst allows short spikes — a client that's been idle accumulates tokens and can burst when it resumes. This is the natural behavior for user-facing APIs: a human browsing a website generates bursty traffic (click → rapid page loads → long pause). A token bucket accommodates this gracefully.

The token bucket is the most common rate limiting algorithm in API gateways. It's what nginx's limit_req implements (with the burst parameter), what AWS API Gateway uses, and what most SaaS rate limiters default to. The Redis implementation stores the bucket state (token count + last refill timestamp) per client key, with atomic Lua scripts to decrement tokens.

Leaky bucket: smooth the flow

A leaky bucket is a queue with a constant drain rate. Requests enter the bucket; if the bucket is full, they're rejected. The bucket leaks at a constant rate — one request is processed every N milliseconds. The output is a smooth, constant stream regardless of how bursty the input is.

The difference from a token bucket: a token bucket allows bursts (if tokens are available). A leaky bucket never allows bursts — the output rate is constant. A token bucket shapes by rate-limiting the input; a leaky bucket shapes by smoothing the output.

Leaky bucket vs token bucket
Token bucket: allows 100 req/s average, with bursts up to bucket capacity
             → Good for user-facing APIs (bursty is natural)

Leaky bucket: processes exactly 10 req/s, always, regardless of input pattern
             → Good for network traffic shaping (constant rate prevents
               downstream buffer overflows)

The leaky bucket is the algorithm behind traffic shaping in networking (the "leaky bucket" in ATM networks, QoS policing). It's less common in API rate limiting because most APIs want to allow bursts — a user clicking through pages shouldn't be rate-limited just because they clicked quickly.

Fixed window counter: the simplest algorithm (and its trap)

The simplest rate limiter: a counter that resets every window (minute, hour). "100 requests per minute." Reset on the minute boundary.

predict A fixed-window limiter allows 100 requests per minute, resetting at :00 seconds. A client sends 100 requests at 12:00:59 (the last second of minute 1) and 100 requests at 12:01:00 (the first second of minute 2). In 2 seconds, the client sent 200 requests. Does the limiter stop this? What's the effective rate over those 2 seconds?

The limiter does not stop it — 100 is within the minute-1 limit, and 100 is within the minute-2 limit. The effective rate over 2 seconds is 100 req/s, double the configured 100 per minute averaged over the minute but 60× the intended instantaneous rate. This is the edge-of-window burst problem. The fix: sliding window — instead of resetting at boundary, count requests in the last N seconds. Or use a token bucket, which naturally smooths the edge-of-window burst by limiting the refill rate. The sliding window is more expensive (needs per-request timestamps, not just a counter), which is why fixed windows persist — they're cheap and good enough for most use cases where precise rate limiting isn't critical.

the distributed problem Rate limiting a single server is straightforward. Rate limiting across a cluster of 100 servers, where each server sees a fraction of traffic, is a distributed counting problem. The naive approach — each server maintains its own bucket — fails because a client's total rate is the sum across servers, and no single server sees the whole picture. The fix: a centralized rate limiter (Redis, with per-key buckets), or a distributed agreement protocol (each server broadcasts its local counts, consensus on the total). The centralized approach is simpler and more common; the distributed approach avoids the Redis dependency but adds complexity. Rate limiting, like idempotency, is a distributed systems problem in API-gateway clothing.

The other half of flow control — backpressure is what happens when the server pushes back on the client instead of rejecting it. Rate limiting says "no." Backpressure says "slow down."
