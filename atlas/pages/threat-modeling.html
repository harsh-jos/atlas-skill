Threat Modeling

Thinking like the attacker, on a whiteboard, before you ship — the practice that turns security from a scanner report into an engineering discipline.

Security tools find bugs. A SAST scanner tells you about SQL injection on line 47. A dependency scanner tells you about a CVE in your YAML parser. These are valuable — use them. But they don't answer the question that matters most: what should I be worried about?

Threat modeling is the practice of answering that question systematically. You draw a diagram of your system, enumerate what could go wrong, and decide which threats are worth defending against. The output isn't a list of bugs — it's a prioritized list of risks with mitigations. And the most useful output is often the decision not to mitigate something, because the cost of defense exceeds the cost of the compromise.

Threat modeling is risk management applied to software. The goal isn't to find every vulnerability — it's to find the ones worth fixing.

STRIDE: the six things that can go wrong

Microsoft's STRIDE framework (1999, still the most widely used) categorizes threats into six families. For each component in your system, ask:

  • Spoofing: Can someone pretend to be someone else? (Forged JWT, stolen session cookie, IP spoofing.)
  • Tampering: Can someone modify data they shouldn't? (Modifying a request body in transit, writing to a database without authorization.)
  • Repudiation: Can someone deny doing something? (No audit log for sensitive actions, deletion of logs.)
  • Information disclosure: Can someone see data they shouldn't? (Leaked S3 bucket, verbose error messages exposing stack traces, missing authentication on an internal endpoint.)
  • Denial of service: Can someone prevent legitimate users from using the system? (Unbounded requests, expensive queries without rate limiting, resource exhaustion.)
  • Elevation of privilege: Can someone gain permissions they shouldn't have? (A user becoming admin by exploiting a missing authorization check.)

The value of STRIDE isn't the categories — it's that it gives you a checklist. Without a framework, threat modeling becomes "what are we worried about?", which biases toward the last incident the team experienced. With STRIDE, you systematically work through each category for each component, and you find threats you'd miss if you were brainstorming.

How to actually run one

A threat modeling session that produces useful output in two hours:

  1. Draw the data flow diagram. Not the architecture diagram with every microservice. The data flow diagram: where does data enter the system? Where does it go? Where are the trust boundaries (the lines between "we control this" and "we don't")? A trust boundary is where threats concentrate — every crossing is a potential attack surface.
  2. Enumerate threats. For each data flow crossing a trust boundary, apply STRIDE. "User → Login API: spoofing (what if the JWT is forged?), tampering (what if the request body is modified?), information disclosure (what if the error message leaks whether the email is registered?)"
  3. Prioritize. Not every threat is worth mitigating. Use a simple risk matrix: likelihood × impact. A threat that's easy to exploit and exposes customer PII is critical. A threat that requires physical access to a machine in a locked data center is not.
  4. Decide and document. For each high-priority threat, choose: mitigate (fix it), transfer (buy insurance, use a third-party auth provider), accept (explicitly decide the risk is acceptable), or avoid (remove the feature that creates the risk). Document the decision — the most valuable output of a threat model isn't the fixed bugs, it's the accepted risks that the team explicitly chose not to fix.
predict Your team threat-models a new payments integration. The data flow diagram shows a trust boundary between your backend and Stripe's API. You apply STRIDE and identify tampering: "what if an attacker modifies the payment amount in transit?" What mitigations apply, and why is one of them already handled without your team doing anything?

The transmission is over TLS — the amount is encrypted in transit, so a network attacker can't modify it. This is already handled by the protocol. But TLS only protects data in transit. If an attacker compromises your backend and modifies the amount before it's sent, TLS doesn't help. The threat model distinguishes between the two scenarios: network tampering (mitigated by TLS, already handled) and application-level tampering (mitigated by input validation, audit logging, and idempotency keys — things your team builds). The value of the exercise is separating the threats that infrastructure handles from the threats that application code handles.

When to threat model

Not every feature needs a formal threat model. The triggers that justify one:

  • New trust boundary (a new external integration, a new authentication flow, exposing an internal service to the internet)
  • Sensitive data in a new context (PII moving between services, financial data being cached where it wasn't before)
  • Major architectural change (monolith → microservices, new third-party dependency with broad permissions)
  • Compliance requirement (SOC 2, HIPAA, PCI DSS — the auditor will ask for threat models)
the real value The best threat modeling output is a change to the design before code is written. If a threat model finds a problem in a deployed system, it's a bug report — valuable, but expensive to fix. If a threat model finds a problem in a design document, it's a design change — cheap and fast. The practice compounds when it's applied early in the development lifecycle, not as a pre-launch audit. A threat model during design is architecture. A threat model during a security review is a rescue mission.
