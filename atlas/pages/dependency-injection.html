Dependency Injection

A fancy name for a simple idea that makes code testable — and the three ways to do it, from manual wiring to full frameworks.

Consider a class that sends emails. It needs an SMTP server. The naive approach:

class WelcomeEmail {
    private val smtpClient = SmtpClient("smtp.example.com", 587)
    fun send(user: User) { smtpClient.send(/* ... */) }
}

This class has two problems: you can't test it without a real SMTP server, and you can't change the SMTP configuration without editing the class. The fix is so simple it barely deserves a name: pass the dependency in from outside instead of creating it inside.

class WelcomeEmail(private val mailer: Mailer) {
    fun send(user: User) { mailer.send(/* ... */) }
}

// Production:
val realEmail = WelcomeEmail(SmtpMailer("smtp.example.com", 587))
// Test:
val testEmail = WelcomeEmail(FakeMailer())

That's dependency injection. The class declares what it needs (a Mailer) and receives it from the outside. It doesn't know or care whether the Mailer is real or fake, SMTP or SendGrid. This is the D in SOLID made mechanical: the class depends on an abstraction (Mailer), not a concretion (SmtpClient).

Dependency injection is just passing arguments. The rest is tooling around the fact that passing 50 arguments through 20 layers of a large application is tedious.

The three styles

1. Constructor injection: Pass dependencies through the constructor. The class can't be instantiated without them — the compiler enforces the contract. This is the default. Use it unless you have a reason not to.

2. Setter injection: Create the object, then set dependencies via setters or properties. Useful when the dependency is optional or changes over the object's lifetime. The risk: the object can exist in an invalid state (missing a required dependency) between construction and setter calls.

3. Interface injection: The dependency provides an injector method; the class implements an interface the injector calls. Rare. Used when the injection framework doesn't have access to the class's constructor (e.g., some Android patterns).

The container: when wiring gets complex

In a real application with hundreds of classes, each with multiple dependencies, manual wiring becomes tedious and error-prone:

val db = PostgresDatabase(config.dbUrl)
val userRepo = PostgresUserRepo(db)
val emailService = SmtpMailer(config.smtpHost, config.smtpPort)
val notificationService = NotificationService(emailService, userRepo)
val orderService = OrderService(userRepo, notificationService, /* 5 more deps */)
// ... 100 more lines like this

A DI container (or injector) automates this. You register types with the container, and it resolves dependency graphs automatically:

// Registration (happens once at startup)
container.register<Database>(PostgresDatabase::class)
container.register<UserRepo>(PostgresUserRepo::class)
container.register<Mailer>(SmtpMailer::class)

// Resolution (the container figures out what depends on what)
val orderService = container.resolve<OrderService>()

The container discovers that OrderService needs a UserRepo and a Mailer, which need a Database and SMTP config, and constructs the whole tree. This is powerful and dangerous: a misconfiguration that would be a compile error in manual wiring becomes a runtime error ("no binding for type PaymentGateway") that surfaces when the container resolves. The container also obscures the dependency graph — you can't see what depends on what by reading the code; you need the container's configuration or runtime diagnostics.

predict Your DI container creates every service as a singleton by default. A new developer adds a class that holds mutable per-request state (a request context) and registers it as a singleton. What happens under concurrent load?

All requests share the same instance. Request A's data leaks to request B. If the class is registered as a singleton but should be per-request, the container creates one instance and reuses it for every request — a concurrency bug caused by lifecycle misconfiguration. The fix: register the class with a scoped lifetime (one instance per HTTP request) instead of singleton. This is the most common DI container bug: singleton-by-default for stateful objects. The rule: if a class holds state specific to a single user or request, it must be scoped, not singleton. If your container defaults to singleton, every registration of a scoped object requires explicit scope configuration — and eventually someone forgets.

the honest take DI containers solve a real problem (wiring large object graphs) and create a new one (opaque dependency graphs, runtime failures, and lifecycle bugs). For a service with 20 classes, manual wiring is less code and clearer than a container. For a service with 200 classes spread across 15 modules, a container earns its keep. The framework-specific containers (Spring for Java/Kotlin, inject for Python FastAPI, Angular's injector for TypeScript) are so deeply integrated with the ecosystem that avoiding them is more work than using them. The pragmatic rule: use constructor injection everywhere. Use a container when manual wiring becomes the bottleneck. Never use setter injection for required dependencies. And always prefer compile-time DI (Dagger for Java/Kotlin, MacWire for Scala) over runtime DI when you have the option — a misconfiguration that's a compile error is infinitely better than one that's a production incident.
