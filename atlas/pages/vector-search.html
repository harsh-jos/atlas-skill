Vector Search & Approximate Nearest Neighbors

Finding "similar" in a billion vectors without comparing all of them — the algorithmic trick under every modern search, recommendation, and RAG system.

You have a billion embeddings and a query vector. You want the 10 nearest neighbors. The brute-force answer — compute the distance from the query to every single database vector — is O(n·d): a billion dot products, each over 768 or 3072 dimensions. On a GPU, this takes seconds. In a production search system where the user expects an answer in under 100ms, it's unusable.

Approximate Nearest Neighbor (ANN) search replaces "find the exact closest vectors" with "find vectors that are very close to the closest ones, with high probability." The approximation is the trick: you trade a tiny amount of recall (99.9% instead of 100%) for a 1000× speedup. In practice, the "approximate" in ANN rarely matters — the difference between the 10th and 11th nearest document is usually noise anyway.

HNSW: the navigable graph (and the algorithm that won)

Build a graph where edges connect nearby vectors. To search, start somewhere and greedily walk toward closer nodes. The graph navigates you to the right neighborhood in O(log n) steps.

Hierarchical Navigable Small World (HNSW, Malkov & Yashunin, 2016) is the dominant ANN algorithm in 2026. It builds a multi-layer graph during indexing:

  • Bottom layer (level 0): contains every vector. Highly connected — each node links to its M nearest neighbors (typically M=16–64).
  • Higher layers (level 1, 2, …): contain progressively fewer vectors (each vector is probabilistically promoted to higher layers with an exponentially decreasing probability). Edges are longer — these are the "highways" of the graph.

Search starts at the top layer's entry point. At each layer, the algorithm greedily moves to the neighbor closest to the query until it reaches a local minimum. It then descends to the next layer and repeats. The top layers provide long-range navigation (think: interstate highways); the bottom layer provides local precision (think: local streets). The result: search time scales as O(log n), not O(n).

HNSW is the default in FAISS, the index behind Milvus, and the algorithm you activate when you set index_type="HNSW" in pgvector. It's not the only algorithm, but it's the one you'll reach for first.

Why approximate search works (and when it doesn't)

The quality of an ANN index is measured by recall@k: what fraction of the true top-k results does the approximate search return? An index with recall@10 = 0.95 returns 9.5 of the true top-10 neighbors on average, missing 0.5.

The recall-speed trade-off is a dial, not a switch. More graph edges (higher M) → higher recall, slower search, larger index. Deeper search (more candidates explored, controlled by the ef_search parameter) → higher recall, slower per-query. The art of tuning an ANN index is finding the M and ef_search that give you acceptable recall at your latency target.

ANN fails gracefully in a specific way: it misses isolated nearest neighbors. If a query embedding happens to be very close to one specific vector that's poorly connected in the graph, the search may miss it. This is the "search didn't find the one document I knew was there" failure mode. The fix is usually higher M (more edges, harder to miss isolated nodes) or a hybrid approach: ANN for recall, then exact reranking of the top candidates.

The vector DB landscape: when you need more than a library

FAISS is a library, not a database. It gives you fast ANN search in memory, but if you need persistence, replication, metadata filtering, or multi-tenancy, you're building those yourself. The vector database adds those on top of the ANN core:

  • Qdrant (Rust, OSS) — the strongest all-rounder in 2026: fast filtering, good performance, modest ops overhead. Default recommendation for production RAG.
  • Milvus (Go/C++, OSS) — best at extreme scale (billions of vectors), GPU-accelerated, but demands Kubernetes expertise.
  • Pinecone (SaaS) — easiest developer experience (API key and go), but most expensive at scale and vendor-locked.
  • pgvector (Postgres extension) — if your vectors live alongside structured data and you want ACID transactions. Slower at pure ANN than purpose-built DBs, but the simplicity of one database for everything wins for moderate-scale workloads.
predict You build your RAG pipeline with an embedding model that produces 3072-dim vectors. Six months later, you upgrade to a model that produces 1536-dim vectors with better MTEB scores. What happens to your existing vector index?

You rebuild it from scratch. Every vector in the old index is in the old model's coordinate system — 3072 dimensions with the old model's geometry. The new model's 1536-dim vectors live in a completely different space. There's no migration path. You re-embed every document with the new model, build a new index, and swap the index atomically via an alias. This is the hidden cost of the embedding-index coupling: upgrading your embedding model is a full re-index operation. Plan your pipeline accordingly — keep the raw documents, not just the vectors, because you'll need to re-embed someday.

Hybrid search: vectors + keywords

Pure vector search fails in a specific way: it misses exact matches. "Error code E10052" should match a document containing "E10052" exactly, but the embedding of "E10052" might be close to embeddings of other error codes, pulling in irrelevant results. Hybrid search fixes this: run both a dense (vector) search and a sparse (BM25 keyword) search, merge the results. Dense catches semantic similarity; sparse catches exact tokens. Fusion is typically done via reciprocal rank fusion (RRF) or a learned scorer.

This is why the vector DB is rarely the whole story. A production RAG system in 2026 typically runs a vector index + a keyword index + a cross-encoder reranker + metadata filters. The vector search is the engine; the rest is the steering and brakes.

the commoditization Vector search is becoming a checkbox feature in existing databases, not a standalone category. PostgreSQL (pgvector), Elasticsearch, Redis, and MongoDB all ship vector search now. The pure-play vector DBs (Pinecone, Qdrant, Milvus) are competing on filtering performance, distributed scale, and developer experience — not on "we support vectors." For most teams, pgvector is the right starting point, and you graduate to a dedicated vector DB only when ANN search becomes your primary workload and you're hitting PostgreSQL's scaling limits.
