MVCC: How Databases Avoid Locks

Why your read doesn't block someone else's write in Postgres — and the version-chain trick that makes it possible.

You're reading from a database while someone else is writing to the same row. In a naive system, the writer locks the row, your read waits, and the application slows to a crawl under concurrent load. You could add read locks — now readers block writers — and you've traded one bottleneck for another.

Multiversion Concurrency Control (MVCC) solves this by keeping multiple versions of each row simultaneously. A writer creates a new version; readers continue reading the old version until their transaction completes. No locks, no waiting. Readers see a consistent snapshot of the database as it existed at the moment their transaction began, even while writers are actively modifying it.

MVCC trades storage for concurrency: keep old versions around so readers never have to wait for writers.

How Postgres does it

Every row in Postgres carries four hidden system columns:

  • xmin: The transaction ID that created this version of the row.
  • xmax: The transaction ID that deleted or updated this version (0 if still live).

An UPDATE doesn't modify the existing row. It sets xmax on the old version (marking it as dead to future transactions) and inserts a new version with a higher xmin. The old version physically remains in the table until VACUUM reclaims it.

An UPDATE in Postgres, through the MVCC lens
Before:  [xmin=100 | xmax=0  | id=7 | balance=40]
UPDATE accounts SET balance = 50 WHERE id = 7;  -- runs in txn 150

After:   [xmin=100 | xmax=150 | id=7 | balance=40]  ← old version, marked dead
         [xmin=150 | xmax=0   | id=7 | balance=50]  ← new version, current

A reader running in a transaction that started before transaction 150 will see the old row (xmin=100, xmax=0 at the time its snapshot was taken). A reader in a transaction that started after 150 commits will see the new row. Each transaction gets a snapshot of the database that ignores versions created after it began and versions deleted before it began (with some subtlety around in-flight transactions).

Snapshot isolation: what you actually get

MVCC provides snapshot isolation — the I in ACID's "I" word, and the source of most confusion about what databases guarantee. Under snapshot isolation:

  • You read a consistent snapshot of committed data as of your transaction's start time.
  • Two concurrent transactions can modify different rows without conflict.
  • Two concurrent transactions can modify the same row — the first to commit wins, the second gets a serialization error (in Postgres, at REPEATABLE READ and SERIALIZABLE levels).
predict Alice and Bob both read row 7 (balance = 40) at the same time. Alice updates it to 50. Bob updates it to 30. Alice commits first. What happens to Bob's transaction?

Bob gets a serialization error (Postgres: "could not serialize access due to concurrent update"). Bob's transaction saw the row at xmax=0, but when he tries to commit, xmax is now 150 (Alice's transaction). Bob's update was based on a version of the row that's no longer the latest. The database refuses to commit, because if it did, Alice's change would be silently overwritten — a lost update. The application is expected to retry Bob's transaction: re-read the row (now showing balance = 50), re-compute the update (perhaps 50 − 10 = 40, if Bob was debiting 10), and re-attempt the commit.

This is the sharp edge of MVCC: you get non-blocking reads, but you pay with the possibility of serialization failures that your application must handle. Most ORMs and frameworks retry automatically (with exponential backoff), but the failures exist whether you see them or not.

MVCC across the database landscape

Nearly every modern relational database uses MVCC, but the implementations differ in ways that matter:

  • Postgres: Stores old versions in the table itself (heap). VACUUM cleans up dead rows. The WAL logs changes for recovery. The table can bloat with dead rows under heavy write load — VACUUM is not optional, it's load-bearing infrastructure.
  • MySQL/InnoDB: Stores old versions in a separate undo log (rollback segments). The table itself only holds the latest version. Reads reconstruct old versions by applying undo records. This is cleaner for the table but means even a read might need to chase undo chains if the row has been heavily modified.
  • Oracle: Similar to InnoDB's undo approach — the original commercial MVCC implementation and still one of the most sophisticated.
  • SQL Server: Added MVCC relatively late (Snapshot Isolation in 2005, Read Committed Snapshot as the default in Azure). Uses a version store in tempdb.
  • CockroachDB / Spanner: MVCC with hybrid-logical clocks instead of transaction IDs. Versions are identified by timestamps, not integers, because in a distributed database there's no central transaction ID counter.

The vacuum problem (and why it's interesting)

Postgres's MVCC design — old versions live in the table — means dead rows accumulate. Without VACUUM, a heavily-updated table grows unboundedly. VACUUM scans the table, marks dead rows as reusable, and updates the visibility map so that index-only scans can skip heap fetches for pages where all rows are visible to all transactions.

The tricky part: VACUUM can only reclaim a dead row when no active transaction can still see it. If you have a long-running query (a reporting job running for hours), its snapshot keeps old versions alive. VACUUM can't clean them up, the table bloats, and the bloat slows down queries, which makes them run longer, which makes bloat worse. This feedback loop — long-running queries → bloat → slower queries → more bloat — is the signature failure mode of Postgres MVCC. The fix is either shorter transactions, aggressive autovacuum tuning, or periodic maintenance windows.

when not MVCC If your workload is append-only (time series, event logs), MVCC's version overhead is pure waste — every row is immutable, so there's only ever one version. An LSM-tree or a log-structured engine fits better. If your workload is single-writer (a queue processing one message at a time), the concurrency MVCC provides is unused but you still pay the VACUUM tax. And if your data is small enough to fit in memory and your concurrency is low, a simpler WAL + in-memory store is both faster and less operationally complex.
