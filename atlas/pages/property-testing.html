Property-Based Testing

Stop writing examples; describe what must always be true — and let the machine attack it with ten thousand random inputs.

A unit test checks one example: assert add(2, 3) == 5. If add has a bug that manifests only for negative numbers, or large numbers, or zero, the test passes and the bug ships. You can add more examples — add(0, 0), add(-1, 1), add(INT_MAX, 1) — but you're guessing at the interesting inputs. The test suite covers the examples you thought of, not the space of possible inputs.

Property-based testing inverts the relationship: you specify a property that must hold for all inputs, and the framework generates random inputs to test it. Instead of add(2, 3) == 5, you write: "for all integers a and b, add(a, b) == a + b (the reference implementation), add(a, b) == add(b, a) (commutativity), and add(a, 0) == a (identity)." The framework generates thousands of random a, b pairs and checks all properties. If it finds a counterexample, it shrinks it — finds the simplest input that still fails — and reports: "add(-1, INT_MIN) failed: overflow."

You specify what should always be true. The machine finds the edge cases where it isn't.

What makes a good property

A good property is a mathematical invariant — something that should hold for all inputs, independent of implementation details:

  • Round-trip: decode(encode(x)) == x — serialization, compression, encoding. The classic property: any value should survive a round trip.
  • Idempotence: f(f(x)) == f(x) — sorting a sorted list, normalizing normalized text, applying a database migration that's already been applied. Doing it twice should be the same as doing it once.
  • Commutativity / associativity: merge(a, b) == merge(b, a) — CRDT merges, set operations, mathematical operations where order shouldn't matter.
  • Inverse: add(x, negate(x)) == 0 — operations that have an inverse. Addition/subtraction, push/pop (on a stack, pop(push(x)) returns x).
  • Oracle / model: Compare against a simpler reference implementation. The optimized B-tree should return the same results as a linear scan. The compiled regex should match the same strings as an interpreted version.
  • Metamorphic: If you change the input in a known way, the output should change in a predictable way. sort(list.append(x)) == sort(list).insert(x) — inserting before sorting should be equivalent to inserting into a sorted list at the right position.

Shrinking: the magic that makes property testing useful

When a property test fails, the failing input might be a 500-element list with deeply nested structure. The framework shrinks it — iteratively simplifies the failing input while checking that it still fails. A 500-element list shrinks to a 3-element list. An integer of 847293874 shrinks to 0 or -1. The reported counterexample is the simplest input that triggers the bug.

This is what makes property testing practical for debugging. A failing unit test says "your code breaks on input X, which you thoughtfully provided." A failing property test says "your code breaks on this specific minimal input, which the machine discovered by trying thousands and distilling the failure to its essence." The shrinking is the difference between "somewhere in this 500-element list is a problem" and "your sort function breaks on a 2-element duplicate list [3, 3]."

predict You write a property test for a JSON parser: for any string s, toString(parse(s)) should equal s. The test fails — but only for very specific inputs. Before reading on: what kind of string breaks this property, and is it a bug in the parser or the property?

Several cases break it. Leading zeros in numbers: parse("01") might return 1, and toString(1) returns "1" — not equal. Whitespace: parse(" 42 ") strips whitespace, toString(42) returns "42". Unicode escapes: parse("\"\\u00e9\"") produces "é", toString("é") produces "\"é\"". These are property failures, not parser bugs — the property is too strong. The parser is correct; the normalization is expected behavior. The fix: refine the property to "for any valid JSON value, parse(toString(x)) == x" (the other direction: serialize then parse should be identity). Or use a canonicalization function in the property: canonicalize(toString(parse(s))) == canonicalize(s). Getting the property right is the hard part of property-based testing.

The tools

  • Haskell: QuickCheck — the original (1999). Every property testing library descends from it.
  • Python: Hypothesis — the gold standard for dynamic languages. Generators, shrinkers, and stateful testing (test sequences of operations on a stateful system).
  • Java: jqwik — property-based testing on the JVM, with composable generators.
  • Rust: proptest — property testing with Rust's type system, generating values from Arbitrary trait implementations.
  • Go: rapid — Go's standard library property testing package (testing/quick is the older, simpler version).
when not to property-test Property testing finds bugs in functions with well-defined mathematical properties: parsers, serializers, sorters, data structure operations, compression, encoding. It's less useful for side-effectful code (I/O, database calls), code whose correctness is defined by human judgment rather than invariants ("this UI looks right"), and code where the properties are so complex to specify that they're effectively a second implementation. The property is the test oracle. If specifying the property is harder than writing the code, property testing isn't the right tool.

Threads worth pulling

Throwing garbage at your code — fuzzing is property testing's sibling: randomly generated inputs without explicit properties, looking for crashes rather than assertion failures.

Testing your tests — mutation testing verifies that your property tests actually catch bugs by injecting them deliberately.
