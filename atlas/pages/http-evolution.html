HTTP/1.1 → 2 → 3

What each version actually fixed, why HTTP/3 left TCP behind, and the 30-year evolution of the protocol that delivers the web.

HTTP/1.1 was standardized in 1997. It's still the fallback for every web request made in 2026. The fact that a protocol designed for documents with inline images still works for interactive SPAs, video streaming, and real-time APIs is either a testament to good design or evidence that the web has been dragging around legacy baggage for three decades. Both are true.

Each HTTP version solves the most painful problem of its predecessor, and that pain tells you more about how the web changed than any feature list.

HTTP/1.1: one request at a time (per connection)

The problem HTTP/1.1 solved was connection overhead. HTTP/1.0 opened a new TCP connection for every request — image, stylesheet, script. TCP's three-way handshake + slow start meant each connection spent more time setting up than transferring data. HTTP/1.1 added persistent connections (keep-alive): reuse the same TCP connection for multiple requests. This was a massive improvement and is still the default behavior.

But HTTP/1.1 had a structural limit: head-of-line blocking. A request must fully complete before the response to the next request on the same connection can begin. If request 1 is a slow database query, requests 2–6 queue up behind it. Browsers worked around this by opening 6 parallel connections per origin — a hack that became a de facto standard. But 6 connections means 6 TCP handshakes, 6 congestion control states, and 6× the connection overhead.

HTTP/2 fixed head-of-line blocking at the HTTP layer. HTTP/3 fixed it at the transport layer. The difference is everything.

HTTP/2: multiplexing over one TCP connection

HTTP/2 (standardized 2015, now used by ~70% of websites) introduced stream multiplexing: many requests and responses interleaved over a single TCP connection. Each request gets a stream ID; frames on different streams are multiplexed onto the wire; the receiver reassembles them. One TCP connection, many concurrent requests, no head-of-line blocking at the HTTP level.

HTTP/2 also brought header compression (HPACK), server push (the server can send resources before the client asks — now largely deprecated because it was hard to get right), and binary framing (HTTP/1.1 is text; HTTP/2 is binary, which is faster to parse).

The catch: TCP itself still imposes ordering. If a TCP packet is lost, every stream on that connection stalls until the packet is retransmitted. HTTP/2 fixed head-of-line blocking in the application layer but inherited it from the transport layer. On a lossy network (mobile, congested WiFi), HTTP/2 can be slower than HTTP/1.1 with 6 connections — one lost packet on the single TCP connection blocks everything, whereas with 6 connections only one stalls.

HTTP/3: QUIC replaces TCP

HTTP/3 (standardized 2021, ~35% of websites as of 2026) makes the biggest architectural change in HTTP's history: it replaces TCP with QUIC, a transport protocol built on UDP. The key insight: TCP's in-order delivery guarantee is the problem. QUIC delivers streams independently — a lost packet on stream 5 doesn't block streams 1, 2, 3, or 4. Each stream has its own loss recovery.

QUIC folds TLS 1.3 into the transport layer. There's no separate handshake — the QUIC handshake includes key exchange, so you get encryption and transport setup in one round trip (0-RTT for returning connections). Connection migration is built in: a QUIC connection is identified by a Connection ID, not by IP:port. Switch from WiFi to cellular, and the connection survives — the server sees the new IP, recognizes the Connection ID, and continues. This is invisible to the HTTP layer.

QUIC also eliminates the TCP ossification problem. TCP's wire format is so widely understood by middleboxes (firewalls, NATs, load balancers) that evolving it is nearly impossible — any new TCP option gets stripped or blocked by some box in the path. QUIC runs over UDP and encrypts almost everything (including metadata like packet numbers), so middleboxes see opaque UDP datagrams. The protocol can evolve without asking permission from every network operator between the client and server.

predict You deploy HTTP/3 behind your load balancer. Users on corporate networks report your site is inaccessible. Your site works fine on home connections. What's the most likely cause?

UDP port 443 is blocked. QUIC runs over UDP, and many corporate firewalls block UDP by default (or restrict it to DNS on port 53). The fix is both technical and operational: ensure your deployment advertises HTTP/3 via the Alt-Svc header (which tells the browser "this origin also supports HTTP/3 at this endpoint"), and the browser will attempt QUIC, fall back to HTTP/2 over TCP on failure, and remember the outcome. Users on UDP-blocked networks get HTTP/2 — slightly slower but functional. This graceful degradation is load-bearing: without it, HTTP/3 adoption would be near zero because network operators can't be trusted to unblock UDP.

The numbers that matter

On a low-loss network (data center, good WiFi), HTTP/3 is 10–20% faster than HTTP/2. On a lossy network (mobile, 1%+ packet loss), HTTP/3 is 30–60% faster. The gains come almost entirely from per-stream loss recovery: one lost packet doesn't freeze the whole page. This is why HTTP/3 adoption correlates with mobile traffic — the more mobile users you have, the more HTTP/3 matters.

the CPU tax QUIC runs in userspace, not in the kernel. TCP has decades of kernel optimization (GSO, GRO, TSO, hardware offload); QUIC is catching up. On a high-throughput server, QUIC can consume 2–3× more CPU than TCP for the same throughput. This is improving (kernel-bypass QUIC stacks, hardware offload), but as of 2026, HTTP/3 deployment is a trade: better user experience on lossy networks for higher server CPU cost. For an internal service running on a 10GbE LAN with near-zero packet loss, HTTP/2 is faster and cheaper. For a global CDN serving mobile users, HTTP/3 is a clear win.
