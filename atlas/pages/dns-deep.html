DNS Is a Distributed Database

The naming system you forget about until it takes down half the internet — and the most successful distributed database ever built.

Every time you load a web page, your browser makes a DNS query. That query traverses a hierarchy of servers, any of which might have the answer cached. The system handles billions of queries per second, tolerates misconfigured servers, survives cable cuts and DDoS attacks, and has been running continuously since 1985. If you proposed building a distributed database with DNS's architecture today, you'd be laughed out of the design review. And yet it works, and it teaches you more about distributed systems than most "proper" databases do.

The DNS tree: a hierarchy, not a flat namespace

DNS names form a tree. The root is . (dot). Beneath it are the top-level domains (.com, .org, .uk). Beneath those are second-level domains (example.com). And so on. Each level delegates authority to the level below it: the root servers don't know where www.example.com lives; they know which servers are responsible for .com, and those servers know which servers are responsible for example.com.

DNS is a tree of delegations, and resolution is walking from the root down to the leaves — one level at a time.

This is a hierarchical design, not a flat key-value store. It means you can add a domain without telling any central authority (you tell your TLD's registry, which is one step up the tree). It means a misconfigured .com server doesn't affect .org lookups. And it means the system scales horizontally by adding more levels, not more entries to a single database.

What actually happens during a lookup

When you type www.example.com into a browser, a resolver (usually your ISP's recursive resolver, or a public one like 8.8.8.8 or 1.1.1.1) walks the tree on your behalf:

A DNS resolution, step by step
Client → Recursive resolver: "what's the A record for www.example.com?"

Resolver → Root server (.):        "who handles .com?"
Root → Resolver:                    "here are the .com nameservers"

Resolver → .com TLD server:        "who handles example.com?"
.com → Resolver:                    "ns1.example.com (IP: 93.184.216.34)"

Resolver → ns1.example.com:        "what's the A record for www.example.com?"
ns1 → Resolver:                     "www.example.com → 93.184.216.34"

Resolver → Client:                  "93.184.216.34"

Each step is a DNS query — typically a single UDP packet (53 bytes of payload, though EDNS0 extends this). If a UDP response is too large, the resolver retries over TCP. In practice, most responses fit in a single UDP datagram, and the whole resolution takes 20–100ms.

The resolver caches at every level. If someone else just looked up example.com, the .com nameserver's response is already in the resolver's cache, so the walk starts from step 3. The TTL on each record tells the resolver how long to cache it — from seconds (for records about to change) to days (for stable infrastructure).

The root servers: 13 identities, hundreds of machines

There are 13 root server identities, named a.root-servers.net through m.root-servers.net. The number 13 is a UDP artifact: a DNS response listing all 13 root servers fits in a 512-byte UDP packet without fragmentation. In reality, each of these 13 identities is served by dozens or hundreds of machines distributed globally, using anycast routing — the same IP address announced from multiple locations, with BGP routing your query to the nearest instance.

Your query to a.root-servers.net doesn't go to one computer in Virginia. It goes to the nearest of ~80 instances, likely within your continent, sometimes within your city. Anycast is what makes the root server system survive DDoS attacks: an attacker can saturate one instance, but the rest of the world routes around it, and the specific instance under attack is only serving users in that region.

This is the physical reality underneath the abstract hierarchy: a distributed, anycast-routed, UDP-based query system that journalists call "the internet's phone book" and engineers call "the distributed database that accidentally works."

predict You're the DNS administrator for example.com. You need to change the IP address of www.example.com from 93.184.216.34 to 93.184.216.100. You update the zone file and reload the nameserver. But half your users still reach the old IP two hours later. What did you forget?

The TTL. When you published the old record, it had a TTL — say, 3600 seconds (1 hour). Every recursive resolver that looked up www.example.com during the last hour cached that result for the full TTL duration. Changing the authoritative record doesn't invalidate those caches; they'll serve the stale IP until their TTL expires. If you know a change is coming, the standard practice is to lower the TTL in advance (e.g., from 3600 to 300 seconds), wait for the old TTL to expire, make the change, then raise the TTL back. DNS propagation is not magic — it's TTL expiry, and it's entirely under your control.

Why DNS is a better distributed database than you think

DNS has properties that most distributed databases only aspire to:

  • Read scalability. Caching is built into the protocol (TTL on every record). The authoritative servers only handle cache misses — the actual query volume is absorbed by resolvers. A popular domain might be cached a million times for every one query that reaches the authoritative server. This is a protocol-level cache, not an afterthought.
  • Fault isolation. The hierarchy means a failure in one subtree doesn't affect others. If .io's nameservers go down, .com resolutions continue unaffected. There's no global failure mode — only per-zone failures.
  • Eventual consistency. DNS is eventually consistent by design. There's no transaction across the hierarchy, no global ordering, no consensus protocol. You update your zone; resolvers see the change when their cached TTL expires. The "inconsistency window" is explicit and bounded (the TTL), not a bug you discover in production.
  • Data sovereignty. The .uk registry is operated by Nominet, a UK organization, under UK law. The .cn registry is operated by CNNIC, under Chinese law. The hierarchy maps to political boundaries where it matters and abstracts over them where it doesn't. Few distributed databases have a story for jurisdiction that's this clean.

And why it's also terrible

the costs DNS is unencrypted by default. Every DNS query you make is visible to your ISP, your coffee shop's WiFi operator, and anyone on the network path. DNS over HTTPS (DoH) and DNS over TLS (DoT) encrypt the resolver-to-client leg, but the resolver-to-authoritative leg is still plaintext. DNS responses are not authenticated unless you're using DNSSEC, which adoption remains patchy for (it's cryptographically sound but operationally painful — key rollovers and signed zone maintenance add real operational complexity). And the 13-root-identities architecture, while elegant, is a concentration of control that periodically becomes the subject of geopolitical tension. The protocol is 40 years old, and every extension since has been a backwards-compatible hack. The result is a system that works remarkably well and is remarkably hard to evolve.

The practical bits for an engineer

The DNS record types you'll actually encounter:

  • A / AAAA: Maps a name to an IPv4 (A) or IPv6 (AAAA) address. The workhorse record.
  • CNAME: "This name is an alias for that name." www.example.com → example.com. The resolver follows the chain and returns the final IP. CNAMEs can't coexist with other record types at the same name (except DNSSEC records).
  • MX: Mail exchanger — which server handles email for this domain, with a priority value. Lower priority = tried first.
  • TXT: Arbitrary text. Used for SPF (email sender verification), DKIM keys, domain ownership verification ("add this random string to your TXT record to prove you control the domain"), and a dozen other things that were bolted onto TXT because it was the extensibility escape hatch.
  • NS: Delegates a subdomain to a nameserver. This is the record that structures the hierarchy.
  • SOA: Start of Authority — metadata about the zone: the primary nameserver, the admin's email, and timing parameters (refresh, retry, expire, and the negative TTL — how long to cache "that name doesn't exist").

One detail that bites people: the difference between a CNAME and an A record at the apex. You can't put a CNAME at the root of a zone (example.com) because the SOA and NS records are already there, and a CNAME can't coexist with other records. If you want your bare domain to resolve to the same place as www, you either use an A record (with a fixed IP) or an ALIAS/ANAME record (a non-standard extension that some DNS providers support, which acts like a CNAME but resolves at query time on the authoritative server). This single constraint has spawned thousands of "why isn't my bare domain working" debugging sessions.
