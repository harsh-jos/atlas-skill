Consistent Hashing

How to add a server without reshuffling everything you stored — the hash-ring trick behind CDNs, distributed caches, and Dynamo-style databases.

You have four memcached servers. Keys are distributed by hash(key) % 4. One server gets hot — memory's full, CPU's pegged. You add a fifth.

hash(key) % 5 maps nearly every key to a different server. Your cache hit rate, which was 94%, drops to roughly 20% — the hit rate you'd get from random assignment. The database behind the cache, sized for the 6% of requests that normally miss, is now fielding 80% of all traffic. It falls over. You added capacity and caused an outage.

This is the rehashing storm, and the root cause isn't that modulo is dumb. It's that the mapping function depends on the exact count of servers. Change the count, change everything.

The ring

Consistent hashing, introduced by Karger et al. in 1997, changes the mapping so that adding or removing a server only moves the keys that logically belonged to that server — roughly 1/N of the total.

Instead of mapping keys to servers by modulo, map both keys and servers onto a circle — the hash ring. The circle wraps around at 264 (or 2160, depending on the hash function). Every key, every server, gets a position on this ring. A key is assigned to the first server encountered moving clockwise from the key's position.

When a server joins, it inserts itself at its hashed position. It inherits keys from only one neighbor — the server clockwise from it. All other key-to-server mappings stay unchanged. When a server leaves, its keys migrate to its clockwise neighbor. In a cluster of N servers, roughly 1/N of the key space moves on a membership change.

Modulo reassigns nearly every key when the count changes. The ring reassigns roughly one Nth of them. That's the entire idea, and it pays for itself a thousand times over.

The uneven ring — and the fix

There's a catch. Hash four servers onto a 264-sized ring and you'll get four points at roughly random positions. The arcs between them — the key ranges each server owns — will be wildly different sizes. One server might get 42% of the ring; another, 8%. The load distribution is terrible.

The fix is virtual nodes: each physical server gets many positions on the ring, not just one. Hash server-A-1, server-A-2, ..., server-A-256 — 256 points for server A, 256 for server B, and so on. By the law of large numbers, each server's total arc length converges toward its fair share. More virtual nodes → more even distribution → more memory to store them. A few hundred vnodes per server is typical.

Virtual nodes also solve a subtler problem: heterogeneous hardware. If server A has twice the capacity of server B, give A twice as many virtual nodes. The ring distributes load proportionally without any special-case code.

What real systems do with it

Discord's voice servers. When Discord was growing fast, they needed to move voice channels between servers without dropping calls. They built a consistent hashing ring: each voice server registers at many points on the ring, and each channel is assigned to the nearest server clockwise. When a server is drained for maintenance, its channels slide to the next server on the ring — no call dropped, no client reconnected. They wrote about it in 2017, and it's still the architecture.

Dynamo-style databases. Amazon's Dynamo (2007) used consistent hashing to partition data across its storage ring. Each key maps to a coordinator node, which replicates the data to the next N-1 clockwise nodes. When a node joins or leaves, only its immediate neighbors need to shuffle data. Cassandra and Riak inherited this design directly.

CDNs and reverse proxies. NGINX's hash directive with consistent maps client IPs or URLs to upstream servers. HAProxy's balance uri with consistent hashing does the same. When a backend goes down, only the requests hashing to that backend are reassigned — the rest of the cache stays warm.

predict You have 5 servers, each with 200 virtual nodes, on a consistent hashing ring. You add a 6th server, also with 200 vnodes. Before calculating: roughly what fraction of the total key space moves? And why is the answer not exactly 1/6?

The expected fraction is 1/6 — the new server claims roughly 1/6 of the ring from its clockwise neighbors. But the realized fraction varies: with 1,000 total vnodes (5 × 200), the standard deviation is small but not zero. And if the new server's vnodes cluster badly (unlikely with a good hash, but possible), it could claim more or less. In practice, 200+ vnodes per server makes the deviation negligible. The deeper answer is that the new server takes keys from its clockwise neighbors only — servers on the opposite side of the ring are untouched — so the fraction of keys that move is 1/(N+1) on average, not the fraction of servers.

Where it sits on the map

Consistent hashing depends on hash tables for the hashing primitive — the same hash functions that distribute keys into buckets distribute keys onto a ring. It's a cousin to sharding: consistent hashing is one strategy for sharding (hash-based, with minimal reshuffling); range-based sharding is another (alphabetical, numerical — predictable, but hot-spot prone).

It's the opposite of a directory-based approach, where a centralized mapping table records which server owns which key. The directory gives perfect control (move any key anywhere) at the cost of a lookup and a SPOF. Consistent hashing gives no central state at the cost of imperfect control.

And it shares a deep concern with gossip protocols: both answer "how does a distributed system know who's alive and who owns what, without a central authority?" In a Dynamo-style ring, gossip spreads the ring membership; consistent hashing determines what each member owns.

The costs

Distribution is approximately fair, not exactly fair. Even with 256 vnodes per server, the standard deviation of load is about 5–10%. For a cache, this is fine — one server running at 55% while another is at 45% is invisible. For a storage system where disks fill up, the unevenness can trigger early resharding. Some systems (like Redis Cluster) use a fixed 16,384 hash slots rather than a continuous ring precisely to bound this variance.

Range queries don't work. Neighboring keys on the ring have unrelated hashes — user:alice and user:bob land on opposite sides. If you need to scan user:*, consistent hashing won't help; you'll touch every server. Range partitioning (alphabetical, numerical) supports scans but breaks the "add a server without reshuffling" property. Pick your poison.

Membership changes are cheap, not free. When a server leaves, its keys migrate to its clockwise neighbor — ~1/N of the data. If that neighbor was already near capacity, it now has ~2/N of the data and might tip over. The 1/N guarantee is about the key space; the load shift is often worse because the departing server's load is concentrated onto one neighbor, not spread evenly. Production systems smooth this by moving data gradually in the background (Cassandra's "nodetool decommission" drains a node slowly rather than yanking it from the ring).

when not to If your cluster size is small and static — say, 3 database replicas behind a load balancer — modulo or a static mapping is simpler and doesn't need virtual nodes. If you need strict range scans across keys, reach for range partitioning instead. And if you need the mapping to survive a split-brain (where two halves of the cluster each think the other is dead), consistent hashing alone isn't enough — you need a membership protocol with fencing tokens on top.
