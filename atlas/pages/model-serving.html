Model Serving: Batch vs Online

The architecture choice that decides your latency and your bill — and the inference engines that make LLM serving a distinct discipline.

Training a model is a batch job: you submit it, it runs for hours or days, you get weights. Serving is the other half — making those weights available to answer queries — and it's where most of the cost and complexity of production ML lives. A model that costs $10K to train might cost $100K to serve over its lifetime. The architecture choices you make about serving determine whether that number is $50K or $500K.

Batch inference: maximize throughput

Batch inference is the simple case: you have a fixed dataset (a million images to classify, a million documents to embed), no latency requirements, and the goal is to process everything as quickly and cheaply as possible. You load the model, feed it the data in large batches, and write the results to storage. The optimization is purely about tokens-per-second per GPU and cost-per-token.

Batch inference uses the same model weights as online inference but different infrastructure. You don't need a server that stays up 24/7 — you need a job that runs, scales to as many GPUs as your budget allows, and shuts down. Ray Data, Spark ML, and vLLM's offline mode are the standard tools. The cost optimization is straightforward: use spot/preemptible instances, pack batches to GPU memory limits, and choose the smallest model that meets your accuracy requirements.

Batch inference is about throughput. Online inference is about latency. The same model, same weights, radically different infrastructure.

Online inference: minimize time-to-first-token

Online inference means a user is waiting. Every millisecond of latency is a degraded experience. The goal isn't maximum throughput — it's meeting a latency SLO (typically p95 < 200ms time-to-first-token, p95 < 50ms time-per-output-token for LLMs) while handling variable, bursty traffic.

The key technique is continuous batching: instead of waiting for a fixed batch to fill before running inference, the server dynamically adds new requests to the currently-running batch at each iteration. Requests that finish (reaching their end token or max length) are removed; new requests are added. This is the single biggest throughput improvement in LLM serving — it can increase throughput by 3–10× over static batching, because the GPU is never idle waiting for a slow request to finish its generation.

The other key technique: PagedAttention (from vLLM). The transformer's KV cache — the stored keys and values from previous tokens that attention needs — is the dominant memory consumer during generation. Traditional implementations allocate contiguous memory for each sequence's KV cache up to the maximum sequence length, wasting 60–80% of allocated memory (most sequences are shorter than the max). PagedAttention allocates KV cache in non-contiguous blocks (like virtual memory pages), reducing waste to under 4%. This means you can fit 2–3× more concurrent requests on the same GPU.

The practical default in 2026: start with vLLM. It's the easiest to deploy, has the largest community, and its throughput is competitive. Migrate to TensorRT-LLM if you're on homogeneous NVIDIA hardware and need to squeeze every token-per-second from each GPU. Consider SGLang if your workload has heavy prefix reuse (many requests sharing the same system prompt or RAG context).

predict You're serving a 70B model with continuous batching. At low concurrency (5 concurrent users), time-per-output-token is 30ms. At high concurrency (100 users), it's 80ms. What's the bottleneck, and what's the architectural fix that doesn't involve more GPUs?

The bottleneck is likely the KV cache competing for GPU memory. At high concurrency, more sequences means more KV cache, and the working set might exceed the GPU's memory bandwidth. The fix: quantize the KV cache to FP8 or INT8, cutting its memory footprint by 50% with minimal quality loss. Or use prefix caching (SGLang's RadixAttention, vLLM's automatic prefix caching): if multiple requests share a system prompt or RAG context, the KV cache for that shared prefix is computed once and reused. For agentic workloads with long shared prefixes, this can improve throughput 2–5× without adding GPUs.

The disaggregated future

An emerging architecture (2025–2026): separate the prefill (processing the input prompt, compute-bound) from decode (generating output tokens, memory-bound) onto different GPU pools. Prefill GPUs are optimized for high FLOPs; decode GPUs for high memory bandwidth. You can scale them independently — more prefill GPUs during traffic spikes, more decode GPUs for long-generation workloads. SGLang and vLLM both have disaggregated serving modes in 2026. It's not the default yet, but it's the direction of travel.

the real cost LLM serving is expensive not because inference is slow — it's because GPUs are idle between requests. A typical production LLM service has 30–60% GPU utilization. The rest is waiting for the next request. Continuous batching helps (fill the idle time with other requests), but the fundamental economics are: you're renting a GPU 24/7, and it's doing useful work half the time. The cost optimization frontier is about increasing utilization: multi-model serving (multiple fine-tuned variants sharing the same GPU), spot instances for batch inference, and request scheduling that smooths out bursty traffic. The biggest cost reduction in LLM serving in 2026 isn't a faster engine — it's better utilization of the engines you already have.

Threads worth pulling

Making models smaller before serving — quantization & distillation is the preprocessing step that reduces the model's memory footprint, which directly determines how many requests fit on a GPU.

The data pipeline feeding the model — feature stores ensure the features used in training are available at serving time, in the same format, with the same semantics.
