Public-Key Cryptography

How two strangers agree on a secret in public, while everyone is watching — and why this single trick underpins everything from TLS to SSH to cryptocurrency.

Before 1976, cryptography had an obvious problem. If Alice and Bob want to communicate privately, they first need to share a secret key. But how do they share that key? If they have a secure channel to share the key, they could just use that channel to send the message. If they don't, an eavesdropper can intercept the key and read everything.

This is the key distribution problem, and for most of history it was assumed to be inherent. Governments used couriers with briefcases handcuffed to their wrists. It was expensive, slow, and fragile — but there was no mathematical way around it.

Then Diffie and Hellman published a paper called "New Directions in Cryptography," and the assumption collapsed.

The asymmetry trick

Instead of one shared key, use a key pair: a public key that anyone can know, and a private key that never leaves its owner.

The public key can encrypt messages, but only the private key can decrypt them. The private key can sign messages, and anyone with the public key can verify the signature. This inverts the symmetric model: you don't need a secure channel to establish a secure channel. You just need to know someone's public key — which can be broadcast in the open — and suddenly you can send them messages only they can read.

This requires a mathematical function that's easy to compute in one direction but hard to invert unless you know a secret. These are trapdoor functions, and there are surprisingly few of them that have survived cryptanalysis.

Diffie-Hellman: agree on a secret in public

The simplest trapdoor is the discrete logarithm problem: computing gˣ mod p is easy; recovering x from gˣ mod p is hard when p is a large prime. Diffie-Hellman key exchange uses this to let two parties derive a shared secret while an eavesdropper sees every message:

Diffie-Hellman key exchange
Public parameters: a large prime p, and a generator g (both public)

Alice:                                     Bob:
1. picks random a (keeps secret)           3. picks random b (keeps secret)
2. computes A = g^a mod p → sends to Bob   4. computes B = g^b mod p → sends to Alice
5. receives B, computes s = B^a mod p      6. receives A, computes s = A^b mod p

Both now share s = g^(ab) mod p. Eve saw g, p, A, B. But from A = g^a mod p,
Eve can't recover a (discrete log). Without a or b, she can't compute g^(ab).

This is elegant, simple, and widely deployed. Every TLS 1.3 handshake begins with a Diffie-Hellman key exchange (specifically its elliptic-curve variant, ECDHE). The "E" stands for ephemeral — new keys per session, so compromising one session's key doesn't compromise past or future sessions. This property is called forward secrecy, and it's why TLS 1.3 dropped static RSA key exchange entirely.

RSA: encrypt with one key, decrypt with the other

RSA (Rivest-Shamir-Adleman, 1977) uses a different trapdoor: multiplying two large primes is easy; factoring their product back is hard.

RSA, conceptually
Key generation:
  1. Pick two large primes p, q. Compute n = p × q.
  2. Compute φ(n) = (p-1)(q-1).
  3. Pick e (commonly 65537) such that gcd(e, φ(n)) = 1.
  4. Compute d = e⁻¹ mod φ(n)  (the modular inverse).
  Public key: (n, e).  Private key: (n, d).

Encryption:  c = m^e mod n
Decryption:  m = c^d mod n

Why it works: (m^e)^d = m^(ed) = m^1 = m mod n   (by Euler's theorem)
Why it's secure: given (n, e), you can't compute d without knowing φ(n),
which requires knowing p and q — i.e., factoring n.

RSA is used for both encryption and digital signatures. A signature is just encryption in reverse: the signer computes hash(message)^d mod n (using the private key), and the verifier checks signature^e mod n == hash(message) (using the public key). If the check passes, only the holder of the private key could have produced that signature.

predict You receive a message signed with someone's private key. That proves the message came from them — but does it prove they intended to send it? Before reading on: what's the attack where a valid signature exists but the signer never saw the message?

This is a blinding attack (or, more broadly, the class of attacks where the signer signs something they don't understand). If Mallory can trick Alice into signing a carefully crafted blob that happens to be the hash of "Alice transfers 100 BTC to Mallory," the signature is mathematically valid but Alice was signing a different message in her mind. This is why real protocols sign structured data with context ("I am Alice, at time T, signing message M for purpose P"), not raw hashes. It's also why "just sign the hash" is dangerous advice.

Elliptic curves: smaller keys, same security

RSA keys are big. A 2048-bit RSA key provides roughly 112 bits of security. An elliptic curve key providing the same security is 224 bits — almost an order of magnitude smaller. The trapdoor is the elliptic curve discrete logarithm problem: given points P and Q = k·P on a carefully chosen elliptic curve, recovering k is hard.

ECDSA (Elliptic Curve Digital Signature Algorithm) is the signature scheme behind Bitcoin transactions, TLS certificates, and SSH keys. Ed25519 (an Edwards-curve variant) is increasingly the default for new systems: it's faster than ECDSA, immune to several classes of implementation errors (no per-signature randomness needed; deterministic nonces), and has a compact 32-byte public key. If you're generating an SSH key in 2026, it should be Ed25519.

What public-key crypto doesn't do (and what fills the gaps)

Public-key operations are slow — thousands of times slower than symmetric encryption. No one encrypts a 1 GB video with RSA. The pattern in practice is hybrid encryption: use public-key crypto to agree on a symmetric key (or encrypt a small symmetric key), then use AES or ChaCha20 for the bulk data.

This is how TLS works: the handshake uses ECDHE for key agreement and Ed25519/ECDSA for server authentication. Once the handshake completes, the resulting session keys are symmetric (AES-GCM or ChaCha20-Poly1305), and the rest of the connection runs at line speed with authenticated encryption. The public-key part is the expensive bootstrap; the symmetric part is the cheap steady state.

Also, public-key crypto doesn't tell you whose public key you have. If Mallory intercepts your connection and substitutes her public key for Bob's, you're encrypting messages to Mallory while thinking it's Bob. This is the man-in-the-middle problem, and public-key crypto solves it not with math but with infrastructure: certificate authorities, the Web PKI, and the trust anchor model. The math gives you a secure channel to whoever holds that private key; certificates tell you who that is.

the quantum asterisk Shor's algorithm (1994) can factor integers and compute discrete logs in polynomial time on a quantum computer. This breaks RSA, Diffie-Hellman, ECDSA, and Ed25519 — essentially all deployed public-key crypto. Symmetric crypto (AES-256) is largely safe (Grover's algorithm gives a quadratic speedup, so double the key size). The field replacing public-key crypto is post-quantum cryptography (PQC), and NIST standardized the first PQC algorithms (Kyber for key encapsulation, Dilithium for signatures) in 2024. TLS libraries are already shipping hybrid key exchange (ECDHE + Kyber) so that connections are secure against both classical and future quantum adversaries. It's not a today-emergency, but it's a "start planning your migration" signal.
