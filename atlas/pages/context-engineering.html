Context Engineering

The discipline that replaced "prompt engineering" once context windows got big — managing what the model sees, not just how you ask.

In 2023, the skill was prompt engineering: how to phrase your question so the model answers correctly. The context window was 4K tokens. You thought about word choice, delimiters, few-shot examples. By 2025, the context window was 200K tokens — and the skill shifted to context engineering: what information to put in the window, in what order, with what structure, at what level of detail. When the context window is large enough to hold entire codebases, the problem stops being "how do I ask?" and becomes "what do I include?"

Prompt engineering is about the question. Context engineering is about the evidence the model reads to answer it.

The four pillars

Context engineering breaks into four concerns:

  1. Instructions (system prompt): Who the model is, what constraints it operates under, what format it should output. The system prompt is the constitution — it should be stable, version-controlled, and tested. Start minimal (a few sentences), add constraints only in response to observed failures.
  2. Retrieval: What external information enters the context per-query. RAG is the mechanism; context engineering is the discipline of deciding what to retrieve, how much, and where to place it. A focused 300-token retrieval chunk often outperforms an unfocused 3,000-token chunk — the model's attention is a scarce resource, and every irrelevant token dilutes it.
  3. Memory: What the model remembers across turns in a conversation, and across sessions. Short-term memory is the conversation history (managed by summarization and compaction — old turns are summarized and removed when the context window fills). Long-term memory is what persists across sessions (user preferences, past interactions, facts the user has explicitly asked the model to remember). Memory management is the hardest unsolved problem in context engineering.
  4. Tools: What the model can do beyond generating text. Each tool's description occupies context tokens. A bloated tool set (20+ MCP servers, 50+ functions) saturates the context and degrades performance. The rule: fewer, more precise tools outperform many vague ones. Limit active tools to the set genuinely needed for the current task.

The attention budget

The most important concept in context engineering: the model's attention is a finite resource, and every token in the context competes for it. The "lost in the middle" phenomenon (Liu et al., 2023) showed that LLMs attend most strongly to the beginning and end of the context. Information in the middle is less likely to influence the output. A 201-token context and a 200K-token context have the same shape of attention: U-shaped, with peaks at the edges and a valley in the middle.

Practical implications:

  • Put the most important information at the start (system prompt, critical constraints) or the end (the most recent conversation turn, the most relevant retrieved document).
  • If you're stuffing a context with 50 retrieved documents, the ones in positions 20–30 are effectively invisible. Retrieve fewer, better documents, and position them thoughtfully.
  • Token budgeting: a 300-token focused context often outperforms a 113K-token unfocused one. Context length is a cost, not a feature — you pay for every token in latency and attention dilution.
predict Your coding agent has a CLAUDE.md that's grown to 800 lines over six months. The agent's performance has degraded — it ignores instructions that used to work. You trim CLAUDE.md to 40 lines. What specific mechanism explains why this helps?

Attention dilution and the lost-in-the-middle effect. An 800-line instructions file means the model's critical behavioral constraints are buried in a haystack of context. The model's attention at the instruction-rich positions (start and end) is spread across too many competing directives. Trimming to 40 lines puts every instruction in the high-attention zone, and the model can actually follow all of them because they're not competing with each other for a finite attention budget. The general rule: your instructions file should contain only what the model fails at without being told. If the model already gets it right without the instruction, the instruction is context debt.

Conversation compaction

Conversations grow. A coding session with 50 turns and large tool outputs can reach 100K tokens. At that point, new turns are competing with old turns for attention, and the model starts forgetting what happened near the beginning. Compaction is the solution: periodically summarize the conversation so far, replace the full history with the summary, and continue. The summary preserves the decisions and state (what we did, why, what the current state is) while discarding the process (the specific tool calls, the intermediate outputs, the debugging dead ends).

Compaction is lossy compression for conversations. The art is deciding what to keep: errors the model should learn from, decisions that constrain future choices, facts the model will need to reference later. What to discard: successful tool calls whose results are no longer relevant, debugging detours that were resolved, verbose output that's been consumed.

The harness layer

By 2026, the frontier has moved beyond context to harness engineering: the system surrounding the model that enforces correctness regardless of what the model generates. A harness can:

  • Verify the model's output against a schema before returning it to the user
  • Run the model's generated code through a linter/type-checker and feed errors back as context
  • Enforce that the model cannot call certain tools or access certain paths, regardless of what the user's prompt says
  • Track the model's trajectory across a session and detect loops, stalls, or degradation

The insight: the model is one component in a system. Context engineering manages what goes into it. Harness engineering manages what comes out. Together, they're the engineering discipline that turns a raw LLM into a reliable product.

the discipline Context engineering is invisible when it works and catastrophic when it fails. A well-engineered context is short, structured, and focused. A poorly engineered context is long, meandering, and the model's output degrades in ways that are hard to attribute to the context — you just see lower quality and don't know why. The fix is the same as for any engineering discipline: version your system prompts, A/B test changes, and measure the output quality quantitatively. A system prompt is code. Treat it like code.
