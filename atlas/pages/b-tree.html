B-Trees

The 50-year-old data structure under almost every relational index — and why the disk seek, not the CPU, is its real architect.

In 1971, a disk seek took about 50 milliseconds. Reading a block from disk was slow enough that you'd rearrange your entire data structure around avoiding it. Rudolf Bayer and Edward McCreight did exactly that. The result was the B-tree, and it still runs every major relational database.

You don't hear about B-trees much because they work so well they're invisible. When you create an index in Postgres, you get a B-tree. When MySQL stores a table with a primary key, it's a B-tree. When MongoDB looks up a document by _id, it walks a B-tree. The data structure that was designed for spinning rust in 1971 turns out to work remarkably well on SSDs in 2026, because the fundamental asymmetry — sequential access is cheap, random access is expensive — is still true.

Why not a binary search tree?

If you've learned data structures, your instinct is: use a balanced BST. O(log₂ n) lookups, simple code. The problem isn't the algorithm — it's the hardware.

A balanced BST with 10 million rows has a height of about 24 levels (log₂ 10⁷ ≈ 24). Each level is a pointer dereference to a different node. In memory, those nodes are scattered everywhere — every step is a random memory access. In memory, that's maybe tolerable. On disk, 24 random seeks at 5ms each is 120 milliseconds for a single lookup.

A B-tree with the same 10 million rows and a branching factor of 500 has a height of 3 (log₅₀₀ 10⁷ ≈ 2.8). Three seeks instead of twenty-four. That's the entire trick.

A B-tree is a balanced search tree with a very high branching factor, designed so that each node fits exactly in one disk block.

What a B-tree node actually contains

Each node is an array of sorted keys with pointers sandwiched between them:

Internal node (branching factor 4):
      [ptr₀] key₀ [ptr₁] key₁ [ptr₂] key₂ [ptr₃]
         │            │            │            │
         ▼            ▼            ▼            ▼
    subtree with  subtree with  subtree with  subtree with
    keys < k₀    k₀ < keys < k₁  k₁ < keys < k₂  keys > k₂

The number of keys per node is governed by a parameter d (the minimum degree). Every node except the root has between d-1 and 2d-1 keys. The root can have as few as 1 key. This guarantee is what keeps the tree balanced: every leaf is at the exact same depth, always.

When you pick d so that each node fills exactly one disk page (typically 4KB to 16KB), you get the maximum branching factor for free. With 16KB pages and 32-byte key+pointer pairs, you get ~500 keys per internal node. The tree stays shallow even for billions of rows.

Insertion: splitting on the way down

The answer is node splitting, and the crucial insight is doing it preemptively on the way down.

As you descend the tree looking for the insertion point, whenever you encounter a full node, you split it. The median key moves up to the parent, and the remaining keys are divided into two new siblings. This guarantees that by the time you reach the leaf where the insertion happens, that leaf's parent has room. You never need to propagate a split back upward, because you already made space on the way down.

Deletion is the reverse: you ensure a node never drops below d-1 keys by borrowing from a sibling or merging on the way down. Again, one pass, no backtracking.

Why B+ trees won in practice

The classic B-tree stores keys and data (or data pointers) in every node — internal and leaf. The B+ tree, which every major database uses, stores data only in the leaves. Internal nodes contain just keys and child pointers.

This has three consequences that turn out to matter enormously:

  1. More keys per internal node. Without data pointers competing for space, internal nodes hold more keys → higher branching factor → even shallower trees.
  2. Range scans are cheap. All the data is in the leaves, and the leaves are linked together in sorted order. WHERE x BETWEEN 100 AND 200 finds the first leaf, then walks the linked list. In a classic B-tree, you'd be bouncing between internal and leaf nodes the whole way.
  3. Consistent performance. Every lookup goes all the way to a leaf. There's no "lucky" hit in an internal node. This makes performance predictable, which matters more for an index than being fast on average but erratic.

The B+ tree leaf layer often contains the actual table rows (in a clustered index) or pointers to them (in a secondary index). In Postgres, secondary indexes store the primary key value in their leaves, and a second B+ tree lookup on the primary key fetches the row — a double B-tree walk that's fast enough because the trees are so shallow.

B-trees vs LSM-trees: the storage engine divide

The B-tree's main competitor is the LSM-tree, and the trade-off is a clean one:

  • B-trees optimize for reads. The data is in one place, sorted, with O(log n) random reads. But writes are expensive — you're modifying pages in place, which means random I/O. The write-ahead log turns those random writes into a sequential append for durability, but the data file writes are still random.
  • LSM-trees optimize for writes. Writes go to an in-memory buffer and a sequential WAL. Reads might need to check multiple files (memtable + multiple SSTables), which is more I/O than a B-tree read. But the writes are pure sequential — no in-place modification.

The rule of thumb: if your workload is read-heavy (or reads and writes are balanced), use a B-tree. If it's write-heavy (logging, time series, event ingestion), use an LSM-tree. Most OLTP databases pick B-trees; most log-structured storage engines pick LSM.

watch out B-trees suffer from write amplification in a subtle way. Updating a single row might dirty an entire page. Checkpointing writes that whole page to disk, even if only one byte changed. SSDs wear out from write cycles, and B-tree page writes are larger and more frequent than you'd expect from the logical write rate. Most databases mitigate this with a background writer that batches dirty-page flushes, but if you're running on write-limited storage, measure this before assuming the B-tree is free.

A structure that outlived its era

The B-tree was designed for hard drives with moving heads. SSDs eliminated the seek time but preserved the page abstraction — SSDs still read and write in blocks. B-trees map naturally to the block device interface. Even persistent memory (Intel Optane, though discontinued, and its successors) can be addressed byte-by-byte, but the B-tree's real strength isn't about the physical medium. It's about the log-structured property of keeping data sorted under mutations. That's a mathematical need, not a hardware one.

For a structure that's been in production since before C existed, the B-tree has aged remarkably well. It's a reminder that the most durable ideas in computer science aren't the cleverest — they're the ones that understood the constraints correctly and made the right trade, for long enough that the trade became invisible.
