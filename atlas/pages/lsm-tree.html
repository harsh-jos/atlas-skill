Log-Structured Merge Trees

Why modern databases write fast by never writing in place — and the merge dance that happens while you're not looking.

A B-tree writes hurt. Updating a single row means reading the page containing it, modifying it in memory, and eventually writing the whole page back to disk. That's a random write, and random writes are slow — even on SSDs, where the gap between random and sequential is smaller but still real. For write-heavy workloads (time series, event logs, telemetry), B-trees become the bottleneck.

An LSM-tree (Log-Structured Merge Tree) inverts the strategy. Instead of modifying data in place, it only appends. Incoming writes go to a fast in-memory buffer (the memtable). When the memtable fills, it's flushed to disk as a sorted, immutable file (an SSTable — Sorted String Table). Writes are purely sequential: append to a WAL for durability, build the memtable in memory, flush to disk as a new file. No seeking, no page modification, no write amplification from in-place updates.

Write fast by appending. Read by merging. Let compaction clean up in the background.

The anatomy of a write and a read

A write in an LSM-tree takes microseconds:

  1. Append the operation to the WAL (sequential write, for durability).
  2. Insert into the memtable — an in-memory sorted data structure, typically a skip list or a red-black tree. Return success to the client.
  3. When the memtable reaches its size threshold (typically a few MB to tens of MB), freeze it, flush it to disk as a new SSTable, and start a new memtable.

A read, however, may need to check multiple places:

  1. Check the active memtable. Found it? Done (the fastest path).
  2. Check recently flushed (still uncompacted) SSTables.
  3. Check older, compacted SSTables — each level containing progressively more data.
  4. To avoid scanning every SSTable for every read, each SSTable has a Bloom filter that can cheaply answer "this key is definitely not in this file." If the bloom filter says no, skip the file.

This is the trade: writes are fast because they're pure append. Reads are slower because they potentially search through multiple files. An LSM-tree is optimized for writes, and you mitigate the read cost with bloom filters, compaction, and caching.

Compaction: the merge that keeps reads fast

If you just kept appending SSTables forever, reads would slow to a crawl (hundreds of files to check per lookup) and disk space would be consumed by obsolete keys. Compaction is the background process that merges SSTables, discards overwritten or deleted keys, and produces a smaller set of sorted files.

Leveled compaction (RocksDB, the most common strategy)
Level 0:  freshly flushed SSTables, no order between files, ~4 files max
Level 1:  sorted, non-overlapping SSTables, ~10 MB total
Level 2:  sorted, non-overlapping SSTables, ~100 MB total
Level N:  sorted, non-overlapping SSTables, ~10× larger than N-1

When level L exceeds its size limit:
  Pick one SSTable from level L
  Find overlapping SSTables in level L+1
  Merge-sort them all together, discard deleted/overwritten keys
  Write the result as new SSTables in level L+1

Compaction is the hidden cost of the LSM-tree. It's a background I/O and CPU consumer. A burst of writes fills up Level 0 quickly; compaction races to merge them into lower levels before reads have to search through too many files. If writes consistently outpace compaction, you get write stall — the database deliberately slows or pauses writes until compaction catches up. This is the LSM-tree's signature failure mode: "my database was fast until it wasn't, and then it stopped accepting writes entirely."

LSM vs B-tree: the write amplification showdown

Both structures amplify writes — each logical write to the database becomes multiple physical writes to disk — but in different ways:

  • B-tree write amplification: Updating one byte in a 16KB page writes the entire page to the WAL (one write) and eventually the data file (another write). That's 32KB of physical I/O for a one-byte logical write — amplification of 32,000× in the worst case.
  • LSM-tree write amplification: A write goes to the WAL (one write), the memtable flush (one write per SSTable, amortized across many inserts), and then compaction rewrites it multiple times as it moves down levels. Each key may be rewritten 5–15 times before it reaches the bottom level.

The crossover point depends on the workload. For write-heavy workloads (insert-only with few updates), LSM-trees win because appends are cheap and compaction is mostly copying data forward, not modifying it. For update-heavy workloads (lots of overwrites), B-trees can win because they modify in place — the old value doesn't need to be compacted away later. For read-heavy workloads, B-trees win outright: one read, one file, no merging.

predict You're storing 100 GB of time-series data — a sensor reading every second, keyed by (device_id, timestamp). The data is written once and read by time-range queries. B-tree or LSM-tree?

LSM-tree, and strongly so. The writes are append-only (no updates, no deletes), so compaction doesn't need to discard anything — it's just copying data forward to maintain the level structure. The reads are range scans over sorted keys, which SSTables serve efficiently (they're sorted files — a range scan reads contiguous blocks). A B-tree would pay random-write costs on every insert with no benefit, since the data is never modified in place. TimescaleDB (a time-series extension for Postgres) and InfluxDB's storage engine are both LSM-backed, confirming this analysis in production.

Where LSM-trees run the world

The LSM-tree is the storage engine under:

  • RocksDB — Meta's embeddable key-value store, used inside MySQL (MyRocks), Kafka Streams, and half the NoSQL databases in existence.
  • LevelDB — Google's original, now mostly superseded by RocksDB but still powering Chrome's IndexedDB implementation.
  • Cassandra / ScyllaDB — The wide-column databases that power Discord, Apple, and Netflix. LSM-trees with leveled or size-tiered compaction.
  • HBase / Bigtable — Google's original LSM-tree database, the paper that introduced the idea (Chang et al., 2006).
  • WiredTiger — MongoDB's default storage engine since 3.2. An LSM-tree with a B-tree option.
  • InfluxDB (TSM engine) — A time-series-optimized LSM variant.
the tuning trap RocksDB has so many tuning knobs (compaction strategy, level size multiplier, bloom filter bits per key, memtable format, compression algorithm, block cache size, rate limiter...) that you can spend a career tuning it. The defaults are good for general workloads. The tuning is for when your specific workload is pathological for the defaults — typically: write stalls, read latency spikes during compaction, or memory pressure from oversized memtables. Start with the defaults, measure, and only touch the knobs when you can articulate which specific problem you're solving.
