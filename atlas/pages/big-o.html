Big-O & Amortized Analysis

Why "it works on my machine" and "it works at scale" are claims made by different people — the lens that separates algorithms that scale from ones that don't, and the accounting trick that makes worst-case analysis honest.

An engineer pushes a change: a lookup that walks a flat list. Works fine in dev with 50 items. Production has 200,000. The endpoint goes from 2 ms to 14 seconds. The code is correct. The algorithm is wrong.

Big-O is the language that would have caught this before a line was written. But the version most engineers carry around — "O(n) bad, O(log n) good" — strips out the parts that actually matter for making decisions.

What Big-O actually measures

Big-O describes how a function's resource consumption grows as the input size grows. It's not about absolute speed — it's about the shape of the growth curve. An O(n²) algorithm on a supercomputer will eventually lose to an O(n log n) algorithm on a phone, given enough input.

Formally: f(n) = O(g(n)) means there exist constants C and n₀ such that f(n) ≤ C · g(n) for all n ≥ n₀. In English: beyond some input size, g(n) is an upper bound on f(n), times some constant. It's an asymptotic upper bound — it describes behavior at scale, not at n=5.

The standard shapes, in order of growth:

O(1) — constant. The cost doesn't change with input size. Array indexing, hash table lookup (average case), stack push. The holy grail, and rarer than it looks — a "constant time" operation that hashes a 500-byte string is doing more work than one that hashes an 8-byte integer, but neither grows with N.

O(log n) — logarithmic. Each operation cuts the search space in half. Binary search, balanced tree lookup, finding a key in a B-tree. The curve flattens so fast it's effectively constant for any real N — log₂(1,000,000) = 20. A million-item binary search costs ~20 comparisons.

O(n) — linear. Cost grows proportionally to input. Scanning an unsorted list, summing an array, copying a buffer. Respectable and often optimal — if you must inspect every element, you can't beat O(n).

O(n log n) — linearithmic. The product of linear and logarithmic. Comparison-based sorting lives here (merge sort, heap sort, Timsort). It's the theoretical lower bound for comparison sort, and it's fast enough for nearly everything.

O(n²) — quadratic. Nested loops over N. Bubble sort, insertion sort, checking all pairs. Tolerable for N < ~1,000. At N=10,000, it's 100 million operations — you feel it. At N=1,000,000, it's unusable.

O(2ⁿ) — exponential. Each additional input element doubles the work. Brute-force subset problems, naive recursion for Fibonacci. At n=30, it's a billion operations. At n=100, the universe ends first. These are problems where you need dynamic programming, approximation, or a different formulation.

The only numbers that matter are the ones that grow when N grows. Everything else is a constant factor, and constant factors are what profiling is for.

The hidden parts: Ω, Θ, and the constant that kills you

Big-O is an upper bound. It says "no worse than." The other two are used less often but complete the picture:

Ω (Omega) is a lower bound — "no better than." Comparison-based sorting is Ω(n log n): you cannot sort by comparing elements without doing at least n log n comparisons. Θ (Theta) is a tight bound — "exactly this growth rate." Merge sort is Θ(n log n): both the upper and lower bounds are n log n.

And then there's the constant factor. Two O(n) algorithms: one does 3n operations, the other does 10,000n. Both are O(n). The 10,000n one might be slower than an O(n²) algorithm for small N. Big-O tells you what happens at scale; the constant tells you whether "at scale" matters for your actual N. This is why linear search beats binary search for small arrays (~ < 50 elements) — binary search's O(log n) has a higher constant (branch mispredictions, non-sequential access) that doesn't pay off until N crosses a threshold.

Amortized analysis: when the worst case is a liar

Hash table insert is O(1) on average — except when it resizes, which is O(n). If you claim O(1) for insert, you're either ignoring resize or lying by omission. Amortized analysis resolves this honestly: spread the cost of the expensive operation across all the cheap ones that led up to it.

The classic case is a dynamic array (Python's list, Java's ArrayList, Go's slice). Appending is O(1) — until the array is full and must be reallocated and copied, which is O(n). If you double capacity each time, resize happens at sizes 1, 2, 4, 8, 16, 32, ... The total cost of n appends is: n cheap appends + (1 + 2 + 4 + 8 + ... + n/2) copies ≈ n + n = 2n. Amortized per-append: O(1). The occasional O(n) resize doesn't change the average.

Hash table insert with resize is amortized O(1) by the same argument. The key insight: the expensive operations are rare enough that their cost, spread out, is still constant per operation.

predict A dynamic array grows by adding 10 slots each resize instead of doubling. n appends — what's the amortized cost per append now? And why does every real implementation double instead?

With fixed-increment growth, resizes happen at n/10, 2n/10, 3n/10, ... The total copy cost is roughly n²/20 — amortized O(n) per append. Doubling gives O(1); fixed-increment gives O(n). The ratio of the growth policy determines the amortized complexity. This is not a detail — it's why garbage-collected languages with growable arrays double rather than add-a-few.

Where it sits on the map

Big-O is the analytical lens you apply to every other topic on this map. It's how you know a hash table lookup is O(1) average, O(n) worst; how you see that a B-tree lookup is O(log n) with a high fan-out making the constant tiny; how you understand that consistent hashing key reassignment is O(1/N) of the key space per membership change.

It's not a topic you learn once and move on from — it's a habit of thinking about growth curves that compounds with every new data structure and algorithm you encounter.

watch out Big-O makes two assumptions that reality doesn't: uniform cost per operation (RAM model) and infinite addressable memory. A hash table lookup is O(1) in the RAM model; on real hardware, cache misses make some O(1) operations 100× more expensive than others. And an algorithm that's O(n) in time but O(n²) in memory will exhaust RAM long before it exhausts your patience. The theory tells you the shape of the curve; profiling tells you the constant, and the constant sometimes wins.
