Loss Functions

The number a model is actually trying to make smaller — and why the choice of loss encodes everything you care about.

A model doesn't "know" what you want. It knows what you measure. The loss function is the measurement — a single number that summarizes how badly the model is doing on the task you care about. Gradient descent minimizes it. Backpropagation computes its gradient with respect to every weight. The loss function is the objective; everything else is the machinery to reach it.

If the loss function is wrong, the model optimizes for the wrong thing. A classifier that minimizes classification error but ignores the cost of false negatives (missing a tumor) is optimizing the wrong objective. A language model that minimizes perplexity but produces toxic text is optimizing the wrong objective. The loss function is where your values enter the math — and choosing it well is the most consequential decision in the modeling pipeline.

The basic taxonomy

Regression asks "how far off?" Classification asks "how confident was the wrong answer?"

For regression (predicting a continuous value):

  • MSE (Mean Squared Error): (y_true − y_pred)². Penalizes large errors quadratically — a prediction off by 10 is 100× worse than a prediction off by 1. Best when large errors are genuinely much worse than small ones. Sensitive to outliers — one terrible prediction dominates the loss.
  • MAE (Mean Absolute Error): |y_true − y_pred|. Penalizes errors linearly. More robust to outliers than MSE. Best when you care about the typical error, not the worst-case. The gradient is constant magnitude — doesn't get smaller as you approach the minimum, which can cause oscillation near convergence.
  • Huber loss: Quadratic for small errors, linear for large errors. The best of both — MSE's smooth convergence near the minimum, MAE's outlier robustness far from it. Has a tunable delta parameter that sets the crossover point.

For classification (predicting a category):

  • Cross-entropy (log loss): −log(p_correct). If the model assigns 0.9 probability to the correct class, the loss is −log(0.9) ≈ 0.105. If it assigns 0.1, the loss is −log(0.1) ≈ 2.30. The logarithm means that confident wrong answers are punished severely. This is the standard loss for essentially all classification tasks — from logistic regression to GPT's next-token prediction.
  • Binary cross-entropy: The two-class special case. For each example: −[y·log(p) + (1−y)·log(1−p)]. If the true label is 1 and the model predicts 0.99, the second term vanishes and loss is tiny. If it predicts 0.01, the loss is large. This is the loss behind every binary classifier.

Why cross-entropy won

Cross-entropy has a mathematical property that makes it the default choice: when combined with a softmax output layer, the gradient of the loss with respect to the logits simplifies to p̂ − y — predicted probability minus true label. This is stunningly clean. The gradient is just the error vector. No chain rule explosion, no vanishing terms, no numerical instability.

MSE with a softmax, by contrast, produces a gradient that includes p̂(1−p̂) — which vanishes when the model is confident (p̂ ≈ 0 or p̂ ≈ 1). A confidently wrong model produces tiny gradients and stops learning. This is the vanishing gradient problem, and it's why you should never use MSE for classification with softmax. The cross-entropy + softmax combination is not a convention — it's a derivation that avoids a known failure mode.

The LLM loss: next-token cross-entropy

When an LLM trains, the loss at each position is the cross-entropy between the model's predicted probability distribution over the vocabulary and the actual next token (a one-hot distribution where the correct token has probability 1). The model's loss over a sequence is the average of these per-token losses:

Language model loss (next-token prediction)
For each position i in the sequence:
    loss_i = -log(P(token_{i+1} | tokens_{1..i}))
Total loss = average(loss_i over all positions)

The perplexity you see in model reports is exp(total_loss). A perplexity of 10 means the model is, on average, as uncertain as if it were choosing uniformly from 10 equally likely tokens. Lower perplexity = better predictions. This single number — next-token cross-entropy averaged over a held-out corpus — is the objective that produced every major language model.

predict You're training a classifier to detect fraud (rare — 0.1% of transactions). You use standard cross-entropy loss. The model achieves 99.9% accuracy by predicting "not fraud" for every transaction. What's wrong, and what's the loss-function fix?

The model learned that accuracy is optimized by ignoring the rare class. The fix is a weighted loss: multiply the loss for fraud examples by a factor (e.g., 1000×) so that each fraud example contributes as much to the total loss as 1000 non-fraud examples. This forces the model to care about fraud detection. A more sophisticated fix is focal loss (Lin et al., 2017): down-weight the loss for examples the model already classifies correctly, so the model focuses on the hard cases. In a fraud setting, the model quickly learns to classify non-fraud correctly, and focal loss automatically shifts the training signal to the fraud examples — without needing to manually set the class weight. The general principle: the loss function should encode the cost of being wrong, not just the fact of being wrong.

the RLHF twist The cross-entropy loss trains a model to mimic the training data. But "mimic the internet" is not the same as "be helpful, harmless, and honest." RLHF (Reinforcement Learning from Human Feedback) replaces the loss function with a learned reward model that predicts human preferences. The model is no longer optimizing "predict the next token accurately" — it's optimizing "produce outputs that a human judge would rank highly." The loss function changes from a mathematical measure of prediction accuracy to a learned proxy for human values. This is why RLHF is philosophically interesting beyond its engineering details: it's the point where the objective stops being objective.
