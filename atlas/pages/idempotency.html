Idempotency

The one property that makes "just retry it" a safe instruction — and the key pattern that turns unreliable networks into reliable systems.

A payment request times out. The client retries. The server processes both requests — charging the user twice. This is the most expensive bug class in distributed systems: the double charge. The network failure was transient, but the duplicate processing is permanent. And it could have been prevented with a single property: idempotency.

An operation is idempotent if performing it multiple times has the same effect as performing it once. SET x = 5 is idempotent — running it twice still sets x to 5. x += 5 is not — running it twice adds 10. In HTTP, GET, PUT, and DELETE are idempotent by specification. POST is not. But "idempotent by specification" doesn't mean "the server correctly handles duplicates" — it means the HTTP method's semantics are idempotent. Your PUT handler can still be non-idempotent if it increments a counter internally.

Idempotency is the contract that makes retries safe. Without it, every timeout is a potential correctness bug.

Idempotency keys: the implementation pattern

The standard approach: the client generates a unique idempotency key (a UUID) for each logical operation and sends it with every request (including retries). The server stores the key along with the response. If the server receives a request with a key it has already processed, it returns the stored response — no double charge, no duplicate side effect.

Idempotent payment, with a key
Client:                          Server:
POST /charges                    1. Received key "abc-123"
Idempotency-Key: abc-123         2. Check: key "abc-123" → not found
{amount: 50}                     3. Process charge (create txn 42)
                                 4. Store: key "abc-123" → {txn_id: 42, status: ok}
                                 5. Return: {txn_id: 42, status: ok}

[timeout — client retries]
POST /charges                    6. Received key "abc-123"
Idempotency-Key: abc-123         7. Check: key "abc-123" → found!
{amount: 50}                     8. Return stored response: {txn_id: 42, status: ok}

The critical detail: step 3 (process the charge) and step 4 (store the key) must be atomic. If the server processes the charge, crashes before storing the key, and the client retries, the charge is processed twice. The fix: store the key in the same database transaction as the operation's side effects. If the database supports unique constraints, the idempotency key is a unique column, and inserting a duplicate key fails — the server catches the constraint violation and returns the stored response.

The subtlety: concurrent retries

predict Two retries of the same request arrive at the server simultaneously. Both check the key store — neither finds the key. Both process the operation. Both try to store the key. What prevents the double charge?

The database's unique constraint on the idempotency key. The first INSERT succeeds. The second INSERT fails with a unique constraint violation. The handler for the second request catches the violation, fetches the now-stored response (which the first request stored atomically with its INSERT), and returns it. No double charge. But this requires that the idempotency key store and the operation's side effects are in the same database transaction — otherwise, the first request could succeed but the key INSERT fails, or vice versa. The idempotency key is not a separate service; it's a column in the same database that stores the operation's results.

This is also why idempotency keys should be scoped to the resource: a key for "create payment for order 42" should be different from a key for "create payment for order 43," even if the client accidentally reuses the same UUID. The standard pattern: the key is a combination of the client's request ID and the resource identifier, ensuring uniqueness within the scope of the operation.

Where idempotency appears (and where it's missing)

  • Stripe / payment APIs: The canonical example. Every Stripe API accepts an Idempotency-Key header. Keys are stored for 24 hours. Retrying a charge with the same key returns the original result.
  • Kafka: The idempotent producer (enabled by default in modern clients) assigns each message a producer ID and sequence number. The broker deduplicates by (producer ID, sequence number), ensuring exactly-once semantics per partition — an idempotency guarantee at the broker level.
  • S3 / object storage: PUT Object is idempotent — uploading the same object twice results in the same object. But multipart uploads are trickier; incomplete multipart uploads accumulate and must be explicitly aborted.
  • AWS Lambda / serverless: At-least-once invocation means your function may be called multiple times for the same event. Your function must be idempotent. AWS provides the event ID for deduplication, but the burden is on the function to use it.
the gap Idempotency keys prevent double-processing from retries. They don't prevent two different requests from creating conflicting state. If a user clicks "Pay" twice (generating two different idempotency keys), both payments go through — that's a different problem (duplicate submission), solved differently (button debouncing, confirmation modals, idempotency keys scoped to the user's session + action type). Idempotency solves the infrastructure problem (retries). It doesn't solve the product problem (preventing the user from doing a thing twice). Know which one you're solving.
