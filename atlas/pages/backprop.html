Backpropagation

The chain rule, industrialized — how a neural network learns from its mistakes, one derivative at a time.

You've trained a neural network. You fed it data, it made predictions, you computed a loss, and the weights updated. The magic step between "here's how wrong we were" and "here are better weights" is backpropagation. It's not a learning algorithm — it's an efficient way to compute gradients. But without it, training deep networks would be computationally impossible, and the deep learning revolution wouldn't have happened.

At its core, backpropagation is one idea from high school calculus applied with a specific computational pattern. The idea is the chain rule. The pattern is that you compute once and reuse.

Why forward-mode differentiation doesn't scale

Say you have a function f(x, y) = (x + y) × (x − y). You want the partial derivatives ∂f/∂x and ∂f/∂y. With forward-mode autodiff, you track how a small change in one input propagates through the computation. For each input you want the gradient with respect to, you do a full forward pass. A neural network has millions or billions of inputs (the weights). Forward-mode would require one pass per weight. That's non-starter territory.

Reverse-mode autodiff — which is what backpropagation is — starts from the output and works backward, propagating the gradient of the loss with respect to each intermediate value. One reverse pass gives you the gradient with respect to every input simultaneously. For a network with n parameters, forward-mode costs O(n) forward passes; reverse-mode costs one forward pass plus one backward pass. The backward pass is about 2–3× the cost of the forward pass. This is the computational miracle that makes training deep networks feasible.

The chain rule, at scale

Consider a tiny two-layer network:

Input x ──→ [W₁] ──→ a₁ = W₁x    ──→ [ReLU] ──→ h₁ = max(0, a₁) ──→ [W₂] ──→ ŷ = W₂h₁
                                                                          │
                                                                   loss = (ŷ - y)²

The loss depends on W₂ through ŷ, and on W₁ through a₁ → h₁ → ŷ. The chain rule gives us:

∂loss/∂W₂ = ∂loss/∂ŷ · ∂ŷ/∂W₂                           (direct path)
∂loss/∂W₁ = ∂loss/∂ŷ · ∂ŷ/∂h₁ · ∂h₁/∂a₁ · ∂a₁/∂W₁        (through the whole chain)

The key efficiency: notice that ∂loss/∂ŷ appears in both expressions. The backward pass computes ∂loss/∂ŷ once, then reuses it. At each layer, backprop computes the local gradient (how this layer's output depends on its input) and multiplies it by the upstream gradient (the gradient of the loss with respect to this layer's output). This product becomes the upstream gradient for the previous layer.

The backward pass, layer by layer
G = ∂loss/∂ŷ                     # upstream gradient starts at the loss

# Layer 2 (the output layer):
∂loss/∂W₂ = G · ∂ŷ/∂W₂           # gradient for W₂'s update
G = G · ∂ŷ/∂h₁                    # new upstream gradient for layer 1

# Layer 1 (the hidden layer):
∂loss/∂W₁ = G · ∂h₁/∂a₁ · ∂a₁/∂W₁  # gradient for W₁'s update

Each layer only needs to know two things: how to compute its own local gradient (the backward function), and the upstream gradient it received. It passes the product backward and the chain rule does the rest. This is why frameworks like PyTorch and JAX only need you to define the forward pass — they can trace the computation graph and derive the backward pass automatically.

The computation graph is the secret

Every mathematical expression is a directed acyclic graph. Backprop is just a traversal of that graph in reverse topological order.

When you write y = W2 @ relu(W1 @ x), PyTorch doesn't execute it directly. It builds a computation graph — a DAG where nodes are operations (matrix multiply, ReLU, add) and edges are the data flowing between them. During the forward pass, each node stores any intermediate values it'll need for the backward pass (this is why you need more memory during training than inference — those stored intermediate activations, called the activation checkpoint or "saved tensors").

During the backward pass, the graph is traversed from the loss node backward to the input nodes. Each node receives the gradient of the loss with respect to its outputs and computes the gradient with respect to its inputs. By the time the traversal reaches the weight nodes, you have ∂loss/∂W for every parameter in the network.

This is also why backprop requires differentiable operations. The backward function for ReLU is simple: ∂max(0, x)/∂x = 1 if x > 0, 0 otherwise. For matrix multiply, it's a transpose multiply. For cross-entropy loss with softmax, the backward pass simplifies to a surprisingly clean ŷ − y (predicted minus true). Every operation in a neural network needs a defined backward function. If you've ever seen No gradient defined for operation, it means you used something in the forward pass that the framework doesn't know how to differentiate through.

predict A ReLU unit outputs 0 for every input in a batch. What's the gradient of the loss with respect to its input weights? Before reading on: what does this mean for those weights, and is it a bug or a feature?

The gradient is 0. The ReLU's backward function returns 0 for any input ≤ 0, so the upstream gradient is multiplied by 0, and the weights feeding that neuron get zero update. This is the dying ReLU problem: once a ReLU neuron receives a large negative gradient that pushes its weights into a region where it always outputs 0, it never recovers — it's permanently dead, contributing nothing to the network. It's a bug in the sense that it reduces capacity, but it's also why variants like Leaky ReLU (small positive slope for negative inputs), PReLU (learnable slope), and GELU (smooth approximation, used in transformers) exist. The GELU in your transformer is, at its heart, "ReLU but we fixed the dying problem."

Why this took decades to matter

Backpropagation was derived in the 1960s (in the control theory community), rediscovered in the 1970s, and made famous by Rumelhart, Hinton, and Williams in 1986. Deep learning didn't take off until 2012. What changed wasn't the algorithm — it was three things backprop needs to work well:

  1. Enough data. Backprop computes gradients from data. With small datasets, the gradients are noisy and the optimization surface is treacherous. ImageNet (2009) gave the field a large enough dataset.
  2. Enough compute. GPUs turned the backward pass from "wait overnight" to "a few seconds." The backward pass is mostly matrix multiplies, which GPUs excel at.
  3. Better initialization and normalization. If your initial weights produce activations that saturate (gradients → 0) or explode (gradients → ∞), backprop can't get a useful signal through. Xavier/Glorot initialization (2010) and BatchNorm (2015) fixed the signal propagation problem, making it possible to train networks deeper than ~10 layers.

Backprop is a beautiful algorithm, but it was waiting on infrastructure for 25 years. The algorithm wasn't the bottleneck — everything around it was.

the memory cost Backprop's dirty secret is memory. To compute the gradient through a layer, you need the layer's forward-pass activations. For a transformer with a 2048-token context, that's gigabytes of intermediate tensors. Gradient checkpointing trades compute for memory: instead of storing all activations, store a few and recompute the rest during the backward pass. This roughly doubles the backward-pass FLOPs but cuts memory by O(√n). Most large-model training uses it. It's a reminder that backprop's O(1)-pass gradient computation is a time win, not a memory win — and when memory is the scarcer resource, you trade time back.
