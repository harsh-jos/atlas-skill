Event Sourcing & CQRS

Storing what happened instead of what is — and reading it back however you need to.

A traditional database stores the current state: a row says balance = 50. When the balance changes to 40, the row is updated to balance = 40. The old value is gone. You can reconstruct that the balance changed (from the WAL or an audit table, if you set one up), but you can't reconstruct why — was it a withdrawal? A fee? A correction? And you can't answer "what was the balance at 3:15 PM last Tuesday?" unless you were snapshotting or logging.

Event sourcing inverts the model. Instead of storing the current state, store the sequence of events that produced it: AccountOpened(id=7), Deposited(100), Withdrew(30), Withdrew(20). The current balance (50) is derived by replaying the events: 0 + 100 − 30 − 20 = 50. The events are the source of truth; the current state is a cached projection.

The database stores the journal, not the ledger. The ledger is a view over the journal.

Why you'd do this

Event sourcing buys you three things that are expensive or impossible with a state-based model:

  1. Complete audit trail. Every state change has a corresponding event with a timestamp, a causation ID, and the full context of what happened. Compliance and debugging are built in — there's no separate audit table to maintain.
  2. Temporal queries. "What was the account balance at 3:15 PM?" Replay events up to that timestamp. "What was the user's shopping cart before they added the coupon?" Revert the CouponApplied event. With event sourcing, time travel is a feature of the architecture, not a bolt-on.
  3. Multiple read models (CQRS). You can project the same events into different shapes for different consumers. The account service reads AccountSummary (balance, status). The analytics service reads AccountActivity (daily aggregates). The fraud service reads SuspiciousTransactionStream. All built from the same events, each optimized for its own read pattern.

The fourth benefit, more subtle: events capture intent. "Withdrew 20" carries more semantic meaning than balance = balance − 20. Domain experts speak in events ("a withdrawal happened"), not state diffs ("the balance column changed"). When the code models the domain's actual language, the gap between requirements and implementation shrinks.

CQRS: separate the write model from the read model

Command Query Responsibility Segregation (CQRS) is the natural companion to event sourcing. The idea: use a different model for writes than for reads.

CQRS with event sourcing
Commands (writes)          Events (append-only)        Queries (reads)
───────────────           ─────────────────────        ───────────────
PlaceOrder           →    OrderPlaced              →   OrderView (for display)
CancelOrder          →    OrderCancelled           →   OrderHistoryView
AddItem              →    ItemAdded                →   DailySalesProjection
                                                       FraudAlertProjection

Commands are validated against business rules, then produce events. Events are appended to the event store (a log). Projections (read models) subscribe to the event stream and update their own denormalized views — one event might update five different projections, each optimized for a specific query. The read side can be eventually consistent (the projection updates a few milliseconds after the event is appended), which is the standard trade: write consistency for read flexibility.

The costs (and they're real)

when not to Event sourcing multiplies complexity. You now have an event store, an event schema (which must evolve — you can't just ALTER TABLE), projections to maintain, and eventual consistency to reason about. A simple CRUD app becomes a distributed system with all the attendant failure modes. Use event sourcing when the domain genuinely needs audit trails, temporal queries, or multiple read models. For a blog, a to-do list, or an internal dashboard, it's overengineering. The test: if "what happened and why?" is the most common question your system answers, event sourcing is a fit. If "what is the current state and give it to me fast?" is the question, a relational database with a few indexes is the better answer.

Specific sharp edges:

  • Event schema evolution. An event is immutable once written. But the schema of Withdrew might change (adding a fee field, splitting into WithdrewCash and WithdrewATM). You need an upcasting strategy — transform old events to the new schema on read, or support multiple versions in the projection code.
  • Rebuilding projections. If you change a projection's logic, you must replay every event from the beginning to rebuild it. For an account with 10 years of history, that's millions of events. The fix: snapshots — periodically save the projection state, then replay only events after the snapshot.
  • Eventual consistency. The read model is a few milliseconds behind the write model. Most applications handle this fine. The ones that don't (a user withdraws money, refreshes, sees the old balance, withdraws again) need stronger consistency — which means waiting for the projection to update before confirming the write, defeating the purpose of CQRS.
predict Your event store has 500 million events. You need to add a new field to the Withdrew event and rebuild the balance projection. The projection currently takes 2 hours to rebuild from scratch. What's the architectural pattern that brings this down to seconds?

Snapshotting. Every N events (or at a schedule — nightly), save the current projection state as a snapshot. On rebuild, load the most recent snapshot, then replay only the events that occurred after it. If you snapshot daily and the rebuild happens mid-day, you replay ~12 hours of events instead of 10 years. The snapshot is purely an optimization — it's derivable from the events — but in practice, it's the difference between an architecture that works and one that's paralyzed by rebuild times.
