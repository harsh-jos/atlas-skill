Transaction Isolation Levels

The spectrum from "fast and slightly wrong" to "correct and slow" — and the specific phenomena each level prevents.

A transaction is supposed to be atomic, consistent, isolated, and durable. Three of those are well-defined. The fourth — isolation — is a spectrum, and the choice of isolation level is the most consequential performance-correctness trade-off in database design. The SQL standard defines four levels. Every database implements them differently. And the standard's definitions are ambiguous enough that two databases at the same nominal level can provide different guarantees.

The phenomena

Isolation levels are defined by which anomalies they prevent. The anomalies are what go wrong when transactions interleave without isolation:

  • Dirty read: Transaction A reads uncommitted data written by transaction B. B rolls back. A has read data that never existed. This is the worst anomaly — A made a decision based on fiction.
  • Non-repeatable read (fuzzy read): Transaction A reads the same row twice. Between the reads, transaction B updates the row and commits. A sees two different values for the same row within the same transaction. The row changed under A.
  • Phantom read: Transaction A runs the same query twice (e.g., SELECT * FROM orders WHERE status = 'pending'). Between the queries, transaction B inserts a new pending order and commits. A's second query returns a row that wasn't in the first — a "phantom." The set of rows changed, not just an individual row.
  • Lost update: A and B both read the same row, both modify it, both commit. The second commit silently overwrites the first. One transaction's work is lost without either transaction seeing the other's changes. (Not in the SQL standard's list but the most common real-world anomaly.)
  • Write skew: A and B read two different rows (e.g., two doctors checking on-call coverage), both see a condition met ("at least one doctor is on call"), both update their respective rows (each goes off call), and now zero doctors are on call. The invariant ("at least one doctor on call") is violated even though neither transaction modified the same row. This is the subtle anomaly that only SERIALIZABLE prevents.
Dirty reads are reading uncommitted data. Non-repeatable reads are individual rows changing. Phantoms are result sets changing. Write skew is invariants breaking across rows.

*Postgres's REPEATABLE READ actually prevents phantom reads (using snapshot isolation). The SQL standard says it shouldn't. This is one of the standard's ambiguities.

READ COMMITTED is the default in Postgres, MySQL/InnoDB, SQL Server, and Oracle. It prevents dirty reads: you only see committed data. It does not prevent non-repeatable reads or lost updates. Most applications run at this level and handle the remaining anomalies in application code (SELECT FOR UPDATE for lost updates, optimistic locking with version columns).

REPEATABLE READ is the default in MySQL/InnoDB (and it prevents more than the standard requires — InnoDB's REPEATABLE READ uses gap locks to prevent phantom reads, making it close to SERIALIZABLE in practice). Postgres's REPEATABLE READ uses snapshot isolation: reads see a snapshot of the database as of the transaction's start time. No locks, but serialization failures (two concurrent updates to the same row) are detected and one transaction is aborted.

SERIALIZABLE is the strongest guarantee: transactions execute as if they were run one at a time (serially), even though they actually run concurrently. Postgres implements this with Serializable Snapshot Isolation (SSI), which detects and aborts transactions that would create a serialization anomaly, without locking. Other databases use strict two-phase locking (S2PL) — locks that can cause deadlocks and contention.

predict Two transactions concurrently transfer $10 between accounts at READ COMMITTED. Both read the source account balance (both see $100). Both compute the new balance ($90). Both write $90. What's the final balance, and what isolation level would have prevented this?

The final balance is $90 — one transfer is lost. This is a lost update: both transactions read the old value, computed a new value, and wrote, with the second write overwriting the first. REPEATABLE READ in Postgres would detect this (serialization failure — "could not serialize access due to concurrent update") and abort one transaction. The application retries the aborted transaction, re-reads the now-updated balance, and computes correctly. Alternatively, at READ COMMITTED, using SELECT ... FOR UPDATE on the source account row locks it, preventing the second transaction from reading it until the first commits — a pessimistic lock that turns the concurrent transactions into sequential ones for the critical rows.

the practical rule Start at READ COMMITTED with explicit locking (SELECT FOR UPDATE, optimistic version columns) for the operations that need stronger guarantees. Upgrade to REPEATABLE READ if you're seeing lost updates that application-level locking can't catch (rare, but possible in high-concurrency update workloads). Use SERIALIZABLE only when you have invariants spanning multiple rows that are genuinely hard to enforce with locks — and measure the abort rate, because SERIALIZABLE means your transactions can fail at commit time and must be retried. For most applications, READ COMMITTED + application-level optimistic locking is the sweet spot.
