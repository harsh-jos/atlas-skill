Hexagonal / Ports & Adapters

Keeping your business logic from caring whether it's behind HTTP or a CLI — and why the shape reminds you that there are always more sides than you think.

Alistair Cockburn drew a hexagon in 2005 and accidentally created one of the most useful and most over-diagrammed ideas in software architecture. The hexagon isn't the point — Cockburn picked six sides because he wanted room to draw ports and didn't want people to think a rectangle implied exactly four. The architecture matters; the shape is a mnemonic.

The idea: your application's core logic should have no knowledge of the outside world. Not HTTP. Not Postgres. Not Kafka. Not a CLI. The core knows about ports — interfaces that define what it needs and what it offers, in its own language. The outside world connects to those ports through adapters — concrete implementations that translate between the core's language and the outside technology's language.

The core is pure. The adapters are the necessary impurity that lets the core interact with things that aren't pure.

Ports: the contract in the middle

A port is just an interface. It's defined in the core's language, using the core's types, and it says nothing about HTTP status codes, SQL dialects, or message formats. There are two kinds:

  • Driving ports (primary): what the outside world calls to drive the application. "Create an order," "cancel a subscription," "get a user's balance." These are use cases, and they're defined as interfaces that the core implements.
  • Driven ports (secondary): what the application calls to get things done. "Save this order," "send this notification," "look up this user's permissions." These are interfaces that the core calls, defined by the core, implemented by adapters.
The shape, in code
// In the core (knows nothing about HTTP or Postgres):
interface OrderRepository {          // ← driven port
    fun save(order: Order): OrderId
    fun findById(id: OrderId): Order?
}

interface NotificationService {      // ← driven port
    fun notifyUser(userId: UserId, message: String)
}

class PlaceOrder(
    private val orders: OrderRepository,
    private val notify: NotificationService
) {                                   // ← driving port (use case)
    fun execute(userId: UserId, items: List): Order {
        val order = Order.create(userId, items)
        orders.save(order)
        notify.notifyUser(userId, "Order ${order.id} placed")
        return order
    }
}

// In the adapter layer (knows about Postgres):
class PostgresOrderRepository(db: DataSource) : OrderRepository {
    override fun save(order: Order) = db.run("INSERT INTO orders ...")
    override fun findById(id: OrderId) = db.run("SELECT ...")
}

// In the adapter layer (knows about HTTP):
class OrderController(repo: OrderRepository, notify: NotificationService) {
    @Post("/orders")
    fun create(req: CreateOrderRequest): OrderResponse {
        val useCase = PlaceOrder(repo, notify)
        val order = useCase.execute(req.userId, req.items)
        return OrderResponse.from(order)
    }
}

The core defines OrderRepository as an interface. Postgres is an implementation detail. If you want to switch to MongoDB, you write a MongoOrderRepository that implements the same interface. If you want to test PlaceOrder, you pass in an in-memory fake. The test doesn't start Postgres — it doesn't even know Postgres exists.

What this actually buys you

The hexagonal architecture is often sold as "so you can swap your database later." That's not wrong, but it's the least important benefit. Database swaps are rare. The real benefits are more subtle and more valuable:

  1. Testability without infrastructure. The core has no dependencies on frameworks, databases, or network services. You can test every business rule with plain unit tests that run in milliseconds. The adapters get integration tests (does this SQL query actually work against a real Postgres?), but those are fewer and focused on the integration boundary.
  2. Decisions you can defer. You can build the entire core — all the business logic, validation, and workflows — before deciding whether the API will be REST, gRPC, or GraphQL. The core doesn't care. The adapter choice is the last decision, not the first, and it's reversible without touching business logic.
  3. Boundaries that match the organization. A team that owns the core can move fast because changes to the core don't affect the HTTP adapter, and changes to the HTTP adapter don't affect the core. When the product team wants to add a CLI interface, they write a new adapter for the existing ports. The core team isn't involved.
  4. The framework becomes a plugin. Spring, Express, FastAPI — these are adapters. Your application doesn't depend on them; they depend on your application. This inverts the typical relationship where the framework owns main() and your code fills in the blanks. In a hexagonal architecture, main() is in your code, and the framework is wired in through adapters.
predict You've built a beautifully hexagonal app. The core has zero framework dependencies. The HTTP adapter is 200 lines. A new requirement arrives: this service needs to also accept commands via a Kafka topic. How much of the core needs to change?

Nothing. Zero lines. You write a Kafka adapter — a consumer that deserializes messages from the topic, calls the same PlaceOrder.execute() that the HTTP controller calls, and serializes the result (or publishes an event). The core's driving ports don't know or care whether the caller is an HTTP request or a Kafka message. This is the architecture paying rent: the cost you paid up front (defining ports, writing adapters) returns every time a new entry point or a new infrastructure dependency materializes. Without ports, the Kafka integration would be scattered through the codebase — parsing Kafka headers in the order service, mixing Kafka commit offsets with business logic, testing through an embedded Kafka broker.

The boundaries are real, and so is the overhead

the real cost Ports and adapters add indirection. Every driven port is an interface with (initially) one implementation. If you have 47 driven ports, you have 47 interfaces that exist solely to enable a swap that may never happen. The overhead isn't the interface declaration (that's cheap); it's the cognitive load of navigating an extra layer every time you trace a call path. The pragmatic stance: apply the pattern at the boundaries that actually change or that you actually need to test in isolation. The database is worth abstracting. The filesystem is worth abstracting. An in-process calculation utility that's 30 lines and pure math? An interface for that is ceremony, not architecture.

The hexagonal architecture is not a religion. It's a tool for managing coupling at the system boundary. When the boundary is between your code and something slow, external, or likely to change, a port is a bargain. When the boundary is between two functions in the same file, a port is overhead. The skill is knowing which boundaries matter.

One more honest note: most "hexagonal" codebases in the wild are better described as "the database is behind an interface and we called it a repository." That's fine. That's 80% of the value for 20% of the ceremony. The full hexagon — driving ports, driven ports, adapters for every external system, the framework as a plugin — is for systems where the cost of getting a boundary wrong is high: payment processing, multi-tenant platforms, systems with a regulatory requirement to keep business logic auditable and separable from infrastructure. A blog engine doesn't need a hexagonal architecture. A payment system that processes millions of transactions across three cloud providers does.
