Load Balancing Strategies

Round-robin is the boring answer; here's what real traffic needs — from Layer 4 to Layer 7, from random to consistent hashing.

A load balancer sits between clients and servers, deciding which server handles each request. The simplest algorithm — round-robin, each server in turn — works beautifully when all servers are identical and all requests cost the same. In production, neither condition holds. Servers have different capacities, requests have different costs, and connections have state (a user's session lives on a specific server). The algorithm choice determines whether your load is actually balanced or just evenly distributed across a fundamentally uneven system.

The algorithm ladder

Round-robin / random: Each request goes to the next server in sequence (or a random one). O(1), stateless, perfectly fair when servers are identical. The default in most load balancers. Breaks when: servers have different capacities, requests have sticky state (user sessions), or request costs vary widely (one slow request blocks the server's queue while fast requests pile up).

Weighted round-robin: Give each server a weight proportional to its capacity. A server with weight 3 gets 3× the requests of a server with weight 1. Handles heterogeneous hardware (some servers are bigger). Still doesn't account for request cost variation — a server might get 3× the light requests and 3× the heavy ones, and the weights smooth the average but not the tail.

Least connections: Send each request to the server with the fewest active connections. A proxy metric for "which server is least busy right now." Better than round-robin when request costs vary — a server handling a slow request will accumulate connections and receive fewer new ones. The most common algorithm beyond round-robin in production load balancers (nginx, HAProxy, AWS ALB).

Least response time: Track each server's recent response time. Send requests to the fastest-responding server. Adapts to degradation — if a server slows down (GC pause, disk I/O), its response time rises and it receives less traffic. More sensitive than least connections (which only counts in-flight requests, not their cost).

Least connections is about "which server has the shortest queue." Least response time is about "which server is actually processing fastest." Queue depth and processing speed are different metrics.

Consistent hashing: Hash the request (by client IP, session ID, or request key) to a point on a ring. Each server owns a range of the ring. When a server is added or removed, only the keys in its range are reassigned — the rest stay where they were. This minimizes disruption during scaling events, which is why consistent hashing is used for distributed caches and stateful services. If a cache server dies, only the keys that hashed to it are lost from cache; the rest remain cached on their existing servers.

Layer 4 vs Layer 7

Load balancing happens at two layers, and the difference is what information the balancer uses to decide:

  • Layer 4 (TCP/UDP): The balancer sees IP addresses and ports. It can route based on source/destination IP, but can't inspect HTTP headers, cookies, or URL paths. Fast (no payload inspection), transparent to TLS. Used for non-HTTP protocols and as the first hop before an L7 balancer.
  • Layer 7 (HTTP/HTTPS): The balancer terminates TLS and inspects HTTP headers. It can route based on URL path (/api/* → API servers, /static/* → CDN), Host header (multi-tenant routing), cookies (sticky sessions), or any HTTP header. Slower (must decrypt and parse HTTP), but smarter. The standard for web applications.
predict You're using least-connections load balancing with 5 backend servers. Server 3 starts returning 500 errors for every request. The other 4 servers are healthy. What does the load balancer do, and what should it do?

Least connections continues sending traffic to server 3 — it has few active connections (because it returns 500s quickly, completing the connection), so it looks like the least-busy server. The load balancer is optimizing for queue depth, not success rate. The fix: health checks — the load balancer periodically probes each backend's health endpoint, and servers that fail health checks are removed from the pool. A passive health check (observing error rates and ejecting unhealthy servers) catches errors faster than an active health check, at the cost of potentially ejecting a server that's returning errors for a specific client rather than being generally unhealthy. Most production load balancers combine both: active health checks (probe /health every N seconds) and passive health checks (if a server's error rate exceeds a threshold, eject it temporarily).

the state problem Load balancing stateless services (API servers) is straightforward. Load balancing stateful services (applications with server-side sessions, WebSocket connections, stateful game servers) requires sticky sessions — the load balancer must route the same user to the same backend consistently. Sticky sessions break the "any server can handle any request" property that makes horizontal scaling work, and they create hotspots (one user's traffic is expensive, and their server bears it all). The fix: push state out of the application servers and into a shared store (Redis, database). The load balancer distributes requests freely; the application fetches state from the shared store on each request. The load balancer's job gets simpler; the application's job gets more network-dependent.
