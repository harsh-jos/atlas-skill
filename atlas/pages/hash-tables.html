Hash Tables, Really

The data structure you use a hundred times a day — and the surprisingly deep design decisions hiding inside Python dicts, Go maps, and every O(1) lookup you've taken for granted.

In 2016, Python 3.6 shipped a new dict implementation. The release notes mentioned lower memory usage and faster iteration. Then a funny thing happened: people noticed their dicts were preserving insertion order. Keyword arguments kept their declaration order. JSON round-tripped without reordering keys. The language didn't promise this — it fell out of the data structure.

The story of how that happened is the story of how hash tables actually work, and it starts with the one problem every hash table has to solve.

Two keys walk into the same bucket

A hash function maps a key to an integer, and that integer picks a slot. But the hash space is enormous (264 values for a 64-bit hash) and the table has maybe 8 slots. Many keys will map to the same slot. This is the collision, and how a table resolves it is its defining design decision.

The textbook answer is chaining: each slot holds a linked list (or a balanced tree, in Java's HashMap once a bucket grows past 8 entries). Hash to a slot, walk the list, find your key. Simple to describe, simple to implement, and it handles high load factors gracefully — the lists just grow longer.

The problem with chaining is the hardware it runs on. Following a pointer in a linked list is a cache miss, and cache misses dominate modern CPU costs. If your hash table is hot — and most are — you can do better.

The alternative: keep everything in one array

Open addressing stores entries directly in the table array, one per slot. When a key maps to slot 5 and slot 5 is occupied, you probe: check slot 6, then 7, then 8, wrapping around. The probing strategy determines everything about performance.

The naive approach is linear probing — try the next slot, and the next, until you find an empty one. It's cache-friendly (sequential access), but it suffers from primary clustering: a run of consecutive filled slots acts like a magnet, growing longer and slower with each collision. Inserts near a cluster degrade toward O(n).

Quadratic probing jumps by growing squares — slot + 1, slot + 4, slot + 9 — which breaks up clusters but can miss empty slots if the table size isn't prime. Double hashing uses a second hash function to compute the probe step, producing nearly independent probe sequences, at the cost of computing a second hash.

Robin Hood: steal from the rich

There's a subtler idea. In open addressing, not all entries have traveled the same distance from their ideal slot. Some got lucky and landed right where they hashed; others probed 12 slots to find a home. Robin Hood hashing enforces fairness: on insert, if the key you're inserting has already traveled farther from its ideal slot than the key currently occupying this slot, you evict the occupant and keep going with it instead. The rich (close to home) give to the poor (far from home).

This sounds like a gimmick, but it provably bounds the variance of probe lengths — no key gets stranded far from home while its neighbor sits comfortably. The worst-case probe length drops dramatically, and the table stays fast even at high load factors. Rust's standard HashMap uses Robin Hood. So does Go's map, in spirit.

The Python dict: compact, ordered, and a little bit sneaky

The Python 3.6 dict — designed by Raymond Hettinger — replaced the old implementation's sparse table of entries with two arrays:

A dense entries table stores each key-value pair exactly once, in insertion order. This is a flat array of structs: [hash, key_ptr, value_ptr, hash, key_ptr, value_ptr, ...]. Because entries are stored in insertion order and never moved (only marked deleted), iteration is a straight array walk — fast, cache-friendly, and naturally ordered.

A sparse index — the hash table proper — maps each hash to an index into the entries table. It uses open addressing with a custom probing scheme. Each slot stores either an index into the entries array or a sentinel for empty/tombstone.

This split is the trick. When you resize, you rebuild the cheap sparse index without touching the entries table. Insertion order is a free side effect: entries are appended to the dense array and never reordered. The old dict couldn't make this promise because resizing reshuffled everything.

The dictionary didn't gain ordering because Python decided dictionaries should be ordered. It gained ordering because the compact layout was faster, and ordering was a free side effect.

The load factor: the fundamental dial

Every hash table has a load factor — the ratio of filled slots to total slots. It's the single number that controls the speed-versus-space trade.

At load factor 0.5, half the slots are empty. Collisions are rare, lookups are fast, and you're using twice the memory you need. At load factor 0.9, memory is efficient but probe chains are long — you're spending CPU searching instead of space storing.

Most general-purpose hash tables resize around 0.66–0.75. It's an empirical sweet spot where the expected probe length is roughly 2–3 slots, and the memory overhead is roughly 33%. But specialized tables pick different points: an in-memory cache that's mostly read might sit at 0.9; a latency-sensitive path might resize at 0.5.

Resizing: why you can't just grow the array

You can't realloc a hash table. The hash function maps a key to a slot in a table of size N. Double N and every key maps to a different slot. Resizing means rehashing every entry into the new table.

This is an O(n) operation that hits on a single insert — the one that crosses the threshold. Most implementations amortize this cost (the same way dynamic arrays amortize growth), but the latency spike is real. Real-time systems sometimes use incremental resizing: allocate the new table, move a few entries on each insert, and consult both tables on lookup until the migration finishes. Redis does this.

predict A hash table with 1,000,000 entries is at load factor 0.75. You insert one more key and it resizes to 2,000,000 slots, rehashing everything. But before you answer: what determines whether this resize is a latency spike the user notices, or invisible noise in a p99 metric? And what would you do differently if the answer were "the user notices"?

The user notices if the rehash happens on the critical path — a request handler, a render loop, a hot codec. The answer, if it matters, is incremental resizing: spread the work across subsequent inserts so no single one pays the full O(n) bill. The cost is implementation complexity and the need to check two tables on every lookup during the migration window.

Hash flooding: when O(1) becomes O(n)

In 2011, a class of DoS attacks hit web frameworks including PHP, Ruby, and Python. The attacker crafted HTTP parameters whose keys all collided in the framework's hash table — thousands of keys hashing to the same bucket. Every insert probed the entire chain. A single request could burn CPU-minutes. nginx, Tomcat, and Flask all shipped patches within weeks.

The fix is universal hashing with a per-process random seed: mix a random value into the hash function so an attacker can't predict which keys collide. Python added hash randomization in 3.3 (PYTHONHASHSEED). Most language runtimes did the same. But the lesson is deeper: O(1) is an amortized probabilistic guarantee, not an absolute one, and an adversary who can control your keys can break it.

Where hash tables sit on the map

Hash tables occupy a distinct point in the data-structure trade space. They share O(1) average-case access with arrays (indexed by integer), but trade contiguous-memory guarantees for key-flexibility. They're the inverse of a search tree: a B-tree gives you ordered traversal and logarithmic worst-case; a hash table gives you constant-time lookup and no ordering. They're the foundation under bloom filters (probabilistic membership), consistent hashing (distributed lookup), and every hash set ever written.

At the implementation level, a hash table is a small collection of genuinely clever ideas — open addressing, Robin Hood, compact storage — that collectively turn an O(n) worst case into an O(1) everyday reality. The ideas are 60 years old and still being refined.

when not to Reach for something else when you need ordered traversal (a B-tree or sorted array keeps keys in order), when N is small and fixed (a linear scan of an array beats a hash table until roughly 30–50 entries, because hashing costs more than comparing), or when you need strict worst-case guarantees (a balanced tree gives O(log n) always; a hash table gives O(n) on a bad day). Also, if your keys are insert-only and you never need to look them up by key — you just want deduplication — a sorted array with binary search can be smaller and simpler.
