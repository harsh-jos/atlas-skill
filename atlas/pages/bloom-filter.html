Bloom Filters

A set that is wrong on purpose — false positives in exchange for being absurdly small and fast. The trade that makes it worth it, and the math behind the magic number of hash functions.

You're building a URL-shortener. Before assigning a new short code, you want to check it's not already in use. The obvious answer is a database lookup — but that's a disk seek per request. A cache helps for codes that exist, but for new codes — which by definition aren't in the cache — you still hit disk to confirm the miss. Most codes proposed will be new. Each proposal costs a disk seek.

A Bloom filter in front of the database changes this. It's a probabilistic set that fits entirely in memory — a few megabytes for billions of entries. Query it: "have you seen this code before?" If it says no, the answer is guaranteed correct — the code is definitely new, no disk seek needed. If it says yes, the code might exist, and you fall through to the database to confirm. Most answers are "no." Most disk seeks are avoided.

A Bloom filter never says yes incorrectly. It only says no incorrectly — never.

The mechanism

A Bloom filter is a bit array of m bits, initially all zero, plus k independent hash functions, each mapping an element to a position in [0, m).

Insert: hash the element with all k functions. Set each of the k resulting bits to 1.

Query: hash the element with all k functions. If all k bits are 1, return "possibly in set." If any bit is 0, return "definitely not in set."

That's the whole thing. The "possibly" comes from overlap: bits set by different elements can conspire to make a never-inserted element look present. The false-positive rate is a function of m (array size), n (number of inserted elements), and k (number of hash functions).

The math that picks k

After inserting n elements, the probability a specific bit is still 0 is (1 − 1/m)kn ≈ e−kn/m. For a false positive, all k bits for the queried element must be 1, even though the element was never inserted. That probability: ε ≈ (1 − e−kn/m)k.

For fixed m and n, the optimal k minimizes this. Take the derivative, set to zero, and you get: k = (m/n) · ln 2 ≈ 0.693 · m/n. At this optimum, the false-positive rate is ε ≈ (0.6185)m/n.

Concretely: to store 1,000,000 elements at a 1% false-positive rate, you need roughly m = −n · ln ε / (ln 2)² ≈ 9.6 million bits ≈ 1.2 MB, with k ≈ 7 hash functions. That's 1.2 MB for a million-element set. A hash set storing 1,000,000 32-bit hashes would be ~32 MB minimum, plus overhead for collision resolution — 25–50× larger.

predict You double k from its optimal value — twice as many hash functions. The false-positive rate goes up. Before calculating: why? What's the intuition for too many hash functions making the filter worse?

With too many hash functions, each insert sets more bits. The bit array saturates — too many 1s. When you query a never-inserted element, the chance that all k bits happen to be 1 (by coincidence of other insertions) rises. At the extreme, if k = m, every element sets m bits and the filter is all 1s — the false-positive rate is 100%. The optimum k balances: enough hash functions to make accidental collision unlikely, few enough that the array doesn't drown in 1s.

What you can't do with a Bloom filter

You can't delete. Setting bits back to 0 would also remove any other element that shared that bit. There's a variant — the Counting Bloom Filter — where each slot is a small counter instead of a bit. Insert increments; delete decrements. At 4 bits per counter, overflow (an element inserted 16+ times at the same slot) is possible but astronomically unlikely.

You can't enumerate. The filter stores the existence of elements, not the elements themselves. You can't iterate it. It's a membership oracle, not a collection.

You can't resize. Once the bit array hits its false-positive budget, you can't grow it without the original elements (to re-insert into a larger filter). Production systems often pair a Bloom filter with a write-ahead log of inserts so the filter can be rebuilt from scratch.

Where they actually live

LSM-tree databases (Cassandra, RocksDB, LevelDB) use Bloom filters to skip disk lookups for non-existent keys. An LSM-tree has multiple sorted files on disk; a query checks the in-memory Bloom filter for each file before doing a disk seek. Without Bloom filters, every point query would touch every file.

CDN cache eviction. Akamai uses Bloom filters to track which URLs are in cache across edge servers. A false positive means fetching from origin unnecessarily — a small cost paid rarely.

Chrome's Safe Browsing. Google distributes a Bloom filter of malicious URL prefixes to every Chrome install. The browser checks locally; a "yes" triggers a real lookup to Google's servers for confirmation. Most URLs are safe, so most checks are local and instant.

Spell checkers. A Bloom filter of dictionary words tells you in O(k) whether color is a word. A false positive means missing a misspelling — at 0.01%, essentially never.

when not to If you need deletion, enumeration, or guaranteed no-false-positives, a Bloom filter is wrong. A hash set gives you exact membership. A Bloom filter trades exactness for size — and if your dataset fits comfortably in memory, that trade isn't worth making. Also, for tiny N (< ~10,000), a sorted array with binary search is simpler, exact, and still fast. Bloom filters earn their keep at scale: millions to billions of elements.
