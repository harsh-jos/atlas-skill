Chaos Engineering

Breaking production on purpose, before it breaks itself at 3am — the practice of injecting failure to prove your system survives it.

In 2011, Netflix launched Chaos Monkey — a tool that randomly terminated production instances during business hours. The reaction from the rest of the industry was roughly: "you're doing what in production?" The reaction from Netflix's reliability metrics was an upward trend. Chaos Monkey forced every service to handle instance failure gracefully, because instance failure was no longer a rare disaster — it was Tuesday.

Chaos engineering is the discipline of designing and running experiments to test a system's ability to withstand turbulent conditions. It's not random destruction — it's hypothesis-driven experimentation. You form a hypothesis ("if the payment service becomes slow, the order service degrades gracefully by showing a queue position"), you design an experiment that tests it (inject 2s of latency into the payment service for 5 minutes), and you measure the outcome against your SLOs. If the system handles it, you've increased confidence. If it doesn't, you've found a vulnerability before an attacker or a bad deploy finds it for you.

The experiment loop

A chaos experiment, step by step
1. Define steady state:  "p99 latency < 200ms, error rate < 0.1%"
2. Form hypothesis:      "killing one Kafka broker won't breach the steady state"
3. Design experiment:    terminate broker-3, observe for 10 minutes
4. Define blast radius:  production, but only the orders topic (not payments)
5. Run experiment:       execute during business hours with engineers watching
6. Observe:              did the steady state hold? where did it deviate?
7. Learn & fix:          if it broke, fix it before the real outage happens
8. Automate:             add this experiment to the CI/CD pipeline

The progression is important: start in staging, graduate to production, automate. No one's first chaos experiment should be "terminate the primary database in prod during Black Friday." The Netflix chaos engineering team's philosophy: start with the smallest possible experiment that teaches you something, and expand the blast radius as your confidence grows.

What you actually break

Chaos experiments fall into categories, and the progression through them is a maturity model:

  • Infrastructure faults: Kill a pod. Terminate an EC2 instance. Fill a disk. These are the easiest to run and the easiest to recover from — your infrastructure should handle individual node failures already.
  • Network faults: Inject latency (100ms, 500ms, 2s). Drop packets (1%, 10%, 50%). Partition the network (split a cluster into two halves that can't see each other — the hardest network test). Network faults are where most systems break, because timeouts and retries interact in non-obvious ways.
  • Application faults: Return 500s from a dependency for 30 seconds. Make a service respond slowly (not down — just slow, which is harder to detect and handle). Exhaust a connection pool. These test the application's resilience logic, not just the infrastructure's.
  • State faults: Corrupt a write. Run a long transaction that holds locks. Trigger a schema change while traffic is flowing. The hardest class of experiments — these test the correctness of your system under stress, not just its availability.
predict You inject 2 seconds of latency into the payment service. The order service retries 3 times with a 500ms timeout. What happens to the order service's latency distribution, and what's the failure mode you just exposed?

The order service now takes at least 6 seconds to fail (3 retries × 500ms each × 2s of injected latency — actually, the retries also hit the slow payment service, so each retry consumes ~2.5s, totaling 7.5s). If the order service itself has an upstream timeout (say, the API gateway times out at 5s), the gateway returns a 504 before the order service finishes failing — and the order service is still burning threads on retries for another 2.5s after the user has already seen an error. This is retry amplification: retries that don't account for the downstream's actual latency multiply the problem, filling thread pools and cascading into other services. The fix: retries with a budget (max total time across all retries), circuit breakers (stop retrying if the downstream is clearly degraded), and sensible timeout alignment (the upstream timeout should be longer than the downstream's worst-case retry budget).

The tooling landscape

In 2026, chaos engineering is well-tooled:

  • Chaos Mesh (CNCF Graduated, OSS): The most capable open-source option for Kubernetes. Native CRDs for every fault type — PodChaos, NetworkChaos, IOChaos, StressChaos, TimeChaos (skew system clocks), KernelChaos. GitOps-friendly: experiments are version-controlled YAML.
  • LitmusChaos (CNCF Incubating, OSS): Strong GitOps integration, large experiment library (ChaosHub). Best for teams already deep in Kubernetes who want pre-built experiments.
  • AWS Fault Injection Simulator / Azure Chaos Studio: Managed services from cloud providers. Less flexible, but tightly integrated with IAM and cloud-specific primitives.
  • Gremlin (Commercial SaaS): Enterprise focus — safety guardrails, audit trails, compliance. Runs on any infrastructure, not just Kubernetes.
the real prerequisite Chaos engineering isn't a tool — it's an observability practice. If you can't measure your system's steady state (latency percentiles, error rates, throughput by service), you can't run a meaningful chaos experiment. You'll break something, see that something bad happened, and have no idea what to fix. Chaos engineering without observability is just vandalism. The first step is always: instrument your system well enough that you can detect a degradation before your users do. The chaos experiments verify that the instrumentation is right and the degradation is handled gracefully.
