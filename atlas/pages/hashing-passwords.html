Hashing Passwords Properly

Why your password hash needs to be slow on purpose — and the three algorithms you should actually use in 2026.

In 2012, LinkedIn was breached. Attackers stole 6.5 million password hashes. The hashes were unsalted SHA-1 — a general-purpose hash function designed to be fast. Within days, millions of passwords were cracked. The problem wasn't the hashing — SHA-1 correctly hashed each password. The problem was that SHA-1 is designed for verifying file integrity, not storing secrets. It's fast on purpose. Password hashing needs the opposite: it needs to be slow enough that an attacker who steals your database can't brute-force millions of passwords per second.

A password hash is a speed bump. The right algorithm makes the bump tall enough that attacking a database of hashes is economically infeasible.

Why "just hash it" is wrong

A naive sha256(password) fails in three ways:

  1. Too fast. A modern GPU can compute billions of SHA-256 hashes per second. Your users' passwords — even good ones — will be cracked in hours.
  2. No salt. Identical passwords produce identical hashes. An attacker hashes "password123" once, finds 10,000 matches in the database, and has 10,000 accounts. A salt — a random value, unique per user, stored alongside the hash — makes identical passwords produce different hashes and makes precomputed rainbow tables useless.
  3. No memory hardness. GPU-based attacks parallelize perfectly because SHA-256 uses negligible memory. A memory-hard algorithm forces each hash computation to use a configurable amount of RAM, making GPU attacks expensive (GPUs have many cores but limited memory bandwidth).

The three algorithms you should use

In 2026, the password hashing landscape has converged on three algorithms. All three are better than anything you're likely using if your system is more than five years old:

  • Argon2id — The winner of the 2015 Password Hashing Competition and the OWASP recommendation. Three variants: Argon2d (data-dependent, resistant to GPU attacks), Argon2i (data-independent, resistant to side-channel attacks), Argon2id (hybrid — uses Argon2i for the first pass and Argon2d for subsequent passes). Use Argon2id. Configurable memory, time (iterations), and parallelism. Typical settings: 64 MB memory, 3 iterations, 1 degree of parallelism.
  • bcrypt — The battle-tested default (since 1999). Resistant to GPU attacks (uses a modified Blowfish cipher that requires 4KB of state, which doesn't fit in GPU shared memory). Configurable cost factor (2cost iterations). Cost factor 12 is the current minimum; 14 is comfortable. The 72-byte password limit is the main sharp edge — longer passwords are silently truncated.
  • scrypt — The original memory-hard function (2009). Configurable CPU cost, memory cost, and parallelism. Less widely deployed than bcrypt, and the RFC is older, but it's sound. Argon2id is generally preferred for new deployments.
Password hashing, correctly (pseudocode)
func createAccount(email, password):
    salt = crypto.randomBytes(16)         // unique per user
    hash = argon2id(password, salt, {
        memory: 64 * 1024,                 // 64 MB
        iterations: 3,
        parallelism: 1
    })
    store(user_id, salt, hash)            // store both the salt AND the hash

func verifyLogin(email, password):
    salt, hash = lookup(email)
    return argon2id.verify(password, salt, hash)
predict A user sets a 100-character passphrase. Your system uses bcrypt with no additional pre-hashing. What does bcrypt do with the password before hashing, and should you worry about it?

Bcrypt silently truncates the password to 72 bytes. The user's 100-character passphrase (which might be 100–400 bytes in UTF-8) is reduced to the first 72 bytes. If the user's passphrase has its entropy concentrated in the first 72 bytes, they're fine. If it's a passphrase where the security comes from length ("correct-horse-battery-staple-extended-version-with-more-words"), the truncation might be a real loss. The fix: pre-hash the password with SHA-256 (or better, SHA-512) before passing to bcrypt. This gives bcrypt a fixed 32-byte input while preserving the full entropy of the original password. This pattern (hash, then bcrypt) is common enough to have a name: "sha-bcrypt" or "pre-hashing." Argon2id doesn't have this problem — it accepts arbitrarily long passwords.

Pepper: the extra secret you don't need (probably)

A pepper is a secret value stored outside the database (in an HSM, an environment variable, or a secrets manager) that's combined with the password before hashing. If the database is breached but the pepper isn't, the attacker can't brute-force any hashes — even weak passwords are protected.

The cost: pepper rotation is a migration. If you want to change the pepper, every password must be rehashed — which means waiting for users to log in (lazy rotation) or invalidating everyone's password. And pepper breaks password verification during a pepper change if the old and new peppers aren't both tried. Most teams are better served by using Argon2id with strong parameters than by adding pepper to the mix. The threat pepper defends against (database breach without application breach) is real but rare relative to the operational complexity.

when not to roll your own If you're not using a library that has been through a security audit, stop and use one. The password-hashing libraries in every major language (Python's bcrypt or argon2-cffi, Java's argon2-jvm, Go's golang.org/x/crypto/argon2, Node's argon2) are written by people who've studied the side-channel literature and handled the edge cases. The sharp edges in password hashing are not in the algorithm choice (Argon2id vs bcrypt) — they're in the implementation: timing side channels, incorrect parameter selection, storing the salt incorrectly, and failing to upgrade the algorithm when the old one weakens. Use the library, and subscribe to its security announcements.
