Coupling & Cohesion

The two words that explain why one codebase is pleasant to work in and another is tar — and why "low coupling, high cohesion" is the advice everyone agrees on and almost nobody can operationalize.

You open a codebase you've never seen. Within ten minutes, you either have a rough sense of where things live and can make a small change confidently, or you're three layers deep in a 900-line function that touches the database, formats an email, and calls a third-party API, and you're afraid to change anything. The first codebase has low coupling and high cohesion. The second has neither.

Larry Constantine and Edsger Dijkstra wrote about these ideas in the late 1960s. They've been core to software design for over 50 years, and they're still the best diagnostic tool for why a system feels the way it feels.

Coupling: how much A knows about B

Coupling measures the degree of interdependence between two modules. If changing module A forces you to change module B, they're coupled. The goal is low coupling — modules should be connected by narrow, stable contracts, not by shared internals.

Coupling comes in degrees, and the degrees matter:

Content coupling (worst). One module reaches into another's internal data — reads its private field, modifies its internal state, depends on its implementation. A one-line change in the callee can silently break the caller. This is the coupling that makes codebases untouchable.

Common coupling. Two modules share a mutable global variable. They're coupled through that variable: changing when or how one writes to it affects the other. Global config that never changes after startup is fine (low temporal coupling); a global current_user that every function reads is a bug factory.

Control coupling. Module A passes a flag to module B that changes B's behavior. process_order(order, is_priority=True) — A knows about B's internal branching. The caller is coupled to the callee's control flow.

Stamp coupling. Module A passes a large data structure to module B, but B only uses three fields. B is coupled to the structure's shape, not its content. Refactoring the structure breaks B even though B doesn't use the changed fields. Pass only what's needed (user.email, not the whole user object).

Data coupling (best). Modules communicate through small, well-defined parameters. They share no state, reach into no internals, pass no control flags. This is the target.

The rule of thumb: if you can't describe what two modules expect from each other in one sentence, they're too coupled.

Cohesion: how much the stuff inside A belongs together

Cohesion measures how strongly the elements within a single module relate to each other. The goal is high cohesion: everything in a module serves a single purpose. The module has one reason to change.

Coincidental cohesion (worst). Elements are grouped arbitrarily — a utils.py with a function that validates email addresses next to one that formats file sizes for humans. They share nothing except the misfortune of being in the same file. This is the most common and most easily fixed cohesion failure.

Logical cohesion. Elements do related-sounding things but operate on different data. An IOModule that reads CSV, writes JSON, and streams video — all I/O, but the concerns are unrelated. The module has a theme, not a purpose.

Temporal cohesion. Elements are grouped because they all happen at the same time. An init() function that connects to the database, starts a metrics server, and warms the cache. These things share timing but not purpose — debugging one means reading about the others.

Communicational cohesion. Elements operate on the same data. A module with validate_email(), normalize_email(), and send_email() all working on email data. Reasonable, but still mixed: validation and sending are different responsibilities even though they share a data type.

Functional cohesion (best). Every element in the module contributes to a single, well-defined task. EmailValidator validates emails. InvoiceRenderer renders invoices. The name describes everything it does, and it does nothing the name doesn't describe.

Why they move together

Low coupling and high cohesion are two sides of the same design coin. When a module has high cohesion — it does one thing — the interfaces it exposes to the rest of the system tend to be narrow and stable (low coupling). When modules are loosely coupled, each one is free to refactor its internals without cascading changes — which makes it feasible to keep each one cohesive.

The reinforcing cycle cuts both ways. Tight coupling between two modules makes it hard to refactor either one, so responsibilities drift and cohesion degrades. Low cohesion makes a module present a wide, unstable interface, which tightens coupling with everything that depends on it. These codebases don't fail from one big mistake — they degrade through a thousand small compromises, each one making the next one easier.

What this looks like in practice

The test tells you first. If testing a module requires mocking ten dependencies and constructing a dozen fixtures, the coupling is too high. A module with low coupling can be tested with one or two mocks. This is why testability and good design correlate — tests punish tight coupling by making it painful.

Import statements are coupling measurements. A file that imports 40 things is coupled to 40 other modules. Some of those imports belong elsewhere. Kent Beck's rule: if a class has more than ~7 dependencies, it's probably doing too many things (low cohesion) or reaching too far into its neighbors (high coupling).

Changes that always travel together. If every change to the billing module also requires a change to the user module, those modules are coupled — probably through a shared data structure, a fragile interface, or a responsibility that's split between them. The fix is either moving the responsibility entirely into one module or formalizing the boundary between them with a stable contract.

your call You're reviewing a PR. The OrderService now imports EmailTemplate to format order-confirmation emails. Before you comment: is the problem coupling, cohesion, or both? And what's the one-sentence rule you'd give the author?

It's both. OrderService is now coupled to EmailTemplate — a change to email formatting forces a change to order processing. And OrderService now has lower cohesion: it processes orders and formats emails, two different responsibilities. The fix: OrderService emits an OrderPlaced event; a separate OrderConfirmationEmailSender listens for the event and owns the email formatting. The coupling shifts from direct dependency (bad) to event contract (better). The cohesion improves because each module does one thing.

when not to "Low coupling, high cohesion" can be over-applied into a maze of micro-modules, each doing one microscopic thing, connected by an explosion of interfaces. The cure becomes worse than the disease. A module should be cohesive — one reason to change — but that "reason" is a human-scale concept, not a compiler-scale one. EmailValidator is a reasonable module. Splitting it into EmailSyntaxChecker, EmailDomainResolver, and EmailMXLookup is a parody. The unit of cohesion is whatever a teammate can hold in their head and change confidently.
