Event Loops vs Threads vs Coroutines

The three answers to "how do I wait for 10,000 things at once" — and why each of them is the right answer somewhere.

You're writing a server that needs to handle 10,000 concurrent connections. Each connection spends most of its time waiting — for a database, for another service, for the client to send the next byte. Your CPU is bored. The question is: how do you structure the code so that waiting on one thing doesn't block the other 9,999?

There are three families of answers. They all solve the same problem — multiplexing many tasks onto few OS threads — but they differ radically in what your code looks like and where the pain lives.

Threads: one task, one OS thread

The simplest model: spawn a thread per connection. Each thread runs synchronous, linear code. You call read(), it blocks, the OS puts the thread to sleep, and when data arrives the OS wakes it up and your code continues from the next line. The code is easy to write because it looks like normal sequential code.

The problem is scale. Each OS thread costs about 1 MB of stack space (on Linux, by default) and takes microseconds to context-switch. At 10,000 threads, that's 10 GB of stack alone, plus the scheduler overhead of managing them all. The kernel wasn't designed to juggle tens of thousands of mostly-idle threads.

This worked fine in the 1990s, when "concurrent" meant a hundred connections. It falls apart at modern scale. But the programming model — write synchronous code, let the runtime handle the waiting — is so compelling that both of the other approaches have been trying to get back to it ever since.

The event loop: one thread, many callbacks

The event loop inverts control: instead of a thread waiting on its own connection, a single thread monitors all connections and dispatches events to callbacks. Node.js, nginx, Redis, and the classic Python asyncio all work this way.

The event loop, conceptually
while (running) {
    events = poll(all_open_sockets)    // "tell me which ones are ready"
    for (sock : events.readable) {
        sock.read_callback(sock)       // dispatch to registered handler
    }
    for (sock : events.writable) {
        sock.write_callback(sock)
    }
}

The kernel provides the magic: epoll (Linux), kqueue (macOS/BSD), or IOCP (Windows) lets a single thread efficiently wait on thousands of file descriptors. The thread blocks in epoll_wait(), not in individual read() calls. When data arrives on socket 247, the kernel wakes the thread, and the event loop calls socket 247's registered callback.

the cost You've traded thread stacks for callback chains. A simple sequence — read request, parse JSON, query database, write response — becomes a pyramid of nested callbacks (or a flat but still fragmented chain of promises/await). The stack trace you'd normally use to debug is gone, because each callback runs in a fresh frame on the event loop's single thread. And a single CPU-heavy callback blocks the entire loop — every other connection stalls until it finishes.

Coroutines: let the compiler do the callbacks

Coroutines (also called green threads, fibers, or virtual threads) steal the best property of threads — write linear code — and implement it without OS threads. The trick is cooperative scheduling: a coroutine runs until it explicitly yields (at an I/O boundary), at which point the runtime switches to another coroutine on the same OS thread.

This is what Go's goroutine, Kotlin's suspend, Rust's async fn, and Python's async/await all do. The key difference is who implements the state machine:

  1. Stackful coroutines (goroutines, virtual threads). The runtime manages a growable stack per coroutine. When a goroutine blocks on a channel or I/O, the Go runtime saves its stack and switches to another goroutine — all on the same OS thread. The code looks synchronous. You write data, err := conn.Read(b) and it just works. The Go runtime uses a work-stealing scheduler that maps M goroutines onto N OS threads, preempting at function calls and (since Go 1.14) loop backs-edges.
  2. Stackless coroutines (async/await). The compiler transforms an async fn into a state machine. Each .await point becomes a state transition. There's no separate stack — the coroutine's live variables are stored in a struct the compiler generates. This is lighter-weight (no stack allocation) but splits your code into two worlds: async functions can only be called from other async functions (the "colored function" problem), and the state-machine transform means the compiler needs to see the whole async chain.

Java's virtual threads (Project Loom, stable since JDK 21 in 2023) are a fascinating data point: they're stackful coroutines that retrofit the thread model. Existing synchronous Java code that uses Thread.sleep() or blocking I/O gets the scalability of the event-loop model without rewriting to callbacks or async/await. The JVM intercepts blocking calls and parks the virtual thread, freeing the underlying OS thread (called a "carrier thread") to run another virtual thread. As of 2025, adoption has been pragmatic: teams use them for I/O-bound server work where rewriting to reactive streams would be prohibitively expensive, but they're not a universal replacement for platform threads in CPU-bound or latency-sensitive contexts.

predict A Rust async function spawns 100 concurrent tasks on a single-threaded Tokio runtime. One of those tasks calls std::thread::sleep(Duration::from_secs(10)) instead of tokio::time::sleep(...). What happens to the other 99 tasks during those ten seconds?

They freeze. Completely. std::thread::sleep blocks the OS thread, and since there's only one OS thread, the runtime can't poll any other task. The async state machine only yields at .await points — regular blocking calls bypass the runtime entirely. This is the sharp edge of the stackless model: you must never block the thread inside an async context. Rust catches some of this at compile time (you can't hold a mutex guard across .await), but calling a blocking OS function slips through. Go and Java don't have this problem — the runtime intercepts the blocking call and switches goroutines/virtual threads transparently.

Which one do you use?

The choice is partly technical and partly cultural, and it keeps rearranging as runtimes mature:

  • Go: goroutines are the idiomatic answer for everything. The runtime makes the stackful model work so well that you rarely think about the distinction. The cost is the GC and the runtime's opinion about how you structure your program.
  • Rust: Tokio is the dominant async runtime (it's to Rust async what the standard library is to Rust sync). The stackless model gives zero-cost abstractions — no allocation per task, no GC — at the cost of the colored-function split. Worth it when you need C-like performance with high concurrency.
  • Java: virtual threads are steadily replacing reactive frameworks (like Project Reactor and RxJava) for I/O-bound work. The pitch: "your synchronous Spring controller now scales to 10K concurrent requests with no code changes." In 2026, structured concurrency (JEP 480, stable in JDK 24) adds a scoped lifecycle to virtual threads, so a parent thread can't exit until all its children finish — fixing one of the classic sharp edges of thread-per-task models.
  • Python: asyncio is the standard, but it suffers from the two-color problem acutely — any synchronous library call blocks the event loop. Python's GIL limits parallelism anyway, so the practical answer for CPU-bound work is multiprocessing, not threading or async.
  • JavaScript/Node: the event loop with async/await (which is syntactic sugar over Promises, which are syntactic sugar over callbacks). Single-threaded by design; offload CPU work to worker threads.
the real trade The stackful vs stackless debate isn't really about stacks. It's about who pays the cost of the abstraction. Stackful runtimes pay at runtime (growable stacks, GC pressure, runtime scheduler overhead). Stackless compilers pay at compile time (state machine transform, colored functions, async propagation through the whole call graph). The right choice is the one where the cost falls on the axis you have more slack on: if you have runtime headroom, go stackful; if you need every cycle to count and can tolerate the compiler's discipline, go stackless.
