Sessions, JWTs & OAuth

Who you are, proven to a server that has no memory of you — and the three approaches to authentication that define the web.

Authentication is the oldest problem in web security, and the answer keeps changing because the constraints keep changing. A monolith with server-side rendering can use session cookies and server-side state. A mobile app talking to a dozen microservices needs something stateless — the auth server can't be involved in every service-to-service call. A SaaS platform that wants users to "log in with Google" needs delegation — the user grants limited access without sharing their password.

These three answers — sessions, JWTs, and OAuth — are not competitors. They solve different parts of the same problem, and a production system in 2026 likely uses all three at different layers. The confusion comes from treating them as either/or choices when they're components in the same stack.

Session tokens: the server remembers

The simplest model: user logs in (username + password), server validates credentials, server creates a session (a record in a database or cache with an expiry), server sends the session ID back as a cookie (Set-Cookie: session_id=a1b2c3; HttpOnly; Secure; SameSite=Lax). Every subsequent request includes the cookie; the server looks up the session to identify the user.

A session token is a pointer. The server holds the data; the client holds the key.

The advantages are real: the server can invalidate a session instantly (delete the record), session data doesn't travel over the network (the cookie is opaque), and the client can't tamper with the session contents (they only have the ID). The disadvantage: the server must store and look up session state, which is stateful and doesn't scale horizontally without a shared session store (Redis, database). Every request that needs authentication requires a session lookup, and that lookup becomes a single point of failure and a latency floor.

Sessions are the right choice for server-rendered web apps with modest scale and no API surface beyond the browser. They're simple, well-understood, and the cookie security flags (HttpOnly prevents JS access, Secure restricts to HTTPS, SameSite restricts cross-origin sending) give you a solid defense against the most common attacks.

JWTs: the client holds the data

A JSON Web Token inverts the session model. Instead of storing state on the server and sending an opaque ID to the client, encode the relevant claims directly into a signed (and optionally encrypted) token that the client carries:

A JWT, decoded
Header:   {"alg": "RS256", "typ": "JWT"}
Payload:  {"sub": "user_42", "email": "alice@example.com", "role": "admin",
           "iat": 1687381200, "exp": 1687384800}
Signature: base64(RSASHA256(header.payload, private_key))

The server can verify the token without any database lookup — just check the signature against the public key. This is stateless authentication: any service with the public key can authenticate the request. No shared session store, no Redis dependency, no network round-trip to verify identity.

The cost: you can't revoke a JWT. Once issued, it's valid until it expires (the exp claim). If a user's permissions change, or their account is disabled, the JWT they already have still says "admin." The workarounds are all imperfect: short expiry times (5–15 minutes) with refresh tokens (which are stored server-side and can be revoked), token revocation lists (which reintroduce server-side lookups, defeating the stateless purpose), or just accepting the risk window.

predict You're building an API and choose JWTs for stateless auth. You put the user's role ("admin") in the JWT payload. Three months later, an admin is demoted. Their JWT still has 55 minutes until expiry. What can they access during those 55 minutes, and what's the cheapest fix that doesn't involve a database lookup on every request?

They can access anything the "admin" role permits for those 55 minutes. The JWT is still valid, still signed, still says "admin." The cheapest fix that preserves statelessness: short-lived access tokens. Set exp to 5 minutes, not 1 hour. The demoted admin loses access within 5 minutes. To avoid making users log in every 5 minutes, issue a refresh token (stored server-side, revocable) that the client uses to get a new access token. The access token is stateless; the refresh token is stateful. You get the best of both: most requests are authenticated without a database lookup (the access token), and revocation works (the refresh token won't issue a new access token for a disabled account).

OAuth 2.0 & OIDC: delegation, not authentication

OAuth 2.0 is not an authentication protocol — it's a delegation protocol. Its job is to let a user grant a third-party application limited access to their resources without sharing their password. "Log in with Google" is built on top of OAuth via OpenID Connect (OIDC), which adds an identity layer to OAuth 2.0.

The dance involves four parties:

  1. Resource Owner: the user, who owns the data.
  2. Client: the third-party app that wants access.
  3. Authorization Server: Google's login page, the thing the user trusts.
  4. Resource Server: Google's API, the thing that holds the data.

The flow: the client redirects the user to the authorization server → the user authenticates and grants permission → the authorization server redirects back with an authorization code → the client exchanges the code (server-to-server) for an access token and optionally an ID token (a JWT with identity claims). The access token is what the client presents to the resource server. The ID token is what the client reads to know who the user is.

The authorization code step (the server-to-server exchange) is load-bearing security: the access token never touches the user's browser. The browser only sees the redirect with the code, which is useless without the client secret. This prevents token interception from browser history, referrer headers, or malicious JavaScript.

How to put them together in 2026

A realistic authentication stack in 2026:

  • External auth: OAuth 2.0 + OIDC for "log in with Google/GitHub/Apple." Your auth server federates to the identity provider. You get an ID token (who the user is) and an access token (what they can access).
  • API auth: Short-lived JWT access tokens (5–15 min), signed with RS256 or Ed25519. Services verify the signature locally using the public key from a JWKS endpoint. No database call per request.
  • Web app auth: Session cookie, backed by the same auth server. The cookie stores an opaque session ID; the server maps it to the user and their JWT access token for calling downstream services.
  • Service-to-service: mTLS or signed JWTs with short lifetimes. No user involved; the calling service's identity is proven by its certificate or its ability to sign with a private key.
the JWT footgun JWTs have a bewildering number of options, and most of them are dangerous. alg: "none" (no signature — some libraries accepted this in the past). Symmetric signatures (HS256) when you meant asymmetric (RS256) — now every service shares the signing key, and any compromised service can mint tokens for any user. The jku header (JWK Set URL) that tells the verifier where to fetch the public key — a vector for SSRF if the verifier follows untrusted URLs. The safe subset in 2026: RS256 or EdDSA, no jku, explicit algorithm whitelisting on verification, and a library that's been audited. And never put secrets in the JWT payload — the payload is base64-encoded, not encrypted. Anyone who sees the token can read the payload.
