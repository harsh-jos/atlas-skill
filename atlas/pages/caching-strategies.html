Caching Strategies & Invalidation

One of the two hard problems in computer science — and the map of solutions that makes it tractable.

Phil Karlton: "There are only two hard things in Computer Science: cache invalidation and naming things." The joke is famous because both halves are true. Naming is a design problem with no algorithm. Caching is a mathematical problem with several algorithms, all of which are correct in some cases and disastrous in others.

A cache is a smaller, faster store that holds a subset of a larger, slower store. The assumption is that recently accessed data is likely to be accessed again (temporal locality) and that data near a recently accessed location is likely to be accessed soon (spatial locality). The hard part isn't building the cache — it's knowing when the cached copy differs from the source of truth, and what to do about it.

The invalidation spectrum

All caching strategies answer one question: how long is a cached value allowed to be wrong?

  • TTL-based (time to live): The cached value is valid for N seconds. After that, fetch from the source. Simple, universal, and always slightly wrong — during the window between the source changing and the TTL expiring, users see stale data. The TTL is your staleness budget. Good when staleness is acceptable and the cost of precise invalidation exceeds the cost of occasional staleness.
  • Write-through: Write to the cache and the source simultaneously. The cache is never stale, but every write takes the latency of both stores. Good when read latency matters more than write latency, and writes are infrequent relative to reads.
  • Write-behind (write-back): Write to the cache; asynchronously flush to the source. Writes are fast (cache latency only), but the cache can lose data if it crashes before flushing. Good when write latency matters and data loss from a cache crash is acceptable (or the cache is persistent, like Redis with AOF).
  • Cache-aside (look-aside): The application manages the cache explicitly. On read: check cache, if miss, fetch from source and populate cache. On write: update source, invalidate cache. The most common pattern in application code. The danger: the thundering herd — when a hot key expires and 100 simultaneous requests all miss the cache and hammer the source simultaneously.
  • Read-through / write-through (cache-as-SoR): The cache sits between the application and the source. The application only talks to the cache. The cache handles misses (fetching from source) and writes (propagating to source). The application code is simpler; the cache is more complex.
The pattern that generates the most production incidents: cache-aside with lazy TTL extension for hot keys, but forgetting to handle the case where the source is slow — and the stale cache value is better than nothing.

The thundering herd and how to tame it

predict A key with a 60-second TTL gets 1,000 requests per second. It expires. All 1,000 requests miss the cache simultaneously and hit the database. The database, already at 80% load, can't handle the spike. Latency spikes, more requests pile up, the database tips over. What's the fix that adds zero latency to the happy path?

Probabilistic early recomputation. When a request finds that the key's TTL has only X% remaining (say, 20%), it probabilistically decides to refresh the cache before expiry. The probability increases as the TTL approaches zero. Only one of the many requests in the danger window triggers a refresh; the rest see the still-valid cached value. The key never actually expires under load. This adds zero latency to the happy path and prevents the thundering herd entirely. Redis calls this "active expiry" in some contexts; CDNs call it "stale-while-revalidate."

Cache eviction: what to throw away

When the cache is full, you must evict something. The eviction policy is the cache's theory of what won't be needed soon:

  • LRU (Least Recently Used): Evict the item accessed longest ago. Good for temporal locality. Bad for scans (a full-table scan evicts everything useful).
  • LFU (Least Frequently Used): Evict the item accessed fewest times. Good for popularity-based workloads. Bad for items that were popular historically but are now cold — they stick around forever on past popularity.
  • TTL-only: Evict by expiration time only. No access-frequency consideration. Good when the workload is uniform and you want simplicity.
  • Segmented LRU (SLRU): Separate the cache into a probationary segment and a protected segment. New items enter probation; if accessed again, promoted to protected. Protected items evicted back to probation on overflow. This is what real caches use (Memcached, Varnish).
the real cost Cache invalidation is hard because it's a distributed systems problem disguised as a data structure problem. The cache and the source are two independent systems. Keeping them consistent requires either a protocol (write-through, invalidation messages) or accepting staleness. There's no third option. The most common production caching failure is using a TTL that's too optimistic (too long → users see stale data → business logic makes decisions on stale data → writes based on stale reads corrupt the source). Set your TTL based on the business's tolerance for staleness, not on what feels reasonable to an engineer.
