Redis Beyond the Cache

It's a cache until you realize it's also a queue, a lock, a log, a rate limiter, a pub-sub bus, and occasionally a database — with an operational model to match.

Redis is introduced as "an in-memory key-value store." This is technically accurate in the same way that "a smartphone makes phone calls." The key-value store is the foundation. The data structures on top — lists, sets, sorted sets, hashes, streams, geospatial indexes, bitmaps, and HyperLogLog sketches — make Redis the most versatile tool in infrastructure. Every production system eventually grows a Redis instance (or ten), and understanding its full surface area is the difference between using it well and reinventing it poorly.

The data structures that make it dangerous

Redis isn't powerful because it's fast. It's powerful because its data structures are the right abstractions for the problems distributed systems actually have.
  • Lists (blocking ops: BLPOP, BRPOP): A reliable work queue in three commands. LPUSH queue task + BRPOP queue 0 = a blocking consumer. Multiple consumers: the list is partitioned across consumers (each task goes to one consumer). Not a message broker (no acks, no redelivery), but good enough for background job processing. Used by Sidekiq, Celery, and Bull.
  • Sorted sets (ZADD, ZRANGEBYSCORE): A leaderboard in one data structure. ZADD leaderboard 1000 player1. ZREVRANGE leaderboard 0 9 — top 10. ZRANK leaderboard player1 — your position. Sorted sets are also the implementation behind Redis's delayed task queues (score = execution timestamp, consumer polls ZRANGEBYSCORE 0 NOW).
  • Hashes (HSET, HGET, HGETALL): A row in a database. HSET user:42 name "Alice" email "a@b.com". HGET user:42 email. Hashes are memory-efficient for small field counts; Redis can store millions of small hashes in less memory than the equivalent JSON strings.
  • Streams (XADD, XREAD, XREADGROUP): The Kafka-lite data structure (added in Redis 5.0, 2018). Append-only log with consumer groups, acknowledgments, and pending message tracking. Not Kafka (no partitioning, in-memory by default), but for moderate-throughput event streams, it's operationally trivial compared to running a Kafka cluster.
  • Pub/Sub (PUBLISH, SUBSCRIBE): Fire-and-forget messaging. No persistence, no acks, no consumer groups. Publishers fire; subscribers who are connected receive; subscribers who are offline miss the message. Good for real-time notifications (WebSocket fan-out), bad for anything that needs delivery guarantees.
  • Lua scripting (EVAL): Atomic execution of multi-step operations. A token bucket rate limiter in 20 lines of Lua, running atomically inside Redis. The script has access to Redis commands and runs without interruption — no race conditions between the read and the write.

The operational model (and the sharp edges)

Redis is single-threaded for command execution. This is its superpower — no locks, no contention, every command is atomic — and its constraint. One slow command (a KEYS * on a million-key database, an SUNION on large sets) blocks everything else. The fix: use SCAN instead of KEYS, avoid O(N) operations on large data structures in production, and monitor slowlog religiously.

Persistence is optional and configurable: RDB snapshots (point-in-time dumps, good for backup), AOF (append-only file, good for durability — replay on restart), or both. AOF with fsync=everysec is the standard durability configuration: lose up to 1 second of writes on crash, but normal operations are not waiting on disk. This is a WAL by another name.

predict You use Redis as a session store. A server goes down. Redis, on the same machine, goes down with it. The AOF file is corrupted. What's the worst case for your users, and what's the operational fix that prevents it next time?

Users lose their sessions — logged out, shopping carts emptied, in-progress work lost. The fix: Redis replication + Sentinel. A replica on a different machine continuously syncs from the primary. If the primary goes down, Sentinel promotes the replica. The AOF on the primary might be corrupted (machine crash), but the replica's AOF is intact (different machine, different disk). The worst case for users shrinks from "all sessions lost" to "sessions from the last 1 second of writes (the AOF fsync window) lost." For a session store, that's acceptable. For a payment ledger, you'd use Redis Enterprise with synchronous replication or a different database entirely.

the redis-as-database trap Redis is memory-first. If your dataset outgrows RAM, Redis either evicts (if configured with a maxmemory policy) or crashes (if not). Using Redis as a primary database means your dataset must fit in memory, which is expensive at scale. The right pattern: Redis as a cache + operational store (sessions, rate limits, leaderboards, queues), with a disk-backed database as the source of truth. Redis as a database is viable for small-to-medium datasets with strict latency requirements, but the memory cost and persistence model make it a specialized choice, not a general-purpose one.

Threads worth pulling

The durability mechanism — write-ahead logging is what Redis's AOF implements. The AOF is a WAL: commands are logged before being applied, and replay rebuilds state.

The probabilistic counter — HyperLogLog is Redis's built-in cardinality estimator. Count a billion unique visitors in 12 KB of memory.
