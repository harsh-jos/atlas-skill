SOLID, Honestly

Five principles, three of which you should actually internalize — and two that have done more harm than good.

SOLID is the most-cited and most-misapplied set of principles in object-oriented design. Robert Martin (Uncle Bob) codified them in the early 2000s, building on work by Barbara Liskov, Bertrand Meyer, and others. They've been taught as gospel for two decades. Some of them earned it. Others have been stretched far beyond their useful domain and now cause the very problems they were supposed to prevent.

Here they are, with the honesty that classroom versions omit.

S: Single Responsibility Principle

A class should have one reason to change.

Real meaning: A module should be responsible to one, and only one, actor (a person or group who can ask for changes). If the CFO asks for a change to the payroll report and the CTO asks for a change to the database schema, those should be in different modules — because changes from different actors arrive at different times, and a module that serves two actors will be changed for one and break for the other.

How it goes wrong: Interpreted as "a class should do one thing," which leads to classes with one method, files with one class, and a codebase where every tiny behavior lives in a separate file. That's not single responsibility — that's single function, and it's a structuring choice, not a principle. The right granularity: a module changes for one business reason. If two business reasons would cause you to edit the same file, split it.

O: Open/Closed Principle

Statement: A module should be open for extension but closed for modification.

Honest take: In 2026, this is table stakes for library design (you can't modify a library's source, so it must be extensible) and usually overkill for application code. The OCP was designed for a world where recompilation was expensive and libraries shipped as binaries. In a world where you own the entire codebase and can refactor across module boundaries in one PR, adding an abstraction layer so you can add behavior without modifying existing code is often premature. The right heuristic: add the abstraction when the second implementation arrives, not the first. A single implementation behind an interface is a guess about how the second implementation will differ. The second implementation tells you what the interface should actually look like.

L: Liskov Substitution Principle

Statement: Subtypes must be substitutable for their base types without altering the correctness of the program.

Verdict: internalize this one. LSP is the most mathematically sound of SOLID and the one that still causes real bugs when violated. The classic violation: a Rectangle base class with setWidth and setHeight, and a Square subclass that overrides setWidth to also set height. Code written for Rectangle breaks when given a Square, because the invariants differ (a square's width and height are coupled; a rectangle's aren't). The fix isn't inheritance — it's not using inheritance for a relationship where the subclass doesn't preserve the superclass's contracts. Composition usually wins.

I: Interface Segregation Principle

Statement: No client should be forced to depend on methods it does not use.

Verdict: internalize this one — it's the API design principle. A fat interface (50 methods) forces every implementation to stub out 40 of them, and a change to any method forces a recompile of every implementation. Split the interface into role-specific ones: Reader, Writer, Deleter. Each client depends on the role it actually needs. This is also the principle behind ports & adapters: the port is the narrow interface; the adapter is the concrete implementation.

D: Dependency Inversion Principle

Statement: Depend on abstractions, not concretions.

Verdict: internalize this one, with a caveat. DIP is the principle behind dependency injection and hexagonal architecture's driven ports. High-level policy (business logic) should not depend on low-level details (which database, which HTTP library). Both should depend on abstractions. The caveat: "abstractions" doesn't mean "interfaces for everything." It means the high-level module defines the interface it needs, and the low-level module implements it. The dependency arrow points from low-level to high-level (the database adapter depends on the business logic's repository interface, not the other way around). This is the inversion: the database depends on the business logic, not vice versa.

the meta-principle SOLID was designed for large, long-lived, multi-team object-oriented codebases. If you're writing a script, a prototype, or a service that's 500 lines and will be replaced next quarter, SOLID is overhead. The principles earn their keep when the cost of not applying them — the cost of tangled dependencies, fragile base classes, and fat interfaces that break everything when changed — exceeds the cost of the extra abstractions. That threshold is lower than most engineers think for production systems and higher than most SOLID enthusiasts admit for small ones.
