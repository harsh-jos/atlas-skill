How TLS Actually Works

The handshake that turns an open wire into a private channel — not the certificate-authority bureaucracy, but the actual cryptographic machinery that runs in the first few milliseconds of every HTTPS connection you've ever made.

Every HTTPS connection begins with a paradox: two strangers — your browser and a server — must agree on a secret while an eavesdropper watches every message they exchange. Public-key cryptography makes this possible, and TLS is the protocol that orchestrates it, fast enough that you never notice the few hundred milliseconds it costs.

What TLS gives you (and what it doesn't)

TLS provides three guarantees on top of a raw TCP connection:

Confidentiality. Nobody but you and the server can read the bytes. The connection is encrypted with a symmetric cipher — fast, provably secure, and negotiated during the handshake.

Integrity. Nobody can modify the bytes in transit without detection. Each record carries an HMAC or AEAD authentication tag. If a single bit flips or is tampered with, the connection is torn down.

Authentication. The server proves its identity. The client can optionally prove its identity too (mutual TLS, less common on the public internet, standard inside service meshes).

TLS does not hide metadata: an observer can see the server's IP, the SNI hostname (in TLS 1.2; encrypted in TLS 1.3's Encrypted Client Hello), the approximate data volume, and the timing of requests. Traffic analysis is a real attack; TLS prevents content inspection, not metadata surveillance.

The TLS 1.2 handshake: the classic choreography

A TLS 1.2 handshake takes 2 round trips (4 if you count TCP's SYN handshake). Here's every step:

1. ClientHello. The client sends: the highest TLS version it supports, a random nonce (28 bytes), a list of cipher suites it's willing to use (e.g., TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256), and the server name (SNI) so the server knows which certificate to present.

2. ServerHello + Certificate + ServerKeyExchange + ServerHelloDone. The server responds: the chosen TLS version, its own 28-byte random nonce, the chosen cipher suite, and its certificate chain (proving ownership of the domain). Then, for ECDHE key exchange: the server picks an elliptic-curve private key, computes the corresponding public point, signs it with the certificate's private key, and sends it.

3. ClientKeyExchange. The client verifies the certificate chain (signature, expiration, domain match). Then it picks its own ECDHE private key, computes the public point, and sends it. Both sides now have each other's public keys and their own private keys. They compute the pre-master secret via elliptic-curve Diffie-Hellman: combine your private key with the peer's public key, and math guarantees you both get the same result. An eavesdropper with both public keys cannot compute it — the discrete-log problem.

The pre-master secret, combined with both random nonces, is fed through a pseudorandom function (HMAC-based in TLS 1.2) to derive the master secret, which is then expanded into four session keys: client-write encryption key, server-write encryption key, client-write MAC key, server-write MAC key.

4. ChangeCipherSpec + Finished. Both sides send a message that says "everything after this is encrypted with the session keys" followed by a Finished message — the first encrypted record, containing a MAC of the entire handshake transcript. If the Finished messages verify, both sides saw the same handshake and no one tampered with it.

The Diffie-Hellman trick: you and I exchange public numbers, combine them with our private numbers, and arrive at the same shared secret — while an observer who sees both public numbers learns nothing useful.

TLS 1.3: faster, simpler, and forward-secret by default

TLS 1.3 (RFC 8446, 2018) cut the handshake to 1 round trip — and 0-RTT for resumed connections — by removing everything that had been broken or was unnecessary:

Static RSA key exchange is gone. In TLS 1.2, one mode used RSA: the client encrypted the pre-master secret with the server's RSA public key from the certificate. This meant anyone who later obtained the server's private key could decrypt all past connections — no forward secrecy. TLS 1.3 mandates ephemeral Diffie-Hellman: every connection uses a fresh key pair, so compromising the long-term certificate key reveals nothing about past sessions.

Fewer cipher suites. TLS 1.2 had ~300 cipher suites; TLS 1.3 has 5. Everything non-AEAD (CBC mode, RC4, static RSA) was removed. AEAD (authenticated encryption with associated data) combines encryption and integrity in one operation — you can't forget to MAC after encrypting.

The server sends its key share immediately. In TLS 1.3, the ClientHello guesses which key-exchange group the server supports (almost certainly X25519) and includes the client's public key share. The server responds with its public key share in ServerHello — one round trip saved. If the client guessed wrong, the server sends a HelloRetryRequest, and the client retries; this happens essentially never in practice.

0-RTT resumption. If you've connected to a server before, TLS 1.3 can send application data in the first flight — before any round trip completes. The client uses a pre-shared key (PSK) from the previous session to encrypt early data. The trade: 0-RTT data is not forward-secret (replayable if the PSK is compromised) and susceptible to replay attacks. Servers typically limit 0-RTT to idempotent requests (GETs, not POSTs).

The certificate: who vouches for whom

The handshake's key exchange proves the server possesses the private key matching the certificate. But how does the client know the certificate belongs to the right server? The certificate chain:

The server's certificate is signed by an intermediate CA's private key. The intermediate's certificate is signed by a root CA's private key. The root CA's certificate is self-signed and shipped with the OS or browser. The client verifies each link in the chain: signature valid, not expired, not revoked (via OCSP stapling or CRL), and the domain matches (via the certificate's SAN field).

This is the Public Key Infrastructure (PKI). It's a trust-on-first-use model: you trust the root CAs your OS trusts. If any of them is compromised (and they have been — DigiNotar, 2011; WoSign, 2016), every domain under that CA is at risk.

predict An attacker controls a root CA and issues a certificate for google.com. They MITM your connection to Google, presenting this fraudulent certificate. What detects this? And why did Certificate Transparency change the answer?

Before Certificate Transparency (CT), nothing automatically detected this — the certificate was validly signed, and the attack was invisible unless someone noticed the certificate wasn't the one Google's known pin set expected (HPKP, now deprecated). CT (mandatory in Chrome since 2015) requires CAs to publish every certificate they issue to public, append-only logs. Domain owners and browsers monitor these logs; a fraudulent certificate for google.com is visible to Google within minutes. CT turned a silent failure mode into a loud one.

watch out TLS encrypts the connection, not the endpoints. If the server is compromised, your data is exposed regardless of TLS. If the client has malware, TLS doesn't help. And if you're using TLS but your application code logs the plaintext, the cryptographic guarantees are irrelevant. TLS is transport security — it protects data in flight, not at rest and not in memory.
