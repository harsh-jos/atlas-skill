TCP, Past the Handshake

Congestion control, head-of-line blocking, and why TCP feels slow sometimes — the machinery under every connection you've ever made.

You know the three-way handshake: SYN, SYN-ACK, ACK. That's the opening ceremony. Everything interesting about TCP happens after the handshake — in the algorithms that decide how fast to send data, when to retransmit, and how to share a bottleneck link fairly with every other connection on the internet.

These algorithms have been evolving since 1988, and they're the reason your Zoom call doesn't freeze every time someone else on your WiFi starts a download. They're also the reason a single lost packet can stall an HTTP/2 connection for hundreds of milliseconds. TCP is a 40-year-old protocol that works astonishingly well, and understanding its internals is understanding why the internet feels the way it does.

Congestion control: the internet's distributed resource allocation

TCP doesn't know how much bandwidth is available. It probes, backs off, and probes again — treating packet loss as the signal that the network is full.

The fundamental problem: a sender has no way to know the available capacity of the path between it and the receiver. The network doesn't tell it. Routers don't send "slow down" messages (ECN is the exception that proves the rule). The only signal the sender gets is packet loss — a dropped packet, inferred from a missing acknowledgment. So TCP treats loss as a congestion signal: if packets are being dropped, the network is full, and the sender should slow down.

The algorithm that defined this approach is Reno (1990), and its pattern — a sawtooth of increasing speed followed by sharp cuts on loss — is still the shape of most TCP connections:

TCP Reno congestion control
Slow start:        double cwnd every RTT until first loss
Congestion avoidance: increase cwnd by 1 MSS per RTT (additive increase)
On loss (triple dup ACK): cwnd = cwnd / 2, enter fast recovery
On timeout:         cwnd = 1 MSS, enter slow start (much more severe)

The congestion window (cwnd) is the number of unacknowledged bytes the sender allows in flight. It's the sender's estimate of the network's capacity. Slow start probes exponentially (double every RTT) until it hits a loss, then backs off multiplicatively — hence additive increase, multiplicative decrease (AIMD). This is the algorithm that keeps the internet from collapsing under load, and it's running in every TCP stack on every device that connects to it.

Modern variants — CUBIC (Linux default), BBR (Google, used by YouTube and Google Cloud) — improve on Reno in different ways. CUBIC uses a cubic function instead of linear increase, growing faster when far from the last loss point and slower when near it. BBR doesn't use loss as a congestion signal at all — it models the path's bandwidth and RTT explicitly and tries to send at exactly the bottleneck rate. BBR can achieve higher throughput on lossy paths (where Reno would interpret random loss as congestion), but it can also be unfair to Reno flows sharing the same bottleneck.

Head-of-line blocking: TCP's original sin

TCP guarantees in-order delivery. If the sender transmits packets 1, 2, 3, 4, 5 and packet 2 is lost, the receiver buffers packets 3, 4, and 5 — but it cannot deliver them to the application until packet 2 is retransmitted and received. The entire byte stream stalls behind the missing packet. This is head-of-line blocking.

For HTTP/1.1 with persistent connections, this means request 2 stalls behind a lost packet from request 1. For HTTP/2 with multiplexing, this means every stream stalls behind the lost packet, because all streams share the same TCP byte stream. HTTP/2 fixed HOL blocking at the application layer but inherited it from the transport layer. HTTP/3's decision to move to QUIC (per-stream loss recovery over UDP) is a direct response to this — the only way to truly fix HOL blocking is to replace the transport.

Why TCP can't just be replaced

TCP is implemented in the OS kernel. QUIC runs in userspace. This means QUIC can evolve faster (userspace code ships weekly; kernel code ships with OS releases), but it also means QUIC pays a CPU penalty for context switches between kernel and userspace. TCP has decades of hardware offload — NICs that compute checksums, segment data, and manage retransmissions in silicon. QUIC is catching up (GSO/GRO for UDP, kernel bypass with DPDK), but as of 2026, TCP still has a slight edge in raw throughput on clean paths.

The other reason: TCP incast — the problem where many senders converge on one receiver (think: a distributed storage system where 100 servers respond to one query simultaneously). The receiver's switch buffer overflows, packets drop, and all 100 senders back off simultaneously, creating a throughput collapse. TCP's congestion control wasn't designed for this pattern; data center TCP variants (DCTCP) use ECN to signal congestion before loss occurs, giving senders a gentler signal. QUIC inherits the same problem and the same solutions (ECN support in QUIC).

the pragmatic take For most application developers, TCP's internals are invisible — and should be. The socket API hides them. But when your mobile app feels slow on a train, or your video call degrades on hotel WiFi, you're feeling TCP's congestion control responding to packet loss. The fix isn't tuning TCP parameters (don't — the defaults are smarter than you are for general workloads). The fix is understanding that TCP interprets loss as congestion, and loss is high on wireless networks, so your users on mobile are getting throttled by an algorithm that's mistaking radio interference for a full pipe. HTTP/3 directly addresses this, and it's the strongest argument for migrating.
