Atomics & Lock-Free Programming

Coordinating threads without ever taking a lock — using the hardware's native compare-and-swap to build data structures that can't deadlock.

Locks solve concurrency by serialization: one thread at a time. The cost is contention — N-1 threads wait while thread N holds the lock. For most application code, this is fine. For the hot path of a database, a high-frequency trading engine, or an OS kernel, it's not. You need coordination without waiting.

Lock-free programming uses hardware-provided atomic operations — instructions that read, modify, and write memory as a single indivisible unit — to build concurrent data structures where threads never block. A thread might need to retry (if another thread interfered), but it never waits on a lock holder. Progress is guaranteed without serialization.

A lock says "wait your turn." An atomic says "try, and if someone else got there first, try again."

CAS: the universal atomic primitive

Compare-and-swap (CAS) is the atomic operation that makes lock-free programming possible. It takes a memory location, an expected old value, and a new value. If the location currently holds the expected value, it atomically writes the new value and returns true. Otherwise, it returns false and leaves the location unchanged. The entire read-compare-write sequence happens in one hardware instruction (LOCK CMPXCHG on x86, LDREX/STREX on ARM).

CAS, conceptually (but it's a single instruction)
bool CAS(addr, expected, new):
    atomically:
        if *addr == expected:
            *addr = new
            return true
        else:
            return false

The pattern for using CAS: read the current value, compute the desired new value, attempt CAS. If CAS fails (someone else changed the value between your read and your CAS), retry. This is the CAS loop, and it's the fundamental building block of lock-free data structures:

Lock-free counter increment
def increment(counter):
    do:
        old = atomic_read(counter)
        new = old + 1
    while !CAS(counter, old, new)

No lock is ever acquired. Under low contention, CAS succeeds on the first try. Under high contention, threads retry, but they're making progress (someone's CAS succeeds each cycle) rather than waiting for a lock holder to finish. This is the lock-free guarantee: at least one thread makes progress in any finite number of steps.

Memory ordering: the sharp edge

Atomic operations aren't just about indivisibility — they're about visibility. A plain store from one thread might not be visible to another thread for an arbitrarily long time (due to store buffers, cache coherence, and compiler reordering). Atomics let you specify what else must be visible when the atomic operation completes.

The ordering options, from weakest to strongest:

  • Relaxed: Only atomicity is guaranteed. No ordering with respect to other memory operations. Cheapest. Use for simple counters where you only care about the final value, not when other threads see it.
  • Acquire (load) / Release (store): An acquire-load ensures all subsequent reads and writes see the effects of the release-store that wrote that value. Together, they form a happens-before relationship between two threads. This is the minimum you need for most producer-consumer patterns.
  • Sequentially consistent (seq_cst): All threads see all seq_cst operations in a single global order. The strongest guarantee, the most expensive, the default in most languages. Use when you're not sure — it's never wrong, just potentially slower.

The practical rule: use acquire/release when you're building a specific synchronization pattern (flag, queue, lock-free stack). Use relaxed for counters where ordering doesn't matter. Use seq_cst (the default) when you're unsure or the cost isn't measurable. Getting memory ordering wrong produces bugs that reproduce on ARM but not x86, only under load, once a week.

ABA: the CAS bug that looks like it worked

predict Thread A reads the top of a lock-free stack: node X. Thread B pops X, pops Y, then pushes X back. Thread A now does CAS, comparing the old top (X) with the current top — which is also X. CAS succeeds, but the stack's structure is now wrong. What is this called, and what's the fix?

This is the ABA problem. Between thread A's read and its CAS, the value changed from A to B and back to A. CAS sees A and says "nothing changed," but the structure changed profoundly. The fix: attach a version tag (or sequence number) to the pointer. Instead of CAS on just the pointer, CAS on the pointer+tag as a single wide value (128-bit CAS on x86, or a double-word CAS). Each modification increments the tag, so even if the pointer is the same, the tag differs and CAS correctly fails. Languages with garbage collection avoid ABA naturally (a popped node is still allocated, so its address can't be reused immediately), but in C/C++ and Rust, ABA is a real hazard.

when not to go lock-free Lock-free data structures are harder to write, test, and debug than their locked counterparts. A correctly implemented lock-free queue is a minor miracle of engineering. An incorrectly implemented one fails silently and non-deterministically. The standard advice: use the language's built-in lock-free structures (Java's ConcurrentLinkedQueue, Rust's crossbeam, C++'s concurrent_queue). Only write your own if you've profiled the built-in one, found it's the bottleneck, and can explain the memory ordering of every atomic operation in your replacement. The number of people who should write a lock-free data structure from scratch is small and shrinking.
