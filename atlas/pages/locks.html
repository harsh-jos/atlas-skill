Locks, Deadlock & Livelock

The cost of mutual exclusion and the classic ways it goes wrong — from the dining philosophers to your production database.

A race condition happens when two threads touch the same data without coordination. The simplest coordination is a lock: before accessing shared state, acquire the lock. After modifying it, release the lock. While one thread holds the lock, any other thread that tries to acquire it blocks until the holder releases. This is mutual exclusion — the fundamental primitive that every other concurrency tool builds on.

Locks solve one problem and create three new ones. The three new ones — deadlock, livelock, and contention — have killed more production systems than race conditions ever did. A race condition is a needle in a haystack. A deadlock is the haystack catching fire.

Mutexes, spinlocks, and the cost of waiting

When a thread can't acquire a lock, it has two choices:

  • Block (mutex): The OS deschedules the thread. It goes to sleep. When the lock is released and the thread is next in line, the OS wakes it. The cost: a context switch (~microseconds). Best when the expected wait time is longer than the context switch cost.
  • Spin (spinlock): The thread stays on the CPU and repeatedly checks the lock in a tight loop. No context switch, but the CPU burns cycles doing nothing useful. Best when the expected wait time is shorter than a context switch — i.e., the critical section is a few instructions long. Spinlocks are the building block inside OS kernels and database internals; you rarely write one in application code.
A mutex is a "please wake me when it's my turn." A spinlock is "it'll be free in a nanosecond, I'll just wait here."

Application-level mutexes (Rust's std::sync::Mutex, Go's sync.Mutex, Java's ReentrantLock) are almost always the right choice. They're fast enough for most critical sections, they don't burn CPU, and they compose with the language's concurrency primitives. Spinlocks are for when you're writing an async runtime or a database buffer pool manager — code where the critical section is a few atomic operations and the cost of descheduling exceeds the cost of spinning.

Deadlock: the circular wait

Thread A holds lock 1 and waits for lock 2. Thread B holds lock 2 and waits for lock 1. Neither can make progress. Both are deadlocked. The four necessary conditions (the Coffman conditions): mutual exclusion, hold-and-wait, no preemption, circular wait. Break any one and deadlock is impossible.

Deadlock in four lines
// Thread A:           // Thread B:
lock(mu1);              lock(mu2);
lock(mu2);  // waits    lock(mu1);  // waits
// ...                  // ...

The practical prevention, applied in every production codebase: always acquire locks in a consistent global order. If every code path that needs mu1 and mu2 acquires mu1 first, you can't form a cycle. This is why lock ordering is the first thing a deadlock postmortem examines — and why adding a new lock to a system is a higher-stakes change than adding a new function.

Databases detect deadlocks by building a waits-for graph (nodes are transactions, edges are "X is waiting for a lock held by Y") and periodically checking for cycles. If a cycle is found, the database picks a victim — typically the transaction that has done the least work — and aborts it. The application receives a deadlock error and is expected to retry. Postgres, MySQL/InnoDB, and SQL Server all do this automatically. It's expensive (the detector runs every deadlock_timeout ms) but safer than trusting application code to never produce a cycle.

predict A philosopher picks up the fork on her left. Before she can pick up the fork on her right, you preempt her — take the left fork away — and let another philosopher try. Does this prevent deadlock? What's the new failure mode you just introduced?

Yes, breaking the "no preemption" condition prevents deadlock. The new failure mode is livelock: every philosopher picks up the left fork, gets preempted, puts it down, tries again, gets preempted again. The system is making progress in the sense that the CPU is busy and threads are running, but no philosopher ever eats. This is livelock — the system is alive (processes are running) but no useful work gets done. The fix is randomization: each philosopher waits a random amount of time before retrying, so eventually one completes the full acquire before anyone else tries. This is exponential backoff — the same pattern used in Ethernet (CSMA/CD) and every retry-based system since.

Reader-writer locks: the concurrency cheat

If most of your operations are reads, a plain mutex is wasteful — readers don't conflict with each other, only with writers. A reader-writer lock (RwLock) lets multiple readers hold the lock simultaneously, but writers need exclusive access. The standard implementation: readers increment a shared counter (atomically); writers wait for the counter to hit zero, then set an exclusive flag.

The pitfall: under heavy read traffic, writers can starve. If readers keep arriving while a writer is waiting, the counter never hits zero, and the writer waits indefinitely. The fix is write-preferring (block new readers while a writer is waiting) or fair queuing. Most language standard libraries default to OS-level fairness, but the details vary. Go's sync.RWMutex is write-preferring as of Go 1.19.

the real cost Locks don't just cause deadlocks. They cause contention — the hidden tax of serializing parallel work. If 8 threads need the same mutex, 7 of them are waiting at any moment. Your 8-core machine is effectively single-core for that critical section. The fix isn't a better lock — it's reducing the critical section (do the minimum necessary under the lock, compute everything else outside it), sharding the lock (one mutex per hash bucket, not one mutex for the whole map), or avoiding shared state entirely (message passing, copy-on-write). A lock that's contended is a design smell, not a performance problem to tune around.
