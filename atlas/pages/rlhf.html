RLHF & Preference Tuning

How a raw text predictor becomes an assistant that follows instructions — the alignment step that separates base models from the ones you actually use.

A base language model trained on the internet knows how to complete text. It does not know how to be helpful, harmless, or honest. If you ask it "how do I make a bomb?" it will complete that text — not because it's evil, but because "text completion" doesn't include the concept of "refuse harmful requests." Teaching the model which completions humans prefer is the alignment step, and RLHF is how every frontier model does it.

RLHF (Reinforcement Learning from Human Feedback) is the three-stage process that turns a base model into an assistant:

  1. Supervised Fine-Tuning (SFT): Collect demonstrations of desired behavior (human-written or model-generated + human-edited prompt-response pairs). Fine-tune the base model on these. The model learns the format of instruction-following: what a response to a user request looks like.
  2. Reward Model Training: For a given prompt, generate multiple responses. Have humans rank them (A > B > C > D). Train a reward model to predict the human ranking. The reward model is a copy of the SFT model with a regression head instead of a language-modeling head — it outputs a scalar score instead of a token distribution.
  3. RL Fine-Tuning (PPO): Use the reward model as a reward function. The policy (SFT model) generates responses; the reward model scores them; PPO (Proximal Policy Optimization) updates the policy to maximize the reward while staying close to the SFT model (via KL divergence penalty — the model shouldn't drift so far that it loses language coherence).
RLHF doesn't teach the model new facts. It teaches it which facts are appropriate to share and how to say "I don't know" when it doesn't.

DPO: the simpler alternative that took over

RLHF with PPO is complex: you need a separate reward model, you need to keep it in sync with the policy, and PPO training is notoriously finicky. DPO (Direct Preference Optimization, Rafailov et al., 2023) eliminates the reward model entirely. It reparameterizes the RLHF objective so that the optimal policy can be expressed directly in terms of the preference data, without an explicit reward function. Training DPO looks like supervised fine-tuning: you have a dataset of (prompt, chosen_response, rejected_response) triples, and the loss function encourages the model to assign higher probability to the chosen response than the rejected one.

RLHF vs DPO complexity
RLHF:  SFT model → train reward model → train policy with PPO → deploy
DPO:   SFT model → train directly on preference pairs → deploy

DPO has become the default alignment method for most teams in 2025–2026. It's simpler, more stable, and achieves comparable results to PPO-based RLHF on most benchmarks. PPO is still used when the reward is complex, non-differentiable, or comes from a system external to the model (e.g., a code execution sandbox that returns pass/fail, or a human-in-the-loop feedback system).

GRPO: RL without the critic

The newest addition to the alignment toolbox: GRPO (Group Relative Policy Optimization, DeepSeek, 2024). Instead of training a separate value function (critic), GRPO samples multiple responses per prompt, scores them all with the reward model, and uses the group's mean and standard deviation to normalize rewards. This eliminates the critic — a major simplification — while preserving RL's ability to optimize non-differentiable rewards. GRPO was used in DeepSeek-R1's training and is the recommended method for reasoning tasks with verifiable rewards (math problems where the answer can be checked, code generation where tests can be run).

predict You train a reward model on human preferences. Your raters consistently prefer longer, more confident-sounding responses — even when they're wrong. What property of your aligned model breaks, and what's the data fix?

The model becomes sycophantic — confidently wrong, preferring to sound authoritative rather than admit uncertainty. The data fix: explicitly include "I don't know" and uncertainty-expressing responses in the preference data, and ensure they're ranked above confident-but-wrong responses. Some teams add a "honesty bonus" to the reward function. The deeper fix: have raters evaluate factual correctness (not just perceived helpfulness), which is expensive (requires domain experts) but produces a reward model that actually values truth. The reward model learns what you measure. If you measure "which response looks better," it learns style over substance. If you measure "which response is correct," it learns accuracy.

the alignment tax RLHF makes models safer and more helpful. It also makes them less capable on some dimensions — the "alignment tax." A base model might score higher on scientific reasoning benchmarks than its RLHF'd version because the base model learned scientific patterns from papers and the RLHF process (with raters who aren't domain experts) inadvertently penalized the terse, technical style of scientific writing. The alignment tax is shrinking as alignment techniques improve, but it's not zero. The honest trade: RLHF trades some raw capability for safety and usability. For a model that serves the public, that trade is non-negotiable. For a model that serves researchers solving protein folding, you might prefer the base model.
