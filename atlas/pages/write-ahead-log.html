Write-Ahead Logging

The trick that lets a database survive a power cut mid-write — and the reason your data is still there when you come back.

Picture a database in the middle of updating a row. It's modified the index page in memory, flipped a few bits in the heap, and adjusted a pointer. The disk is halfway through writing these changes. Then the power goes out.

When the machine comes back, the database opens its files and finds chaos: a half-written page here, a pointer to nowhere there. Without some other source of truth, that row — and possibly the whole table — is corrupted. Not just the row you were writing; the structural integrity of the database itself is compromised, because index pages and data pages disagree about where things live.

This is the problem durability is meant to solve, and write-ahead logging is how almost every serious database solves it.

The one rule

Before you change the data, write down what you're about to do.

That's it. Before any modification touches the actual data files, the database writes a record to an append-only log describing the change. Only after that log record is safely on disk does it modify the real data structures in memory — and eventually flush those to disk too.

The log is the source of truth. The data files are a cache of the log at a point in time. If the machine dies, on restart the database replays the log: anything that was written to the log but not yet reflected in the data files gets re-applied. Anything that never made it to the log never happened.

This sounds simple, but the implications run deep. Let's follow a single write through the system.

A write, step by step

You execute UPDATE accounts SET balance = 50 WHERE id = 7. Here's what actually happens inside a WAL-backed database:

  1. Build the log record. The database constructs a record that says, roughly: "at time T, in transaction 42, row id=7 in table accounts changed column balance from 40 to 50." It includes the old value (for undo) and the new value (for redo).
  2. Write to the WAL. This record is appended to the end of the write-ahead log file. The database calls fsync() (or its OS equivalent) to make sure the bytes actually hit disk, not just the OS buffer cache.
  3. Now it's durable. Once fsync() returns, the database can tell the client "committed." The change will survive a crash. The actual table data in memory and on disk might still be the old version — that's fine. The log has the truth.
  4. Apply to in-memory data structures. The buffer pool page for this row is updated. Other transactions can now see the new value.
  5. Eventually, flush to data files. At some future checkpoint, the database writes the dirty pages to the main data files and records "log entries up to position X have been applied" — so on the next restart, it can skip replaying those entries.

The critical ordering is step 2 before step 4. The WAL write must happen before the data structure modification. If you do it the other way around and crash between them, the data file says "50" but the log has no record of it — and if another part of the system also modified that row, replay might quietly overwrite your change with something else. The WAL-first ordering means the worst case is redoing work you already did, never losing work you thought was done.

What the log actually looks like

A WAL isn't a text file. It's a sequence of binary records, each with a monotonically increasing Log Sequence Number (LSN). A typical record contains:

WAL record structure (conceptual)
┌──────────┬────────┬──────────┬──────────────┬───────────┬────────┐
│ LSN (8B) │ Txn ID │ Page ID  │  Operation   │ Old value │ New    │
│          │        │          │  (insert/    │  (undo)   │ (redo) │
│          │        │          │   update/del)│           │        │
└──────────┴────────┴──────────┴──────────────┴───────────┴────────┘

In practice, databases use various optimizations. Physiological logging records logical operations (what the user asked for) scoped to physical pages (where it happened). This gives you the best of both: the log record is small (logical), but recovery can reapply it to a specific page without re-executing SQL (physical).

The LSN ordering is what makes everything work. Every page in the database stores the LSN of the last log record that modified it. During recovery, the database compares each page's LSN to the log: if the log has entries past the page's LSN, those entries need to be replayed onto that page.

Checkpoints: stopping the log from eating your disk

If you appended to the WAL forever, you'd run out of disk. Worse, recovery would take hours — replaying a year of log entries on restart is not a good time. The answer is checkpointing.

A checkpoint does two things:

  1. Flush all dirty pages from the buffer pool to the data files.
  2. Write a checkpoint record to the WAL that says "all log entries before LSN N have been applied to the data files."

On recovery, the database finds the last checkpoint record, jumps to that LSN, and replays only the entries after it. Entries before the checkpoint can be safely ignored — and if the WAL is split into segments, those old segments can be recycled or deleted.

Checkpointing is a balancing act. Checkpoint too often and you hammer the disk with writes, killing throughput. Checkpoint too rarely and recovery takes forever. Most databases let you tune this with a target recovery time ("I'm willing to wait up to 30 seconds on restart") or a WAL size cap.

Where the WAL shows up, beyond the obvious

The WAL is the engine under a surprising number of features that seem unrelated:

  • Replication. Instead of shipping data pages, the primary ships WAL entries to replicas. The replica replays the same log and arrives at the same state. This is physical replication (ship the WAL) vs logical replication (ship the SQL). WAL shipping is faster and more precise — the replica is byte-for-byte identical.
  • Point-in-time recovery. Since the WAL is an ordered history of every change, you can replay it up to any LSN. Combine this with a base backup and you can restore the database to any moment — including right before someone ran DROP TABLE.
  • Rollback. The old-value (undo) information in each WAL record is what lets a transaction abort halfway through: the database reads the undo records backward and reverses each change.
  • Crash recovery for in-memory databases. Redis uses an append-only file (AOF) that is conceptually a WAL: commands are logged before they're applied, and replay rebuilds state. Even "pure cache" systems end up needing this pattern the moment durability matters.
your call A database writes the WAL entry, calls fsync(), then modifies the buffer pool page. But fsync() is slow — it might take milliseconds. What if you skip the fsync() and batch WAL writes, flushing every 100ms instead of every commit? Before you read on: what did you just trade away, and when is that trade correct?

If you batch WAL flushes, you're trading durability for throughput. A transaction that committed in the last 100ms window might vanish if the machine dies. This is group commit — the database acknowledges multiple transactions in one fsync — and it's the default in many systems because the throughput gain is substantial and the data-loss window is bounded and known.

Postgres calls this synchronous_commit = off. MongoDB calls it w: 1 with journaling. The trade is explicit: you lose up to N milliseconds of committed transactions on crash, but you gain 10–100x write throughput. For a social media feed, that's probably fine. For a bank ledger, it's not. The WAL gives you a knob; you decide where to set it.

The WAL is everywhere

Once you see the pattern, you notice it in places that don't call themselves databases:

  • Filesystems. ZFS and ext4 journal metadata changes before applying them. A journal is a WAL. Same idea, different name.
  • LSM-trees. Writes go to an in-memory memtable and an on-disk WAL. If the process dies, the memtable is rebuilt from the WAL. (This is why an LSM-tree can acknowledge writes at memory speed.)
  • Kafka. Every message is appended to a log. Consumers track their position in that log. The log is the database.
  • Git. Each commit points to a tree, which points to blobs. The reflog is a WAL of where HEAD has been. git reflog is recovery for your branch pointers.
when not to If your writes are idempotent and your dataset is small enough to reconstruct from scratch quickly, a WAL may be overkill. A cache that can repopulate from a source of truth on restart doesn't need one. And for write-once, read-many workloads (immutable data, append-only tables), the data files are the log — adding a WAL in front of them is redundant. The WAL earns its keep when writes are frequent, mutable, and painful to lose.
