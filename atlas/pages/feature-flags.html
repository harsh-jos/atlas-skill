Feature Flags & Progressive Delivery

Decoupling "deployed" from "released" — and the operational debt that decision quietly accrues.

At most companies, "deploy" and "release" are the same verb. You push code to production, and users see it. This coupling is expensive: a bad deploy is immediately a bad release, and rolling back means rolling back code. If a deploy takes 20 minutes and rollback takes 20 minutes, a bad change means 40 minutes of user-facing impact — plus the blast radius of whatever the broken code did during those 40 minutes.

A feature flag (or feature toggle) decouples them. The code deploys disabled — behind an if (featureFlag("new-checkout")). The flag is off for everyone. Sometime later — minutes, days, weeks — you turn the flag on. Now the code is released. If something goes wrong, you turn the flag off. Release and rollback are instant, because no code needs to deploy.

A feature flag turns a deployment from a binary risk (deploy = release) into a controlled dial (deploy, then release to 1%, then 10%, then 100%).

Beyond on/off: the flag taxonomy

Not all flags are created equal. Pete Hodgson's taxonomy (from the Martin Fowler article that defined the space) is still the standard:

  • Release flags: Hide unfinished code. The "trunk-based development with feature flags" pattern: merge incomplete features to main, ship them disabled, activate when ready. These should be short-lived (days to weeks) and removed once the feature ships. Long-lived release flags are a warning sign — the feature is probably abandoned.
  • Experiment flags: A/B tests. 50% of users see variant A, 50% see variant B. Measure conversion, engagement, or whatever metric you're optimizing. These are medium-lived (weeks to months, the duration of the experiment).
  • Operational flags: Kill switches for infrastructure. "Disable the recommendation engine if it's overwhelming the database." These are long-lived — they protect against failures you hope never happen.
  • Permission flags: "This feature is only for internal users" or "only for users on the enterprise plan." These are long-lived and tied to business logic, not development velocity.

The distinction matters because each type has different lifecycle expectations. A release flag that's still in the code two years later is technical debt. An operational flag that's two years old is a safety net that hasn't been tested recently. The cleanup strategy differs.

Progressive delivery: the dial, not the switch

Turning a feature on for 100% of users at once is a feature flag, but not progressive delivery. Progressive delivery uses the flag as a dial:

  1. Deploy code (flag off for everyone).
  2. Enable for internal users / dogfooding (0.1%). Watch metrics for an hour.
  3. Enable for 1% of external users. Watch metrics for a day.
  4. Enable for 10%. Watch for a day.
  5. Enable for 50%. Watch.
  6. Enable for 100%. Remove the flag.

At each step, you're checking: did error rates change? Did latency increase? Did the business metric (conversion, engagement) move in the right direction? If any check fails, you roll back the flag — which is instant. The blast radius of a bad release is the percentage you've rolled out to, not 100%.

A canary deploy is the infrastructure variant of the same idea: deploy new code to a small subset of instances, route a fraction of traffic to them, and monitor. If the canary instances show elevated error rates or latency, stop the deploy and roll back. Kubernetes natively supports this with rolling updates and health checks; service mesh tools (Istio, Linkerd) add traffic-splitting on top. The pattern is the same as feature flags: don't bet the farm on "it worked in staging."

The debt that flags accrue

the real cost Every flag doubles the number of code paths for the feature it gates. Two flags = 4 paths. Five flags = 32 paths. Your test matrix explodes combinatorially, and testing every combination is infeasible. The practical answer: test the flag-on and flag-off states independently (not combinatorially), and rely on monitoring in production to catch interactions. But the monitoring is load-bearing — you will ship flag combinations you haven't tested, and the only safety net is observability. The second cost: flag removal. Every release flag should have an expiration date and an owner. When the flag ships to 100% and stabilizes, remove it — the flag, the old code path, and the tests for the old code path. Teams that are disciplined about flag hygiene ship faster over time. Teams that aren't accumulate a rat's nest of flags, and the if statements obscure the actual business logic. A feature flag system without a removal process is a liability, not an asset.
predict You remove a release flag that's been at 100% for three months. You delete the old code path behind the else. The deploy goes out. Panic: a critical internal tool that was still importing the old function breaks. What category of problem is this, and what's the process fix?

This is a flag removal failure caused by incomplete dependency scanning. The if branch was disabled and inactive, but the old code was still importable. The process fix: treat flag removal as a code change, not a cleanup. Search for all references to the flagged function before deleting it. Better: use a flag system that tracks which flags are currently active and alerts when a 100% flag has been stable for N days. Even better: automate the removal — a bot that files a PR deleting the flag and the old code path, runs tests, and assigns it to the flag owner. Flags are infrastructure; infrastructure needs automation to stay healthy at scale.
