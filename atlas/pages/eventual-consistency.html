Eventual Consistency

Wrong now, right later — and why that's often the correct trade, not a bug you couldn't fix.

Strong consistency means: after a write completes, all subsequent reads see that write. This is what a single-machine database gives you by default. It's also what you give up when you distribute data across machines and want the system to remain available during network partitions. The CAP theorem tells you that during a partition, you choose between consistency and availability. Eventual consistency is the choice for availability: the system accepts writes during a partition, replicas diverge, and when the partition heals, they converge.

Eventual consistency is the promise that if no new writes arrive, eventually all replicas will return the same value. It says nothing about how long "eventually" takes, or what happens while they differ.

What "eventually" actually means

The formal definition (from Werner Vogels, Amazon CTO and Dynamo architect):

  1. If no new updates are made to an object, eventually all accesses will return the last updated value.
  2. The system guarantees that updates are propagated to all replicas, and all replicas eventually agree on the order of updates (the "reconciliation" or "merge" process).

What this doesn't say: how long "eventually" takes. In a healthy system, it's milliseconds — the time for a write to propagate from one datacenter to another. During a partition, it's the duration of the partition plus reconciliation time. During a sustained outage, it's hours or days. "Eventually" is a bound on correctness, not latency. The system will eventually be correct; right now, it might not be.

This is the trade you make. Strong consistency says "reads always return the latest write, but the system might be unavailable during a partition." Eventual consistency says "the system is always available for reads and writes, but reads might return stale data." The choice depends on what your application can tolerate.

What happens during the inconsistent window

During the window between a write and its propagation to all replicas, different replicas return different values for the same key. This leads to three specific anomalies:

  • Stale reads: You write to one datacenter, read from another, and see the old value. This is the most common eventual consistency anomaly and the one users notice (refresh the page, see your change disappear, refresh again, see it reappear).
  • Read-your-writes violation: You write a comment, navigate to a different page, come back, and your comment is gone — because the read hit a replica that hasn't received the write yet. The fix: route reads for a user's own data to the same replica that accepted their writes (sticky sessions, or a consistency token that encodes which replicas have been written to).
  • Monotonic read violation: You read from replica A (which has the latest data), then read from replica B (which doesn't), and it appears that data has disappeared. Time appears to move backward. The fix: ensure each user's reads are served by a replica that's at least as current as the one they last read from.
predict A user in Europe writes a tweet. A user in the US reads their timeline. The tweet is visible. A user in Asia reads their timeline. The tweet is absent. All three requests hit different datacenters. The system is eventually consistent. What specific property is being violated for the Asian user, and what's the architectural fix if Twitter wanted to prevent this?

The Asian user is experiencing a stale read. The write to the European datacenter hasn't propagated to Asia yet. The architectural fix: synchronous replication to a quorum before acknowledging the write. If W replicas must acknowledge the write (across datacenters), and the user reads from R replicas with R+W > N, the Asian user will always see the tweet. But this costs latency — the European user's write now waits for acknowledgment from at least one remote datacenter before the "tweet sent" confirmation appears. Twitter chose availability (the write succeeds instantly in the local datacenter, propagation happens async) over consistency (the tweet appears everywhere simultaneously). For most social media, this is the right trade. For a bank balance, it's not.

Convergence: how replicas actually agree

Eventual consistency doesn't mean "replicas merge randomly." It means they converge using a deterministic merge strategy:

  • Last-writer-wins (LWW): Every write has a timestamp. The write with the latest timestamp wins. Simple, deterministic, and silently wrong when concurrent writes both have valid data (one user's edit is discarded). Used by Cassandra for conflict resolution when no custom resolver is configured.
  • Version vectors: Each replica tracks which writes it has seen using a vector of per-replica counters. When two replicas exchange state, the version vectors identify which writes are missing on each side — no timestamps needed, no clock synchronization. Used by DynamoDB for causality tracking.
  • CRDTs: Data structures designed to merge deterministically. An increment-only counter as a CRDT converges to the sum regardless of merge order. A set CRDT converges to the union. No conflict resolution needed — the data structure's merge operation is mathematically guaranteed to converge. Used by Riak and Redis Enterprise for specific data types.
the business question Eventual consistency is a product decision, not a database configuration. The question is: what's the business cost of showing stale data for N seconds? For a shopping cart, the cost is low (a stale cart for 2 seconds might show the wrong item count; the next page load corrects it). For a payment ledger, the cost is existential (a stale balance might let someone withdraw money they don't have). The database gives you knobs (R, W, N). The business gives you the acceptable staleness window. Set the knobs to meet that window, and no tighter — over-consistency is over-engineering.

Threads worth pulling

The data structures that merge cleanly — CRDTs are the mathematical answer to "how do replicas converge without conflicts?" — counters, sets, and registers that merge deterministically.

The clock that tracks causality without timestamps — vector clocks solve the "which write happened first?" problem without requiring synchronized clocks across machines.
