Blue-Green & Canary Deploys

Shipping to some users first, and rolling back before anyone notices — the deploy patterns that turn releases from an event into a dial.

A traditional deploy replaces the old version with the new version. If the new version breaks, you're broken for everyone until rollback completes. Rollback typically takes as long as deploy — minutes to tens of minutes — during which users see errors. The deployment is a binary risk: you're either fine or you're down.

Blue-green and canary deploys are the two patterns that eliminate this binary risk by running both versions simultaneously and shifting traffic between them. The difference is how quickly you shift.

Blue-green: swap everything at once

In a blue-green deploy, you have two complete environments: blue (current production) and green (new version). You deploy the new version to green, run smoke tests against it, then flip a load balancer to send all traffic to green. If green fails, flip back to blue. Rollback is a load balancer change — seconds, not minutes.

Blue-green: deploy to an idle environment, test it, then swap. Rollback is instant because the old environment is still running.

The cost: you need double the infrastructure during deploys. For a 100-server fleet, you need 100 servers for blue and 100 for green — briefly, during the cutover. In cloud environments, this is a cost-optimization problem (spin up green, test, swap, spin down blue — you only double-pay for ~30 minutes). In on-premise environments with fixed capacity, it's a harder constraint.

The other cost: database schema changes must be compatible with both versions. If the new version adds a column, the old version must still work with the new schema (or the schema change must happen before the deploy, in a separate migration). This is the expand-contract pattern: expand the schema to support both versions, deploy the new code (which uses the new schema), then contract by removing support for the old schema once the old code is gone. Blue-green deploys force this discipline, which is a feature — schema changes that break the running version are also schema changes that make rollback impossible.

Canary: shift traffic gradually

A canary deploy shifts a small percentage of traffic to the new version and monitors it before shifting more. The progression: 1% → 5% → 25% → 100%, with automated checks at each step. If the canary's error rate, latency, or business metrics deviate from baseline, the deploy is automatically halted and traffic shifts back to the old version.

The name comes from coal mining: miners carried canaries into tunnels because the birds were more sensitive to toxic gases. If the canary died, the miners evacuated. In software, if the canary instances start returning 500s, the deploy is aborted before 99% of users are affected.

Canary deploys require more infrastructure sophistication than blue-green: the load balancer (or service mesh) must support fractional traffic splitting, the monitoring system must compare canary metrics to baseline in real time, and the deploy pipeline must automate the progression and rollback. Tools: Argo Rollouts (Kubernetes native), Spinnaker, AWS CodeDeploy, and service meshes (Istio, Linkerd) that can split traffic at the request level.

predict You canary-deploy a backend change: 5% of traffic to the new version. The canary's p99 latency is 200ms — identical to baseline (205ms). You promote to 100%. Suddenly, latency spikes to 2 seconds. The canary looked fine. What happened?

The new code path hits a database query that's fine at 5% load but saturates a connection pool at 100% load. The canary detected code correctness (the new version responds correctly) but not capacity correctness (the new version uses resources differently at scale). The mitigation: canary deploys need both correctness signals (error rate, response validity) and capacity signals (CPU per request, database connections per request, cache hit rate). A canary at 5% traffic should project to 100% — if the new version uses 2× the database connections per request, that's visible at 5% if you're measuring per-request resource consumption. This is also why chaos engineering and canary deploys are complementary — chaos tests capacity under failure, canaries test capacity under normal load.

When to use which

  • Blue-green: Stateless services where the deploy is fast (minutes), the infrastructure cost of double capacity is acceptable, and the rollback must be instant. Good for API servers, static frontends, and any service where "revert to the previous version" is a sufficient rollback strategy.
  • Canary: Services where the blast radius of a bad deploy is unacceptable and you want to measure the new version under real traffic before committing. Good for payment processing, authentication, and any service where "a few users saw errors" is preferable to "everyone saw errors."
  • Both (canary within blue-green): Deploy to a green environment, canary within green (shift 1% of total traffic to green), then swap 100% to green when the canary succeeds. This gives you instant rollback via blue-green and gradual validation via canary.
the database problem Both patterns assume the hard part is the application code. The database is the hard part. Blue-green requires both versions to work with the same database schema (or at least compatible schemas). Canary requires both versions to write data that both versions can read. A canary that writes data in a new format (a new required field, a changed enum value) means the old version can't process that data — and at 5% canary, 5% of your data is now in a format the old version can't read. The fix: forward-compatible schema changes. New fields are optional. Enum values are added, not changed. The old version ignores what it doesn't understand. This is a database design discipline, not a deploy tooling feature.
