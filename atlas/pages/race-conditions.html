Race Conditions & Memory Models

Why correct-looking concurrent code is wrong in ways your tests won't catch — and why the hardware is playing a different game than you think.

You write x = 1 on thread A and print(x) on thread B. Thread A goes first. What does thread B print? If you answered "1," the hardware would like a word. Thread B might print 0. It might print 1. It might print both on different runs of the same binary on the same machine. The reason isn't a bug in your code — it's that your code and the hardware are operating under different rules about what "first" means.

Data races: when two threads touch the same thing without coordination

A data race occurs when two threads access the same memory location, at least one access is a write, and there's no synchronization between them. The result is undefined behavior — not just "any value is possible," but in C/C++, the compiler is allowed to assume your program doesn't contain data races and optimize accordingly, producing results that are logically impossible if you trace through the source code line by line.

A data race, in Go
var counter int

func increment() { counter++ }     // goroutine A: read, add, write
func read()      { fmt.Println(counter) }  // goroutine B

// Run both concurrently. The Go race detector (-race) catches this.

The counter++ operation is not atomic. It compiles to: load counter into a register, increment the register, store back to memory. If thread B loads between thread A's load and store, both threads see the same value, both write back the same incremented value, and one increment is lost. This is the lost update problem, and it's the simplest race condition — the one you catch in code review and protect with a mutex or an atomic. The harder ones are structural.

Memory ordering: when the hardware reorders your instructions

The order you write lines of code is not the order the CPU executes them, and it's not the order other CPUs observe them.

Both the compiler and the CPU are free to reorder memory operations as long as the reordering doesn't change the observable behavior of a single thread. If your code on one thread writes to A and then to B, the CPU might write to B first — because B's cache line is available now and A's is in a different state. On a single thread, this is invisible (no other code depends on the ordering of A and B). But another thread watching A and B can observe B updated before A, and if that thread's logic assumed A updates first, you have a bug that no amount of single-threaded testing will catch.

Message-passing without proper ordering
// Thread A:                           // Thread B:
data = 42;     // "payload"           while (!ready) { /* spin */ }
ready = true;  // "flag"              print(data);

// Expected: prints 42.
// Reality on ARM/PowerPC: can print 0, because thread B
// might see ready=true before data=42 is visible.

Thread B's observation of the writes can be reordered. On x86 (strong memory model), stores are not reordered with other stores, so this pattern mostly works — but "mostly" is not a correctness guarantee. On ARM (weak memory model, used in every smartphone and Apple Silicon Mac), stores can appear in a different order to other threads. The fix is a memory barrier (a fence, or — more practically — an atomic operation with acquire/release semantics) that tells the hardware "everything before this barrier must be visible before anything after it."

Memory models: the contract between your code and the hardware

A memory model is the set of guarantees a language or hardware platform makes about how memory operations are ordered and visible across threads. It answers: "if thread A writes to x and y, and thread B reads y and then x, what values can B see?"

The spectrum, from strongest to weakest guarantees:

  • Sequential consistency (SC): All threads see all memory operations in a single global order. Easy to reason about, expensive to implement. No modern hardware provides this natively.
  • x86-TSO (Total Store Order): Stores are not reordered with other stores. Loads are not reordered with other loads. Stores can be reordered with later loads (store buffering). This is what x86 CPUs actually do. It's strong enough that many common patterns "just work" — and pernicious enough that the ones that don't are surprising.
  • ARM / RISC-V (weak ordering): Almost anything can be reordered unless you insert explicit barriers. The hardware is simpler and more power-efficient; the burden shifts to the programmer (or the compiler, or the language runtime) to insert barriers where needed.

Languages build their own memory models on top of these hardware models. Java's memory model was the first rigorous language-level memory model (2004, after years of debate). C++11 added one. Rust inherits C++'s model. Go has its own model. The key primitives are atomic operations with ordering guarantees — load, store, compare_exchange — each parameterized by a memory ordering: relaxed (no ordering guarantee, cheapest), acquire/release (pairs that create a happens-before relationship), sequentially consistent (global total order, most expensive).

predict You use a Mutex to protect the counter. Problem solved — the mutex provides mutual exclusion and memory ordering. But your code now has a different failure mode: two threads each holding a different mutex, each waiting for the other's mutex. What's the formal name for this, and what's the practical rule that prevents it?

The formal name is deadlock. The practical rule: acquire locks in a consistent global order. If every code path that needs mutexes A and B acquires A before B, you can't deadlock — there's no cycle in the wait-for graph. The other prevention: try_lock with a timeout and backoff (if you can't get the lock after N ms, release what you hold and retry). This is the engineering practice that sits on top of the memory model theory: the hardware guarantees atomics do what they say, and you build higher-level concurrency primitives on top.

the real advice You shouldn't be writing atomics by hand unless you're building concurrency primitives. Use the language's standard tools — mutexes, channels, async runtimes, Arc<Mutex<T>> — which encapsulate the ordering guarantees. They're written by people who've read the memory model, tested on every architecture, and handled the edge cases. The value of understanding memory models isn't that you'll write memory_order_acquire — it's that when your concurrent code fails in a way that makes no sense, you'll suspect reordering instead of dismissing it as impossible.
