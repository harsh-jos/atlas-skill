Replication

Keeping copies in sync — the source of both durability and most consistency bugs in distributed systems.

Data on one machine is a liability. That machine dies, and the data is gone. Replication — keeping copies of data on multiple machines — is the answer. It's also the source of the hardest problems in distributed systems, because now you have multiple copies that must agree, and agreement across a network that can delay, drop, or partition messages is where distributed systems get interesting.

Leader-follower: one writer, many readers

The most common replication model: one replica is the leader (primary). All writes go to the leader. The leader sends the changes to the followers (secondaries, replicas). Followers apply the changes in the same order and arrive at the same state. Reads can go to the leader or to any follower, depending on your consistency requirements.

This is the model behind PostgreSQL streaming replication, MySQL group replication, MongoDB replica sets, and Kafka's partition replication. It's simple, well-understood, and gives you:

  • Read scalability: Distribute reads across followers. Write throughput is still limited to one machine (the leader), but most workloads are read-heavy.
  • Failover: If the leader dies, a follower is promoted to leader. This takes seconds to minutes and typically requires human confirmation (automated failover is possible but risky — a network partition can produce two leaders, which is worse than a brief outage).
Leader-follower replication gives you one copy of the truth (the leader's write order). The cost is that the leader is a bottleneck and a single point of failure.

The critical implementation choice: synchronous vs asynchronous replication. Synchronous: the leader waits for the follower to confirm the write before responding to the client. The follower is always up to date, but writes are as slow as the slowest synchronous follower. Asynchronous: the leader acknowledges the write immediately and sends changes to followers in the background. Writes are fast, but if the leader crashes before the follower receives the change, the write is lost. Most systems use semi-synchronous: one follower is synchronous (for durability), the rest are asynchronous (for read scaling).

Multi-leader: writes anywhere

In multi-leader replication, any replica can accept writes. Each leader replicates its writes to the others. This is the model behind CouchDB, DynamoDB (with global tables), and CRDT-based systems. The benefit: writes can go to the nearest datacenter, reducing latency for globally distributed users. The cost: write conflicts — two leaders accept conflicting writes simultaneously, and the system must resolve them.

Conflict resolution strategies:

  • Last write wins (LWW): Use timestamps; the write with the latest timestamp wins. Simple, deterministic, and wrong for many use cases — if two users edit the same wiki page simultaneously, one's edit is silently discarded.
  • Application-level merge: The database surfaces the conflict to the application. The application resolves it (e.g., merge the two edits, show the user both versions, or apply business logic). More correct, more complex.
  • CRDTs: Data structures that merge deterministically without conflicts. A counter implemented as a CRDT converges to the sum of all increments regardless of order. A set CRDT converges to the union of all additions. The magic under collaborative editors like Google Docs.

Leaderless: everybody can write

In leaderless replication (Dynamo-style), the client writes to multiple replicas directly and considers the write successful when a quorum acknowledges it. Reads similarly query a quorum of replicas. The quorum condition R + W > N (where N is the total number of replicas) ensures that at least one replica in every read quorum has seen the latest write. There's no leader election, no failover, and the system degrades gracefully as replicas fail — you just get fewer redundancy.

The cost: read repair. When a read encounters a replica with stale data, it updates it — on the read path, adding latency to reads. And anti-entropy processes that run in the background, comparing replicas and syncing diverged data. Leaderless systems don't have a leader to sequence writes, so they're always slightly healing themselves.

predict You're using synchronous replication with one follower. The follower crashes. The leader can't replicate. What are your options, and what's the trade-off in each?

Option 1: block writes until the follower recovers or you manually remove it from the synchronous set. The system is unavailable for writes — durability (the D in ACID) is preserved at the cost of availability (the A in CAP). Option 2: degrade to asynchronous mode automatically. The leader keeps accepting writes and queues them for the follower. If the leader crashes before the follower recovers, those writes are lost. You've traded durability for availability. Option 3: promote an asynchronous follower to synchronous. There's a gap — the window between the old synchronous follower's crash and the new one catching up — where writes aren't replicated. Most production systems default to option 2 (degrade, alarm loudly, let a human decide) because blocking all writes because one replica is down is usually the wrong trade.

the replication lag tax Even with synchronous replication, followers are always slightly behind the leader — if only by microseconds. An application that writes to the leader and immediately reads from a follower (a "read-your-writes" consistency requirement) will occasionally see stale data. The fix: route reads that need the user's own writes to the leader, and reads that can tolerate slight staleness (other users' content, analytics) to followers. This is the most common replication pattern in production: write to leader, read from leader for the writing user's session, read from followers for everything else.
