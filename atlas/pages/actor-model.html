The Actor Model

Concurrency as mailboxes and messages instead of shared memory — the model that Erlang, Akka, and Orbit turned into systems that stay up for years.

Shared-memory concurrency — threads, locks, atomics — is local-first: it's designed for one machine, one address space. The actor model (Hewitt, Bishop & Steiger, 1973) inverts this: concurrency is about independent entities communicating via messages. There is no shared memory. Actors don't read each other's state. They send messages. The model was designed for distributed systems before distributed systems were the default, and it took 30 years to become necessary.

An actor is a computational entity with a mailbox. It receives messages one at a time, processes each to completion, and can send messages to other actors or create new ones. Nothing else can touch its state.

The three rules

  1. Process one message at a time. An actor's state is private. No locks, no atomics, no concurrent modification — because only one message is ever being processed per actor at any moment.
  2. Communicate only by message. Actors don't call each other's methods. They send messages to addresses (mailboxes). The caller doesn't know whether the receiver is on the same machine or a different continent — the model is location-transparent by design.
  3. Failure is a message. Actors supervise other actors. If a child actor crashes, its supervisor receives a failure message and can decide: restart the child, escalate the failure, or do nothing. This is the let it crash philosophy — don't program defensively for every possible failure; program the recovery.

Erlang: the actor model in production for 30 years

Erlang (Ericsson, 1986) is the canonical actor language and the proof that the model works at scale. Ericsson's telephone switches — running millions of Erlang processes (actors) — achieved nine nines of availability (99.9999999%, or ~32ms downtime per year). Not because Erlang code doesn't have bugs, but because the supervision hierarchy catches them, restarts the affected actor, and the rest of the system keeps running.

An Erlang process is not an OS thread — it's a lightweight green thread (a few KB of memory, microseconds to create). A single Erlang VM can run millions of processes. Each process has a mailbox (a queue of messages). The VM's scheduler preempts processes at function-call boundaries (reduction counting), so a CPU-bound process can't starve others. This is similar to Go's goroutines or Java's virtual threads, but with a crucial difference: Erlang processes cannot share memory. The only way to share data is to send a message. This eliminates an entire class of concurrency bugs — data races — at the language level.

Akka and the JVM adaptation

Akka (2009, now part of the Lightbend ecosystem) brought the actor model to the JVM. Akka actors are objects with mailboxes, running on a thread pool. The key innovation: Akka actors are location-transparent — an actor reference can point to an actor on the same JVM or a remote one, and message sending looks the same. This made Akka the standard for building distributed systems on the JVM, used by LinkedIn, PayPal, and Intel.

Akka's 2022 licensing change (moving from Apache 2.0 to the Business Source License) caused a fork: Apache Pekko, which continues the Apache 2.0 lineage. Both are in active use in 2026.

predict Actor A sends a message to Actor B, then sends a message to Actor C. Actor B receives A's message, does some work, and sends a message to C. Actor C receives both messages. What ordering does C observe?

No guaranteed ordering. A's message to C and B's message to C are sent by different actors and arrive independently. C might receive B's message first (if B is faster) or A's message first. The only ordering guarantee in the actor model: messages from the same sender to the same receiver are delivered in order. A → C messages are ordered relative to each other. B → C messages are ordered relative to each other. But A → C and B → C messages have no ordering relationship. If C's logic requires that it process A's message before B's, the ordering must be enforced by the application (e.g., A sends to C, C sends back an acknowledgment, B waits for the acknowledgment before sending to C — which reintroduces coupling).

when the actor model shines The actor model is at its best when the problem naturally decomposes into independent entities that communicate — chat servers (each user is an actor), IoT device fleets (each device is an actor), game servers (each player is an actor). It's at its worst when the problem requires strong consistency across actors — a bank transfer from account A to account B requires coordination between two actors, and the actor model doesn't provide transactions. Systems that need cross-actor consistency end up building it on top of actors (sagas, two-phase commit), which is where the complexity of the actor model lives. The model gives you isolation per actor for free. Cross-actor consistency is all on you.
