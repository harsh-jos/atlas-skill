Circuit Breakers & Bulkheads

Stopping one failing dependency from taking the whole system down with it — the two patterns that turn a localized failure into a contained one, and why "just set a timeout" isn't enough.

A payment service calls a fraud-detection API. One day, the fraud API gets slow — not dead, but responding in 30 seconds instead of 100 ms. The payment service's thread pool, sized for fast responses, fills up waiting. Requests pile up in the accept queue. The load balancer's health check starts failing because the payment service can't respond to that either. Now no one can pay — not because the payment service is broken, but because something it depends on is slow.

This is a cascading failure: a fault in one component propagates through call chains until the whole system is degraded. Timeouts alone don't prevent it — if every request still tries the slow dependency, you can exhaust your thread pool waiting for timeouts. You need something that stops trying.

The Circuit Breaker

Michael Nygard named and popularized the pattern in Release It! (2007). A circuit breaker wraps a call to an external dependency and tracks outcomes. When failures cross a threshold, the breaker trips — it stops forwarding requests to the dependency altogether and fails fast instead.

The breaker has three states:

Closed. Requests flow normally. The breaker counts failures (errors, timeouts). If the failure count exceeds the threshold within a time window, it transitions to Open.

Open. Requests are rejected immediately — the breaker fails fast without calling the dependency. This gives the dependency time to recover and prevents the caller from wasting threads on doomed attempts. After a reset timeout (typically 30–60 seconds), the breaker transitions to Half-Open.

Half-Open. A limited number of trial requests are allowed through. If they succeed, the breaker closes — the dependency is healthy. If they fail, it opens again — the dependency needs more time. This is the self-healing mechanism: the breaker automatically detects recovery without manual intervention.

The thresholds matter: too sensitive and you trip on transient blips (a single GC pause); too lenient and the system degrades before the breaker activates. Production breakers typically count failures in a rolling window — "5 failures in the last 30 seconds" rather than "5 consecutive failures" — because a single flapping dependency can look like alternating success/failure forever.

Bulkheads: contain the blast radius

A circuit breaker protects you from a failing dependency. A bulkhead protects you from a failing part of yourself. The name comes from ships: if the hull is breached, watertight compartments (bulkheads) contain the flooding to one section. The ship stays afloat.

In software, the most common bulkhead is thread-pool isolation: assign separate thread pools to different operations. The fraud-check thread pool has 20 threads. The payment-processing thread pool has 50 threads. If fraud checks slow down, their pool fills up — but the payment pool is untouched. New payment requests still process; fraud checks queue or fail fast.

Without bulkheads, one slow operation saturates the shared thread pool and every other operation starves — the definition of a cascading failure. With bulkheads, the failure is contained to its compartment.

Other forms of bulkheading: separate connection pools per downstream service (don't let one slow database exhaust connections the cache needs), separate queues per tenant or operation type, CPU isolation via cgroups or separate processes.

A circuit breaker says "stop calling the broken thing." A bulkhead says "when something breaks, don't let it break everything else."

Real systems that use them

Netflix Hystrix (now in maintenance mode, but the reference implementation) wrapped every external call in a circuit breaker with a bulkheaded thread pool. During the 2011 AWS outage, Netflix's systems largely stayed up because Hystrix tripped breakers against degraded AWS services rather than letting threads pile up. This was the production proving ground for the pattern.

Resilience4j (Hystrix's successor, used in Spring and modern JVM stacks) provides circuit breakers, bulkheads, rate limiters, and retry as a library — lightweight, functional-style, designed for the reactive streams era.

Envoy / Istio push circuit breaking to the service mesh layer. The sidecar proxy trips a breaker against a backend without the application knowing. This is powerful (no code change) but blunt — the proxy can only trip on connection-level signals (refused, timeout), not application-level ones ("the API returns 200 with status: degraded").

predict Your circuit breaker is closed. The dependency starts returning errors slowly — one failure every 10 requests. Your threshold is "5 failures in 30 seconds." It takes 50 requests and 30 seconds to trip. During those 30 seconds, the dependency is also slow (5-second timeouts), and your thread pool of 40 threads is filling up. How many threads are consumed waiting for timeouts before the breaker trips?

At 10 requests/second, 1 failure/second, 5 seconds per failure = roughly 5 threads held at any moment for failing calls. But the real problem is the 9 successful requests/second, each taking 5 seconds — 45 threads held. After 6 seconds, the 40-thread pool is exhausted. The breaker doesn't trip for 30 seconds (the counting window). The system is down for 24 seconds before the breaker even opens. This is why you combine a circuit breaker with a short timeout and a thread-pool bulkhead: the timeout limits thread occupancy; the bulkhead contains the damage; the breaker stops retrying.

The patterns together

Circuit breaker, bulkhead, timeout, and retry form a layered defense:

Timeout — the innermost guard. Don't wait forever. Size it to the dependency's p99 plus some headroom.

Retry — handle transient failures (network blip, brief overload). Must be idempotent. Retry with backoff, and cap the retry budget per request.

Circuit breaker — when failures are sustained, stop calling. Fail fast instead of failing slowly.

Bulkhead — when the breaker is still closed but threads are piling up, contain the damage to one compartment.

Each layer buys time for the layers outside it. Skip one, and the next layer faces a harder problem.

when not to Don't circuit-break a dependency that's in the critical path with no fallback. If the checkout page can't complete without the payment gateway, tripping the breaker just turns a slow checkout into a broken one — both are failures to the user. In this case, you need a fallback (queue the payment for async processing, show a "we'll confirm shortly" page) or you accept that the dependency's availability is your availability. The breaker is a tool for fast failure, not a replacement for the dependency working.
